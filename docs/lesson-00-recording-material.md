# 第 0 课录屏材料：课程导入和预期管理

## 主控验收结论

本录屏材料通过主控窗口验收，可以用于录制第 0 课样片。

通过理由：

- 明确了课程定位：不是工具介绍，而是围绕 AI 辅助研发闭环。
- 同时覆盖领导视角和技术学员视角。
- 能自然说明讲师能力边界，不夸大成大模型底层专家。
- 讲清楚课程交付物、POC 主线、AI 研发工作流、主控 / 子会话分工。
- 强调安全、脱敏、测试和人工验收边界。
- 风格适合真实内部技术培训，不鸡血、不夸大 AI。
- 可和第 6 课联调样片形成“价值说明 + 实际效果”的组合。

## 5. 开场 1 分钟口播稿

大家好，这节是我们 AI 编程入门教程的第 0 课，主要做课程导入和预期管理。

先把话说清楚：这套课不是大模型算法课，也不是单纯教大家某个工具怎么点。我们的目标更务实一点，就是面向银行解决方案研发场景，用 Claude Code 加 DeepSeek，从 0 搭一个最小可运行的 Java 微服务 POC。

过程中我们会看 AI 怎么辅助需求分析、设计、代码开发、测试、联调、文档整理和验收。但 AI 只是辅助研发，不替我们承担责任。代码能不能进项目，接口是不是合理，数据有没有脱敏，测试有没有通过，最后还是人来把关。

所以这套课真正想沉淀的，不是几个提示词技巧，而是一套安全、可控、可复制的 AI 辅助研发工作流。

## 6. 完整半口语口播稿

大家好，这节是第 0 课，课程导入和预期管理。

这一节不急着装工具，也不急着写代码，我们先把这套课到底要解决什么问题讲清楚。

现在大家对 AI 编程应该都不陌生。有些同学已经用它写过脚本、查过报错、生成过单测；也有同学可能还停留在听说阶段，担心环境装不起来、担心网络访问、担心生成的代码不靠谱，或者不知道在真实项目里边界在哪里。

所以这套课的定位不是“展示 AI 很厉害”，也不是讲大模型底层算法。我们更关注研发同学每天真的会遇到的问题：需求怎么拆，接口怎么梳理，代码怎么生成，测试怎么补，联调怎么验收，文档怎么沉淀，出了问题怎么排查。

课程主线会用 Claude Code 加 DeepSeek，从 0 搭建一个最小可运行的 Java 微服务 POC。这个 POC 会包含后端服务、数据库、前端页面、风险评级、操作日志、接口联调和后续微服务演进。它不是生产系统，但它覆盖了一条完整研发链路。

大家后面会看到，AI 在这个过程中不是只负责写代码。它可以帮我们整理需求，列出不确定问题，生成设计草案，拆分任务，补测试，做代码审查，生成验收清单。但每一步都不能直接相信它。我们要看 diff，要跑测试，要人工验收，要检查安全和脱敏。

这里有一个很重要的边界：AI 是辅助研发，不是替代责任。它可以是效率很高的工程搭档，但不是审批人，也不是最终负责人。真实项目里，尤其是银行项目里，真实客户数据、账号、手机号、身份证号、生产地址、密钥，都不能随便丢给外部模型。课程里所有数据都会使用脱敏样例。

这套课的最低目标也很明确。学完之后，希望大家至少能做到四件事：第一，能把 Claude Code 和 DeepSeek 的基本环境跑起来；第二，能把一个大任务拆成 AI 可以处理的小任务；第三，能用 AI 辅助完成一个小型研发闭环；第四，知道哪些内容必须人工检查、测试和验收。

对领导和团队来说，这套课的价值也不只是“大家学个新工具”。更重要的是把个人零散经验整理成团队方法。比如会话怎么拆，提示词怎么写，项目规则怎么放进 `CLAUDE.md`，AI 生成代码怎么验收，敏感信息怎么处理，哪些场景适合先试点，哪些场景暂时不要碰。

后面的课程会按一个比较稳的节奏展开。先讲基础概念和工作流，再讲环境搭建，然后从需求分析、架构设计、后端开发、数据库和测试数据、前端联调，一步步走到微服务演进和验收复盘。每节课都尽量有一个可复制产出，不只是听完觉得有道理。

这里也提前说一下不能期待什么。学完这套课，不代表可以把完整项目丢给 AI 自动完成；也不代表大家就掌握了大模型训练、RAG、复杂 Agent 框架或者企业级私有化部署。这些是后续进阶内容。入门阶段，我们先把日常研发里最能落地的部分做好。

接下来我会先展示一下最终 POC 的效果。大家可以把它理解成这套课要拼出来的目标图：有仪表盘，有客户查询，有详情聚合，有风险重算，有操作日志，也有错误处理和脱敏检查。后面每节课就是把这里的一块能力拆开，讲清楚 AI 怎么帮忙，人怎么验收。

最后再强调一句：这套课不是只教工具，而是沉淀一套安全、可控、可复制的 AI 辅助研发工作流。能用起来是一层，能稳定用、可复用、能推广，才是我们真正想要的结果。

## 8. 领导视角表达

这套课的价值不是单个同学会不会用某个 AI 工具，而是团队能不能形成统一、可复制、可控的 AI 辅助研发方法。

它可以帮助团队降低上手门槛，减少每个人重复摸索环境、提示词和使用边界的成本；也可以把需求分析、设计、测试、文档、验收这些容易被忽略的环节结构化沉淀下来。

推广上建议先小范围试点，用 POC 和样片验证效果，再逐步扩展到更多项目组。这样既能提升研发效率，也能控制安全和质量风险。

## 9. 学员视角表达

学完之后，你应该能：

- 搭起 Claude Code + DeepSeek 的基础环境。
- 写出相对清晰的 AI 任务描述。
- 让 AI 辅助做需求、设计、开发、测试和验收。
- 看懂 AI 输出的风险点，而不是直接复制。
- 用一个小型 POC 跑通研发闭环。

但学完之后不等于：

- 能让 AI 自动完成真实项目。
- 可以跳过需求评审、代码审查和测试。
- 可以上传真实敏感数据。
- 已经掌握大模型底层算法或复杂 Agent 框架开发。

## 10. 讲师自我定位表达

这套课我不是站在大模型底层算法专家的角度来讲，而是站在研发负责人和工程实践者的角度，把自己摸索 AI 编程工具的过程整理成一套更适合同事上手的路径。

我的重点不是证明我懂多少模型原理，而是把工具怎么落到真实研发流程里、哪些地方有效、哪些地方危险、怎么验收、怎么推广讲清楚。

## 11. 必须讲清的边界

- AI 是辅助研发，不是替代责任。
- 本课程不是大模型算法课，不讲模型训练和底层原理深挖。
- 不上传真实客户数据、生产配置、密钥、账号、手机号、证件号。
- AI 输出必须经过测试、代码审查和人工验收。
- 真实项目使用必须遵守公司安全规范和项目管理流程。
- POC 是教学样例，不等于生产级系统。

## 12. 讲义精简版

本课程面向银行解决方案研发同学，目标是用 Claude Code + DeepSeek 从 0 搭建一个最小可运行 Java 微服务 POC，并沉淀一套安全、可控、可复制的 AI 辅助研发工作流。

课程不是大模型算法课，也不是单纯工具教学。它会围绕需求分析、架构设计、后端开发、数据库、前端联调、测试验收和安全边界，演示 AI 如何融入真实研发流程。

学员学完后应能搭建基础环境、拆分 AI 任务、用 AI 辅助完成小型研发闭环，并知道 AI 输出必须测试和人工验收。课程不会承诺 AI 自动完成项目，也不会鼓励上传真实敏感数据。

## 13. 课后行动建议

- 先看完整课程路线，理解后续每节课在拼哪块能力。
- 准备自己的本机环境，记录安装和网络问题。
- 选一个非敏感、低风险的小需求，尝试写一份 AI 任务描述。
- 后续每节课都保留提示词、AI 输出、人工修改和验收结果。
- 把“不确定、没验证、涉及敏感信息”的内容标出来，不要直接交给 AI 处理。

## 14. 失败兜底方案

如果领导问 ROI：

可以回答第一阶段不承诺精确量化收益，先用样片和 POC 做小范围试点。重点观察环境上手时间、需求和设计文档质量、测试补齐效率、联调问题定位效率，再决定扩大推广。

如果同学问没有代理怎么办：

先区分安装问题、网络访问问题和模型 API 访问问题。课程会给 Windows、WSL2、macOS 的建议路径，也会准备截图和 FAQ。没有代理时不强行现场排查，可以先看录屏和文档，环境问题单独处理。

如果有人质疑 AI 会不会乱写代码：

会，而且这正是课程要反复强调验收的原因。AI 生成代码不能直接进项目，必须看 diff、跑测试、做代码审查、做安全检查。课程重点不是“让 AI 多写”，而是“让 AI 在可控流程里写”。

如果环境装不起来怎么办：

第 0 课不依赖现场安装。后续环境课会单独讲。录制和试讲时准备好截图、命令清单、常见报错和备用演示环境，避免课程节奏被个别机器问题卡住。
