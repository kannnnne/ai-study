# 主控窗口工作台

## 当前角色定位

本窗口是整个 AI 教程项目的主控窗口，负责课程主线、任务拆分、上下文管理和最终验收。它不应该塞入所有细节实现，而应该像项目经理加总架构师一样，维护全局路线，并把具体工作拆给不同子窗口完成。

## 主控窗口负责什么

- 维护课程目标、边界和整体节奏。
- 拆分子窗口任务，例如前端、后端、数据库、安装文档、讲稿、领导汇报。
- 给每个子窗口生成清晰提示词。
- 定义每个子任务的验收标准。
- 汇总子窗口产出，判断是否符合课程主线。
- 发现上下文跑偏时，及时开新会话而不是继续纠缠。
- 维护最终课程材料目录和交付状态。

## 主控窗口不负责什么

- 不在一个会话里同时完成所有编码、写稿、改 UI、修 bug 和写 PPT。
- 不把多个无关任务堆在同一个上下文里。
- 不让 AI 在没有需求分析和设计评审的情况下直接开发。
- 不用一次超长提示词要求 AI 完成完整系统。

## 推荐会话拆分

| 会话类型 | 负责内容 | 结束条件 |
| --- | --- | --- |
| 主控会话 | 总体规划、拆任务、验收 | 课程主线清晰，子任务可派发 |
| 环境会话 | Claude Code + DeepSeek 安装配置 | Windows、macOS、WSL2 路径跑通 |
| POC 架构会话 | 业务场景、模块划分、技术选型 | 架构图、模块边界、接口清单完成 |
| 后端会话 | Spring Boot / Spring Cloud / Nacos / 数据库 | 后端最小功能可运行，有测试 |
| 前端会话 | 页面设计、交互、接口联调 | UI 可用、好看、可演示 |
| 数据库会话 | 表结构、初始化数据、迁移脚本 | 本地启动后数据可用 |
| 测试验收会话 | 单测、接口测试、AI 验收、人工验收 | 验收清单全部通过 |
| 讲稿会话 | 视频脚本、口播稿、演示台词 | 可直接录屏 |
| 汇报会话 | 领导版价值说明、PPT 大纲 | 能对上管理视角 |

## 主控工作流

1. 明确当前阶段目标。
2. 拆出一个独立子任务。
3. 写清背景、输入、输出、约束和验收标准。
4. 开新会话执行子任务。
5. 子窗口完成后回到主控窗口汇报。
6. 主控窗口做需求验收、设计验收、架构验收、安全验收和课程价值验收。
7. 通过后进入下一个子任务。

## 判断是否需要新开会话

出现以下情况就应该新开会话：

- 当前任务和原任务已经不是一类问题。
- 上下文里混入了太多历史讨论，AI 开始重复或误解。
- AI 已经多轮修不好同一个问题。
- 需要切换角色，例如从后端开发切到前端视觉设计。
- 需要让 AI 重新从干净上下文做审查。

## 本教程要传达的核心习惯

- 复杂任务先分析，再设计，再开发。
- 每个阶段都有验收标准。
- AI 产出必须可运行、可测试、可审查。
- 一个会话只做一条主线。
- 主线会话负责决策，支线会话负责执行。

## GitHub 同步规则

本项目 GitHub 仓库：

```text
https://github.com/kannnnne/ai-study.git
```

从 2026-06-21 起，主控窗口每次完成一次主控验收并判定“通过”后，应提交所有已验收变动到该仓库。

同步原则：

- 只提交已验收通过的设计、代码、文档和任务看板更新。
- 不提交构建产物、依赖目录、本地环境变量、密钥或真实敏感信息。
- 提交前先查看 `git status`。
- 提交信息要能说明本次通过验收的阶段。
- 如本机认证或网络导致无法推送，应保留本地提交并在主控窗口说明阻塞。
