# 课程交付物与开展方式

## 最终你会得到什么

当所有子任务完成后，这个项目应该产出一套完整的内部 AI 编程入门课程包，而不只是一个 POC 项目。它包含视频、文档、代码、演示脚本、提示词模板、验收材料和领导汇报材料。

可以把最终交付物理解为 6 类：

1. 课程包。
2. 可运行 POC 项目。
3. 讲师录制材料。
4. 学员学习材料。
5. 团队推广材料。
6. 后续迭代资产。

## 1. 课程包

课程包是给学员看的主内容。

### 视频

建议第一版控制为 8 节主课：

| 课程 | 视频主题 | 目标 |
| --- | --- | --- |
| 第 0 课 | 课程导入和预期管理 | 让学员理解为什么学、学完能做什么 |
| 第 1 课 | AI 编程基础概念和工作流 | 建立 AI 辅助研发的正确方法 |
| 第 2 课 | Claude Code + DeepSeek 环境搭建 | 让学员把工具跑起来 |
| 第 3 课 | POC 需求分析 | 演示如何让 AI 帮忙澄清需求 |
| 第 4 课 | POC 架构设计 | 演示单体优先、微服务后置的设计思路 |
| 第 5 课 | 后端和数据库最小实现 | 演示 Spring Boot、MySQL、接口和测试 |
| 第 6 课 | 前端页面和接口联调 | 演示 Vue 企业后台页面和 API 联调 |
| 第 7 课 | 微服务演进与验收复盘 | 演示 Gateway、Nacos、AI 验收和人工验收 |

如果时间允许，可以扩展为 10 节：

- 把“数据库和测试数据”单独拆成一课。
- 把“风险评级规则和测试”单独拆成一课。

### 每节视频配套材料

每节课都应该有：

- 本节目标。
- 适用场景。
- 录屏步骤。
- 使用的提示词。
- 关键命令。
- 常见问题。
- 风险提示。
- 课后练习。

## 2. 可运行 POC 项目

这是课程最重要的演示资产。学员不是只看概念，而是能看到一个真实跑起来的系统。

### POC 代码

最终应该包含：

- `frontend`：Vue 3 + TypeScript + Vite + Element Plus 页面。
- `backend`：Spring Boot 3.x 单体服务。
- `gateway-service`：第二阶段微服务网关。
- `customer-service`：第二阶段客户服务。
- `risk-service`：第二阶段风险服务。
- `docker-compose.yml`：本地 MySQL、Nacos 等依赖。
- `CLAUDE.md`：项目级 AI 协作规则。

### POC 文档

最终应该包含：

- 需求文档。
- 架构设计文档。
- 数据库设计文档。
- API 设计文档。
- 测试计划。
- 验收清单。
- 本地启动说明。
- 常见问题。

### POC 验收标准

至少满足：

- 本地能启动前端、后端、数据库。
- 客户列表能查询。
- 客户详情能查看。
- 风险等级能展示。
- 风险重算能触发。
- 操作日志能查看。
- 后端核心测试通过。
- 页面主流程可演示。
- 不包含真实客户数据、密钥、生产地址。

## 3. 讲师录制材料

这是给你自己用的材料，帮助你稳定录制，不靠临场发挥硬撑。

### 逐字稿或半口语稿

每节课建议准备一份半口语稿，不一定逐字念，但要覆盖：

- 开场怎么说。
- 这节课解决什么问题。
- 每个操作为什么这么做。
- AI 输出哪里对，哪里要人工判断。
- 结尾如何收束。

### 录屏脚本

每节课要有一份操作清单：

- 开哪些软件。
- 运行哪些命令。
- 输入哪些提示词。
- 展示哪些文件。
- 哪些地方需要提前准备。
- 哪些 token、路径、内部信息必须隐藏。

### 演示备份

建议每节课准备：

- 一个录制前初始版本。
- 一个录制后完成版本。
- 一份关键输出备份。
- 一份失败时的补救说明。

这样录制时即使 AI 输出不稳定，也不会被现场结果拖垮。

## 4. 学员学习材料

这是给同事课后复习和照着做的材料。

### 安装与配置手册

包括：

- Windows 安装。
- macOS 安装。
- WSL2 安装。
- Claude Code 配置。
- DeepSeek API 配置。
- 常见报错处理。
- 无代理、内网、安装包和运行访问的区别。

### 学员练习手册

每节课配一个小练习，例如：

- 让 AI 总结一个模块。
- 让 AI 生成一个接口设计。
- 让 AI 补一个单元测试。
- 让 AI 审查一段代码。
- 让 AI 生成一份验收清单。

### 提示词模板库

包括：

- 需求分析提示词。
- 架构设计提示词。
- 后端开发提示词。
- 前端设计提示词。
- 测试生成提示词。
- AI 验收提示词。
- 代码审查提示词。
- 讲稿生成提示词。

## 5. 团队推广材料

这是给领导、项目经理和团队负责人看的材料。

### 领导汇报材料

建议包括：

- 背景和问题。
- 课程目标。
- POC 展示。
- 团队收益。
- 推广计划。
- 风险控制。
- 后续扩展方向。

### 团队规范草案

包括：

- AI 使用边界。
- 敏感信息处理原则。
- 会话拆分原则。
- AI 生成代码验收原则。
- `CLAUDE.md` 编写建议。
- 项目级提示词沉淀方式。

### 试点计划

建议先小范围试点：

- 第一次给 3 到 5 个技术骨干看样片。
- 第二次给一个项目组试讲。
- 第三次再面向更大范围推广。

## 6. 后续迭代资产

课程完成后，不应该停在一次培训。

可以继续沉淀：

- 内部 AI 编程 FAQ。
- 常见报错库。
- 提示词库。
- 项目 `CLAUDE.md` 模板。
- 银行研发场景案例库。
- AI 审查清单。
- 新工具对比：Codex、Trae、Qoder 等。
- 进阶主题：MCP、RAG、内部知识库、Agent 自动化。

## 你现在应该如何开展

## 阶段 1：先完成课程骨架

目标：领导和自己都能看懂这门课要做什么。

需要完成：

- 课程蓝图。
- 分集大纲。
- POC 架构设计基线。
- 领导汇报口径。
- 环境安装文档初稿。

当前状态：已基本完成。

## 阶段 2：完成 POC 设计材料

目标：先把代码前的设计打牢。

建议派发子任务：

1. 数据库设计子窗口。
2. API 设计子窗口。
3. 后端最小实现设计子窗口。
4. 前端页面设计子窗口。
5. 测试验收设计子窗口。

产出：

- 数据库 DDL。
- 初始化数据。
- API 明细。
- 页面原型。
- 测试矩阵。
- 验收清单。

## 阶段 3：完成 POC 代码

目标：得到一个能本地运行、能录屏演示的项目。

建议顺序：

1. 创建项目骨架。
2. 完成数据库和后端客户查询。
3. 完成风险评级规则。
4. 完成前端列表和详情。
5. 完成操作日志。
6. 完成微服务演进。
7. 完成测试和验收。

注意：每一步都要保留录屏脚本和关键提示词。

## 阶段 4：录制样片

目标：先做一版最小可展示课程，不追求一次完美。

建议先录：

1. 第 0 课：课程导入。
2. 第 3 课：需求分析。
3. 第 4 课：架构设计。
4. 第 5 课：后端最小实现。

这 4 节足以证明课程方向、讲师能力和 AI 研发方法。

## 阶段 5：内部试讲

目标：验证学员是否听得懂、能不能跟着做。

建议找：

- 1 位领导或负责人。
- 2 位熟悉项目的技术骨干。
- 2 位 AI 使用经验较少的同学。

收集反馈：

- 环境是否能装。
- 概念是否听得懂。
- 案例是否贴近工作。
- 视频节奏是否合适。
- 哪些地方需要补文档。

## 阶段 6：正式发布

目标：形成正式内部课程。

交付形式建议：

- 视频上传到公司内部平台或网盘。
- 文档放到团队知识库。
- POC 代码放到内部 Git 仓库。
- 提示词模板单独整理。
- 建一个问题收集表或群内 FAQ。

## 一句话总结

你最终交付的不是“几段视频”，而是一套完整的内部 AI 辅助研发入门方案：有课程、有代码、有文档、有提示词、有验收、有推广口径，也有后续继续成长的路线。
