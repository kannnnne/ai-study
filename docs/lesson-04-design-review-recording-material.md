# 第 4 课录屏材料：用 AI 做系统设计和评审

## 1. 标题与目标

标题：第 4 课：用 AI 做系统设计和评审

一句话目标：演示如何把第 3 课收敛后的 POC 需求转换成可开发、可验收的第一阶段单体设计，并用 AI 辅助发现过度设计、契约冲突、安全边界和验收遗漏。

## 2. 本课要解决的问题

第 3 课已经完成需求分析，明确了客户风险评级与账户查询 POC 的第一阶段边界：先做一个单体 Spring Boot 后端、Vue 企业后台页面、本地样例数据库，跑通客户查询、详情、账户、风险重算和操作日志闭环。

本课要解决的问题是：需求之后为什么要先设计和评审，而不是直接编码。

如果需求刚收敛完就让 AI 写代码，常见风险有：

- 数据库、API、前端字段各说各话，联调时才发现对不上。
- AI 把登录、权限、MQ、缓存、微服务、分布式事务提前塞进第一阶段，课程复杂度失控。
- 只写正常链路，没有错误码、异常处理、日志、`requestId` 和操作追踪。
- 银行场景的脱敏、安全边界和生产系统隔离没有提前设计。
- 验收标准没有落到接口、页面、数据和日志上，最后只能靠感觉判断完成。

AI 在本课里的价值是两件事：

1. 帮我们快速列出设计草案，覆盖模块、数据流、表、API、页面、异常、日志、安全和测试。
2. 站在评审者角度帮我们查风险，尤其是遗漏、冲突和过度设计。

但架构边界不能交给 AI 自动拍板。第一阶段做什么、第二阶段再做什么、明确不做什么，必须由人根据课程目标、团队能力和验收成本来裁决。

本课核心口径：

```text
AI 可以辅助列方案和查风险，但架构边界由人拍板。
第一阶段先做可运行单体 POC，微服务演进后置。
```

### 4.1 本课设计基线摘要

第一阶段必须围绕最小闭环：

```text
筛选客户 -> 查看客户详情 -> 查看账户信息 -> 查看当前风险等级 -> 触发样例风险重算 -> 查看操作日志
```

第一阶段默认技术边界：

- Spring Boot 3.x 单体后端。
- Vue 3 + TypeScript + Vite + Element Plus 企业后台。
- MySQL 8 / H2 test profile 样例数据。
- Flyway 初始化四张核心表：`customer`、`account`、`risk_record`、`operation_log`。
- REST API 统一使用 `/api/...`。
- 统一响应、错误码、`requestId`、`X-Operator`、操作日志。
- 所有数据只使用 `CUST10000*`、`SAMPLE-*`、`masked`、`sample` 语义。

第一阶段明确不做：

- 真实登录认证。
- 复杂权限模型。
- MQ。
- 缓存集群。
- 分布式事务。
- 真实风控模型。
- 真实核心、风控、征信、反洗钱、认证中心等银行系统接入。
- 生产地址、内部系统地址、密钥、token 或真实配置。

## 5. 第一轮系统设计提示词

目标：让 AI 基于已收敛需求，输出第一阶段单体 POC 的可开发设计草案。

录屏时建议新开一个“设计评审子会话”，把必要背景和文件路径给 AI。不要要求 AI 改代码，不要让它创建文件。

```text
你现在是银行研发场景下的系统设计助手。

背景：
我们正在做一门面向银行解决方案研发同学的 AI 编程入门课。
第 3 课已经把粗需求收敛为“客户风险评级与账户查询 POC”。
本课要演示如何把需求转换为可开发设计，并让 AI 辅助做设计评审。

设计目标：
请基于已收敛需求，设计第一阶段单体 POC。
第一阶段重点是先跑通最小可运行闭环，不要一开始设计成生产级微服务系统。

第一阶段最小闭环：
筛选客户 -> 查看客户详情 -> 查看账户信息 -> 查看当前风险等级 -> 触发样例风险重算 -> 查看操作日志。

已知设计输入：
- POC 架构基线：第一阶段 Spring Boot 单体，第二阶段再演进 gateway-service、customer-service、risk-service。
- 数据库基线：customer、account、risk_record、operation_log 四张表。
- API 基线：统一 /api 路径，统一响应 ApiResponse，错误码 000000、400001、404001、409001、500000。
- 前端基线：仪表盘、客户列表、客户详情、风险重算弹窗、操作日志页。
- 后端基线：common、customer、account、risk、operation-log、openapi 等模块。
- 测试基线：单元测试、集成测试、API 验收、浏览器人工验收、安全脱敏检查。

请输出第一阶段系统设计草案，必须包含：
1. 设计目标和边界
2. 模块划分和职责
3. 关键数据流，尤其是客户查询、详情聚合、风险重算和操作日志
4. 数据库表和字段关注点
5. REST API 清单和统一响应 / 错误码约定
6. 前端页面和主要交互
7. 异常处理和日志设计
8. 安全脱敏设计
9. requestId 和操作追踪设计
10. 测试策略和验收方式
11. 第一阶段明确不做什么
12. 第二阶段微服务演进方向，只说明方向，不展开实现

强约束：
- 第一阶段不引入真实登录、权限、MQ、缓存、分布式事务、真实银行系统。
- 第一阶段不接入真实核心、风控、征信、反洗钱、认证中心。
- 不使用真实客户数据、真实账号、手机号、证件号。
- 不出现生产地址、内部系统地址、密钥、token 或真实配置。
- 如果你需要做假设，请标明“假设”，不要把猜测写成事实。

输出风格：
- 贴近真实项目设计评审，不要架构炫技。
- 每个设计点都要能说明它服务哪个验收目标。
- 对第一阶段和第二阶段边界要明确，不要混在一起。
```

### 5.1 第一轮预期观察点

AI 第一轮输出通常会比较完整，但要重点观察：

- 是否把第一阶段设计成单体，而不是一上来拆服务。
- 是否覆盖 `customer`、`account`、`risk_record`、`operation_log` 四张核心表。
- 是否沿用统一响应、错误码、`requestId`、`X-Operator`。
- 是否写清风险重算事务：新增风险记录、回写客户风险快照、写操作日志。
- 是否强调日志不打印完整敏感字段、SQL、堆栈、连接串。
- 是否把真实登录、权限、MQ、缓存、分布式事务、真实银行系统排除在第一阶段之外。

## 6. 人工点评：识别过度设计和边界漂移

第一轮 AI 输出后，讲师不要马上进入开发，而是先做人工点评。可以用下面这段口径：

```text
大家注意，设计草案看起来越完整，越需要我们判断它是不是适合当前阶段。

如果 AI 在第一阶段就加入真实登录、权限中心、网关、Nacos、MQ、缓存、分布式事务、审计服务拆分，这些不一定是错的，但对入门 POC 来说太早了。

我们当前要证明的是最小闭环：客户能查、详情能看、账户能展示、风险能重算、日志能追踪。只要这个闭环能通过测试和人工验收，第一阶段就成立。

微服务、网关、注册配置、服务拆分不是不做，而是第二阶段再做。架构设计的关键不是显得高级，而是让当前阶段能开发、能联调、能验收、能讲清楚。
```

人工点评重点：

- 哪些设计是第一阶段必须保留的。
- 哪些设计有价值但应该放到第二阶段。
- 哪些设计会误导学员，以为 POC 必须生产级。
- 哪些设计没有验收方式，需要改成可检查的标准。
- 哪些字段或接口可能和既有数据库、API、前端设计冲突。

## 7. 第二轮设计评审提示词

目标：让 AI 以独立评审者角色，按 P0/P1/P2 输出设计风险。

```text
你现在切换为“系统设计评审员”，请评审上一轮第一阶段单体 POC 设计。

评审背景：
- 这是教学用客户风险评级与账户查询 POC。
- 第一阶段必须先跑通单体 POC。
- 第二阶段再演进微服务。
- 设计必须可开发、可联调、可测试、可录屏讲清楚。

请按 P0 / P1 / P2 输出设计风险：
- P0：必须立即修正，否则不能进入开发或不能录屏的问题。
- P1：进入开发前应修正的问题。
- P2：可以作为优化或后续阶段处理的问题。

请重点检查：
1. 是否过度设计：是否把真实登录、权限、MQ、缓存、分布式事务、微服务治理、真实银行系统提前放进第一阶段。
2. 是否遗漏验收：客户查询、详情、账户、风险重算、操作日志、404、409、参数校验、安全脱敏是否都有验收方式。
3. 是否违反安全边界：是否出现真实客户数据、生产地址、内部系统地址、密钥、token，或日志打印敏感字段。
4. 是否和 API 设计冲突：路径、统一响应、错误码、分页、字段命名、requestId、X-Operator 是否一致。
5. 是否和数据库设计冲突：四张表、字段语义、脱敏字段、金额 decimal、时间格式、风险快照和历史记录关系是否一致。
6. 是否和前端页面设计冲突：页面需要的数据是否都有 API 支撑，是否新增了前端不需要的接口或遗漏了错误态。
7. 风险重算是否设计为可追踪事务：risk_record 新增、customer 当前风险快照回写、operation_log 写入是否明确。
8. 日志、错误码、requestId 和操作追踪是否足够支撑银行研发排查。

输出格式：

P0：
- 问题：
  证据：
  影响：
  建议修正：

P1：
- 问题：
  证据：
  影响：
  建议修正：

P2：
- 问题：
  证据：
  影响：
  建议修正：

已通过项：
- ...

要求：
- 只列有证据的问题，不要泛泛而谈。
- 不要建议第一阶段引入过重架构。
- 每条建议都要说明属于第一阶段修正，还是第二阶段演进。
```

### 7.1 第二轮预期输出口径

理想评审结果应该能指出：

- 如果第一阶段引入微服务、Gateway、Nacos、MQ，应列为过度设计风险。
- 如果设计没有明确 `requestId`、错误码、操作日志，应列为开发前风险。
- 如果数据库字段、API 字段、前端字段命名不一致，应列为联调风险。
- 如果风险重算只返回结果、不写历史记录或操作日志，应列为验收风险。
- 如果安全脱敏只写一句“注意安全”，没有落到样例数据、日志、错误响应和配置，应列为安全设计不足。

## 8. 第三轮主控裁决提示词

目标：让 AI 协助把设计评审结果收敛成三类清单：第一阶段必须做、第二阶段再做、明确不做。

```text
请基于上一轮系统设计和设计评审结果，帮我整理一份主控裁决草案。

裁决目标：
把当前设计收敛成三类：
1. 第一阶段必须做
2. 第二阶段再做
3. 明确不做

第一阶段必须做的内容要能支撑开发、联调、测试和录屏验收。
第二阶段再做的内容只保留方向，不展开实现。
明确不做的内容要防止后续 AI 或开发子任务继续扩范围。

请按下面格式输出：

第一阶段必须做：
- 功能：
- 后端：
- 数据库：
- API：
- 前端：
- 异常和错误码：
- 日志、requestId 和操作追踪：
- 安全脱敏：
- 测试验收：

第二阶段再做：
- ...

明确不做：
- ...

主控口径：
- 用 5 到 8 条话总结本次设计裁决，适合写进课程材料或派发给后端 / 前端 / 测试子窗口。

要求：
- 不引入真实客户数据、内部系统地址、密钥或生产配置。
- 不把第二阶段微服务内容混入第一阶段交付条件。
- 语言要清楚，像真实项目设计评审结论，不要写成架构宣传稿。
```

### 8.1 主控裁决参考结果

第一阶段必须做：

- 客户列表、客户详情、账户列表、当前风险、历史风险、风险重算、操作日志查询。
- Spring Boot 单体后端，按 `common`、`customer`、`account`、`risk`、`operation-log` 分模块。
- 四张核心表：`customer`、`account`、`risk_record`、`operation_log`。
- REST API 遵守 `/api/...`、统一响应、分页、错误码和字段命名。
- 前端实现仪表盘、客户列表、客户详情、风险重算弹窗、操作日志页。
- 风险重算必须写入历史记录、回写客户当前风险快照、写操作日志。
- 错误响应必须带 `code`、`message`、`requestId`，不暴露 SQL、堆栈、连接串。
- 样例数据、日志、页面和 HTTP 文件只使用脱敏语义。
- 自动化测试、接口验收、人工验收和安全脱敏检查都要有清单。

第二阶段再做：

- Gateway 统一入口。
- `customer-service` 和 `risk-service` 拆分。
- Nacos 注册与配置。
- 服务间调用和拆分后的回归验证。
- 网关层鉴权占位。
- 共享库分表或更进一步的独立库演进讨论。

明确不做：

- 第一阶段不做真实登录和真实权限系统。
- 不接入真实核心、风控、征信、反洗钱、认证中心。
- 不做真实风控模型或客户画像模型。
- 不做 MQ、缓存集群、分布式事务。
- 不做导出、批量导入、删除、复杂审批。
- 不使用真实客户数据、真实账号、手机号、证件号。
- 不展示生产地址、内部系统地址、密钥、token 或真实配置。

- `docs/test-acceptance-material.md`

重点说明：

- 前端不新增 API，只消费既有契约。
- 后端实现必须按 API 和数据库基线做。
- 测试验收要覆盖接口、页面、日志、安全和失败路径。

### 9.5 新开设计评审子会话

说明为什么新开会话：

- 设计评审是独立任务。
- 避免被后端代码、前端代码或数据库脚本细节拖偏。
- 本子会话只产出设计建议和评审结论，不改代码、不改 SQL。

### 9.6 输入第一轮系统设计提示词

复制第 5 节的提示词，让 AI 输出系统设计草案。

讲师观察：

- 先看它有没有守住第一阶段单体。
- 再看模块、数据流、数据库、API、前端、日志、安全、测试是否完整。
- 最后看它有没有偷偷把第二阶段内容塞进第一阶段。

### 9.7 人工点评是否过度设计

现场指出：

- 微服务、Gateway、Nacos 可以提，但不能作为第一阶段交付条件。
- 日志、错误码、`requestId`、操作追踪不能后置，因为银行研发排查问题离不开它们。
- 安全脱敏不是最后扫一遍，而是字段、样例数据、日志、错误响应和配置都要提前设计。

### 9.8 输入第二轮设计评审提示词

复制第 7 节的提示词，让 AI 按 P0/P1/P2 输出风险。

讲师重点解释：

- P0 是不能进入开发或不能录屏的问题。
- P1 是开发前应修的问题。
- P2 是后续优化，不要用 P2 把当前阶段拖重。

### 9.9 沉淀主控裁决

复制第 8 节的主控裁决提示词。

最终沉淀：

- 第一阶段必须做。
- 第二阶段再做。
- 明确不做。

讲师可以把裁决结果作为后续第 5 课后端开发、第 7 课前端联调、第 9 课验收复盘的共同输入。

## 10. 讲解重点

- 设计不是越复杂越好，设计要服务当前阶段目标。
- 第一阶段单体优先，微服务后置；先跑通闭环，再讲服务拆分。
- AI 可以帮忙列方案和查风险，但架构边界由人拍板。
- 设计必须可验收。不能验收的设计，很容易变成口号。
- 数据库、API、前端字段要对齐，否则联调阶段必然返工。
- 银行场景要提前设计日志、脱敏、错误码、`requestId` 和操作追踪。
- 风险重算这种写操作必须有事务效果和可追踪日志。
- 安全边界不能只写“不泄露敏感信息”，要落到样例数据、字段命名、日志、错误响应和配置。
- 第一阶段不接真实银行系统，不使用真实客户数据，不展示生产地址和密钥。
- 评审时要区分 P0/P1/P2，避免所有建议都变成“马上做”。

## 11. 常见错误

### 11.1 一开始就拆微服务

错误表现：

```text
第一阶段直接设计 gateway-service、customer-service、risk-service、audit-service、account-service，再接 Nacos、MQ、缓存和分布式事务。
```

问题是学员还没看懂单体链路，就被服务治理复杂度淹没。第 4 课要讲的是从需求到设计评审，不是展示一套生产级分布式架构。

### 11.2 设计没有验收标准

错误表现：

- “系统稳定可靠。”
- “日志完整。”
- “安全合规。”
- “接口设计合理。”

这些都太虚。要改成可检查的标准，例如：

- `POST /api/customers/{customerNo}/risk/recalculate` 成功后，响应 `transactionEffect` 三项为 `true`。
- 客户不存在返回 HTTP 404 + `404001`，响应包含 `requestId`。
- 操作日志可按 `requestId` 查询到风险重算记录。
- 错误响应不返回 SQL、堆栈、连接串。

### 11.3 数据库、API、前端字段不一致

错误表现：

- 数据库叫 `customer_name_sample`，API 返回 `customerName`，前端又写成 `name`。
- 数据库保留 `mobile_masked`，前端标题写“手机号”，学员误以为是真实手机号。
- API 返回数据库自增 `id`，前端拿它作为业务主键。

评审时要要求字段能追溯，且保留 `masked` / `sample` 语义。

### 11.4 忽略错误码、日志和 `requestId`

错误表现：

- 只设计成功响应，不设计 400、404、409。
- 前端错误提示只写“请求失败”。
- 日志没有请求号，无法从页面错误追到后端日志。
- 操作日志不记录重算失败样例。

银行研发里排查问题通常要串起页面、接口、日志、操作人、请求号和业务号。`requestId` 和操作追踪不是锦上添花，是第一阶段就该讲清的工程习惯。

### 11.5 把真实登录、权限、生产系统接入放进入门 POC

错误表现：

- 第一阶段就接真实认证中心。
- 要求配置真实系统地址。
- 要求使用真实客户样本。
- 用真实风控模型解释风险等级。

这些内容既超出入门课边界，也有安全风险。本课程第一阶段只用固定操作人或 `X-Operator` 模拟，不连接真实银行环境。

## 12. 半口语口播稿

大家好，这节是第 4 课，用 AI 做系统设计和评审。

第 3 课我们已经完成了一件很关键的事：把一句粗需求，收敛成了客户风险评级与账户查询 POC 的第一阶段边界。第一阶段要做的是一个最小可运行闭环：筛选客户、查看详情、看账户、看当前风险等级、触发一次样例风险重算，然后能在操作日志里追到这次动作。

这节课我们不直接写代码。原因很简单，需求收敛完以后，如果马上让 AI 开写，短期看好像很快，后面却很容易在联调和验收时返工。

比如数据库字段叫 `customer_name_sample`，API 返回 `customerName`，前端又写成 `name`；比如风险重算接口只返回一个结果，没有写历史记录，也没有操作日志；再比如 AI 很积极地把登录、权限、网关、Nacos、MQ、缓存、分布式事务全部加进来。看起来架构很完整，但对入门 POC 来说已经过重了。

所以本课要演示的是：先让 AI 帮我们出设计草案，再让 AI 以评审者身份查风险，最后由人做主控裁决。

我们先看第 3 课留下来的边界。第一阶段是单体 Spring Boot POC，加 Vue 企业后台和本地样例数据库。它不做真实登录，不做复杂权限，不接真实银行系统，不做 MQ、缓存和分布式事务。第二阶段再演进到 Gateway、customer-service、risk-service 和 Nacos。

这里我想强调一句：单体不是低级，微服务也不是高级。它们解决的是不同阶段的问题。第一阶段我们的目标是让学员看见完整链路，能开发、能联调、能验收。微服务演进当然有价值，但要放到后面讲“为什么拆、怎么拆、拆完怎么验证”。

接下来我们打开 POC 架构基线。大家看这里，第一阶段内部其实已经有模块边界：common、customer、account、risk、operation-log、openapi。也就是说，单体不等于没有设计。我们仍然要把模块职责、数据流、异常、日志和测试讲清楚，只是不要过早引入分布式复杂度。

再看数据库设计。第一阶段只需要四张核心表：customer、account、risk_record、operation_log。客户表里保留当前风险快照，方便列表筛选；risk_record 记录历史评级；operation_log 记录请求号、操作人、目标业务号、结果和错误样例。这个设计不是为了炫技，而是为了后面能验收：风险重算之后，历史有没有新增，客户快照有没有更新，日志能不能按 requestId 追到。

然后看 API 设计。所有业务接口都走 `/api`，统一响应里有 success、code、message、data、requestId、timestamp。错误码至少有参数错误、客户不存在、业务状态冲突和系统异常。这里的 `requestId` 很关键，因为它把前端错误提示、后端日志和操作日志串起来。

现在我们新开一个设计评审子会话。为什么要新开？因为设计评审是一个独立任务，我们不希望它被代码细节带偏。这个会话只输出设计草案和评审结论，不改后端代码，不改前端代码，也不改数据库脚本。

第一轮提示词，我们让 AI 基于已收敛需求设计第一阶段单体 POC。提示词里要明确要求它输出模块划分、数据流、数据库表、API、前端页面、异常和日志、安全脱敏、测试策略。还要把边界写死：第一阶段不引入真实登录、权限、MQ、缓存、分布式事务和真实银行系统。

AI 输出之后，我们不要直接照单全收。我们要现场点评它有没有过度设计。比如它如果建议第一阶段就拆五个服务，再加注册中心、消息队列、分布式事务，这些可以作为第二阶段方向，但不能成为第一阶段交付条件。

同时，我们也要看它有没有漏掉银行研发里很实际的东西：错误码有没有，日志有没有，requestId 有没有，操作追踪有没有，脱敏字段有没有，失败路径有没有。很多设计文档看起来很漂亮，但一到排查问题就发现没有请求号，没有操作人，没有业务号，那就不适合我们的场景。

第二轮，我们让 AI 切换成设计评审员，按 P0、P1、P2 输出风险。P0 是必须立即修，否则不能进入开发或不能录屏；P1 是开发前应该修；P2 是后续优化。这个分类很重要，因为真实项目评审里，如果所有建议都变成“马上做”，范围就会失控。

评审提示词里要让 AI 检查几类问题。第一，是否过度设计。第二，是否遗漏验收。第三，是否违反安全边界。第四，是否和 API、数据库、前端设计冲突。第五，风险重算是否有事务效果和日志追踪。第六，错误码、requestId 和操作日志是否足够支撑问题排查。

最后一轮，我们让 AI 帮忙整理主控裁决：第一阶段必须做、第二阶段再做、明确不做。

第一阶段必须做的，就是能支撑最小闭环和验收的内容。比如客户列表、详情、账户、风险重算、操作日志；统一响应和错误码；requestId 和 X-Operator；脱敏样例数据；风险重算的三项事务效果；自动化测试、接口验收、人工验收和安全检查。

第二阶段再做的，是 Gateway、customer-service、risk-service、Nacos、服务间调用和拆分后的回归验证。

明确不做的，是第一阶段不接真实认证中心，不做真实权限，不接真实核心、风控、征信、反洗钱，不做 MQ、缓存、分布式事务，不使用真实客户数据，不出现生产地址、内部系统地址、密钥和 token。

到这里，第 4 课就完成了。我们没有写代码，但我们已经把后续开发必须遵守的设计边界定清楚了。

最后总结一下。本课不是教大家用 AI 生成一份看起来很高级的架构图，而是教大家用 AI 做两件务实的事：先把设计点列全，再把风险查出来。AI 可以辅助我们提高设计和评审效率，但最终拍板的人还是我们。尤其在银行研发场景里，安全、脱敏、日志、错误码、requestId 和操作追踪，必须在设计阶段就想清楚。

## 13. 讲义精简版

### 13.1 本课核心

第 4 课承接第 3 课需求分析结果，把需求转换为第一阶段可开发设计，并用 AI 辅助做设计评审。

核心原则：

```text
先设计和评审，再开发。
第一阶段单体 POC 先跑通，第二阶段再演进微服务。
AI 辅助列方案和查风险，架构边界由人拍板。
```

### 13.2 第一阶段设计边界

第一阶段必须跑通：

- 客户列表查询。
- 客户详情查看。
- 账户列表展示。
- 当前风险等级展示。
- 样例风险重算。
- 操作日志追踪。

第一阶段技术选择：

- Spring Boot 3.x 单体后端。
- Vue 3 企业后台页面。
- MySQL 8 / H2 test profile 样例数据。
- Flyway 初始化脱敏样例数据。
- REST API + 统一响应 + 错误码 + `requestId`。

第一阶段不做：

- 真实登录认证。
- 复杂权限。
- MQ。
- 缓存集群。
- 分布式事务。
- 真实银行系统接入。
- 真实客户数据和生产配置。

### 13.3 设计必须覆盖的内容

- 模块划分：`common`、`customer`、`account`、`risk`、`operation-log`。
- 数据库：`customer`、`account`、`risk_record`、`operation_log`。
- API：客户、账户、风险、日志接口。
- 前端：仪表盘、客户列表、详情、风险重算弹窗、操作日志。
- 异常：400、404、409、500 和统一错误响应。
- 日志：`requestId`、操作人、目标业务号、结果、耗时。
- 安全：脱敏字段、样例数据、错误响应不泄露内部细节。
- 测试：单测、集成测试、接口验收、人工验收、安全检查。

### 13.4 设计评审要查什么

- 是否过度设计。
- 是否遗漏验收。
- 是否违反安全边界。
- 是否和数据库设计冲突。
- 是否和 API 契约冲突。
- 是否和前端页面需要的数据冲突。
- 是否缺少错误码、日志、`requestId` 和操作追踪。
- 风险重算是否可追踪、可验证。

### 13.5 P0/P1/P2 分类

- P0：不修不能开发或不能录屏。
- P1：开发前应修。
- P2：建议优化或后续阶段处理。

不要把所有建议都塞进第一阶段。评审的价值不只是发现问题，也包括控制范围。

### 13.6 本课最重要的一句话

```text
好的设计不是复杂，而是边界清楚、契约一致、风险可控、结果可验收。
```

## 14. 课后练习

请选择第 3 课中你自己的一个低风险小需求，把它转成一个设计提示词，并要求 AI 输出设计评审风险。

要求：

- 不包含真实客户数据、真实账号、手机号、证件号。
- 不包含生产地址、内部系统地址、密钥、token 或真实配置。
- 不要求 AI 写代码。
- 必须区分第一阶段必须做、第二阶段再做、明确不做。
- 必须要求 AI 检查安全、脱敏、日志、错误码、`requestId` 和操作追踪。

练习模板：

```text
背景：
【说明你的低风险小需求和业务场景，不要包含敏感信息】

已收敛需求：
【写清用户、功能范围、最小闭环、不做什么、验收目标】

请先不要写代码。
请帮我设计第一阶段单体 POC，输出：
1. 设计目标和边界
2. 模块划分
3. 数据流
4. 数据库表
5. API
6. 前端页面
7. 异常、错误码和日志
8. 安全脱敏
9. requestId 和操作追踪
10. 测试策略
11. 第一阶段不做什么
12. 第二阶段演进方向

约束：
- 第一阶段先做单体 POC，不引入微服务、MQ、缓存、分布式事务。
- 不接真实生产系统。
- 不使用真实客户数据、生产地址、密钥或 token。
- 如果需要假设，请标明“假设”。
```

完成后再追问一轮：

```text
请作为设计评审员，按 P0/P1/P2 审查上面的设计。
重点检查：
1. 是否过度设计
2. 是否遗漏验收标准
3. 是否违反安全和脱敏边界
4. 是否存在数据库、API、前端字段不一致
5. 是否缺少错误码、日志、requestId 和操作追踪

请只列有证据的问题，并给出第一阶段修正或第二阶段处理建议。
```

最后请自己做一次人工裁决：

```text
请把评审结果收敛成：
1. 第一阶段必须做
2. 第二阶段再做
3. 明确不做
```

## 15. 失败兜底方案

如果 AI 直接开始写代码：

- 暂停输出，重新强调“本轮只做设计和评审，不改文件、不写代码”。

如果 AI 输出过度微服务化：

- 要求它按“第一阶段单体 POC / 第二阶段微服务演进 / 第一阶段明确不做”重写。

如果 AI 只画架构图、不落验收：

- 要求每个设计点补充“对应验收方式”，例如接口测试、页面操作、日志追踪或安全检查。

如果 AI 忽略安全和脱敏：

- 要求重写安全章节，覆盖样例数据、字段命名、日志、错误响应、配置和录屏截图。

如果 AI 和现有 API / 数据库设计冲突：

- 以主控基线为准，不让 AI 临时改契约。把冲突列为评审问题，再由人裁决是否修改基线。

如果 AI 给出很多 P2 优化：

- 不要全部纳入第一阶段。把有价值但不影响当前闭环的内容放到第二阶段或课后扩展。
