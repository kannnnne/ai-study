# AI 研发工作流

## 总原则

复杂任务不能直接让 AI 开写代码。正确流程是：

```text
需求分析 -> 设计和评审 -> 拆分计划 -> 测试和验收标准 -> 开发 -> AI 验收 -> 人工验收 -> 需求验收 -> 设计验收 -> 系统架构验收 -> 安全运维验收
```

这条流程是课程最重要的方法论之一。学员真正要学会的不是某个工具按钮，而是如何让 AI 在可控流程里稳定产出。

## 阶段 1：需求分析

### 目标

让 AI 先理解问题，并暴露不确定点。

### 必须产出

- 需求摘要。
- 用户角色。
- 业务流程。
- 功能边界。
- 非功能要求。
- 待确认问题。
- 不做什么。

### 禁止事项

- 不允许 AI 直接创建代码。
- 不允许跳过待确认问题。
- 不允许把“猜测”写成“事实”。

## 阶段 2：设计和评审

### 目标

把需求转成系统设计，并在开发前发现明显风险。

### 必须产出

- 架构设计。
- 模块边界。
- 数据库设计。
- API 设计。
- 异常和响应码设计。
- 安全和权限设计。
- 测试策略。

### 评审角度

- 需求是否覆盖。
- 设计是否过度。
- 模块边界是否清晰。
- 数据一致性是否可控。
- 日志是否可能泄露敏感信息。
- 是否容易测试。

## 阶段 3：拆分计划

### 目标

把任务拆成可执行的小步，每一步都有清晰结束条件。

### 好计划应该长这样

- 每一步只处理一个主题。
- 每一步都有输入和输出。
- 每一步能单独验证。
- 失败时能定位是哪一步失败。

## 阶段 4：测试和验收标准

### 目标

先定义“怎么算完成”，再开始开发。

### 验收类型

- 需求验收：功能是否符合需求。
- 设计验收：实现是否符合设计。
- 测试验收：单测、接口测试、关键路径是否通过。
- AI 验收：让另一个干净会话审查实现。
- 人工验收：讲师或开发者手动操作确认。
- 架构验收：服务边界、依赖方向、配置管理是否合理。
- 安全运维验收：密钥、日志、配置、部署、监控、错误处理是否合规。

## 阶段 5：开发

### 推荐方式

- 一次只让 AI 做一个小任务。
- 每次修改后看 diff。
- 每次关键修改后运行测试。
- 不满意就回到计划，不要在代码里乱修。

## 阶段 6：AI 验收

### 做法

开一个新会话，把需求、设计、关键代码和测试结果交给 AI，让它站在审查者角度验收。

### 要求

- 按严重程度输出问题。
- 只列有证据的问题。
- 明确哪些是必须修，哪些是建议优化。
- 输出补充测试建议。

## 阶段 7：人工验收

### 做法

由人根据验收清单手动验证。AI 可以帮忙生成清单，但不能替人承担最终责任。

### 重点

- 页面能否正常操作。
- 接口返回是否符合预期。
- 错误场景是否处理。
- 日志是否可排查。
- 文档是否和代码一致。

## 会话管理原则

- 一个会话只做一个主线任务。
- 主控会话负责目标和验收。
- 子会话负责单点执行。
- 上下文变脏就开新会话。
- 复杂审查要用干净会话。
- 不同角色分不同会话：需求、架构、前端、后端、测试、安全。

## 环境和工具偏好

推荐顺序：

1. macOS + zsh + CLI。
2. Windows + WSL2 + Ubuntu + CLI。
3. Windows + Git Bash。
4. Windows + PowerShell。
5. 桌面端。

理由：

- CLI 更贴近真实研发流程。
- macOS 和 Linux 工具链更顺滑。
- WSL2 比 PowerShell 更接近 Linux 部署环境。
- 桌面端适合展示和轻量任务，但复杂工程任务建议优先 CLI。

## 课程要反复强调的一句话

AI 不是替你负责的开发者，而是可以高效执行、协助分析和辅助审查的工程搭档。真正的责任边界仍然在人。
