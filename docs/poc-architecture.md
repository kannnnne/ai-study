# POC 架构设计基线

## 主控验收结论

本架构设计通过主控窗口验收，可以作为后续后端、数据库、前端、测试验收和讲稿子窗口的设计基线。

通过理由：

- POC 定位清楚：教学用、最小可运行、银行研发语境。
- 第一阶段和第二阶段边界清楚，避免一开始把课程做重。
- 数据模型能覆盖客户、账户、风险评级、操作日志四类核心能力。
- API 能覆盖列表、详情、维护、风险重算和日志查询。
- 安全边界明确：只使用脱敏样例数据，不出现真实密钥、客户信息、生产地址和内部接口。
- 设计适合录屏拆解，每个阶段都能解释、运行和验收。

## 主控裁决

为减少后续子窗口反复确认，先确定以下默认选择：

- 数据库：MySQL 8。
- ORM：MyBatis Plus，贴近多数 Java 企业后台团队习惯。
- 第一阶段认证：不做真实登录，使用固定操作人或 `X-Operator` 请求头模拟。
- 第二阶段数据库：先共享库分表，重点演示服务拆分、Gateway、Nacos 和服务间调用；独立库作为进阶扩展。
- 课程节奏：优先按 8 节主线设计，必要时拆成 10 节完整版。
- 前端：Vue 3 + TypeScript + Vite + Element Plus，ECharts 用于风险分布图。

## 1. POC 一句话定位

一个面向银行研发教学的“客户风险评级与账户查询”最小可运行系统，用于演示如何借助 AI 从 0 完成需求、设计、前后端开发、数据库、测试验收和微服务演进。

## 2. 业务场景

柜面、运营或风控辅助人员需要查询客户基础信息、账户余额概览和当前风险等级。系统支持按客户号、客户名称、风险等级筛选客户，查看客户详情，并允许管理员维护客户信息、触发风险等级重新计算、查看操作日志。

业务规则保持轻量：风险等级由样例规则计算，例如风险分数、客户状态、账户异常标记等组合得到低、中、高三档。不接入真实征信、反洗钱、核心账务或生产风控模型。

## 3. 用户角色和核心流程

角色：

- 查询人员：查询客户列表、查看客户详情、查看账户和风险等级。
- 管理人员：维护客户基础信息、触发风险重算、查看操作日志。
- 讲师 / 学员：通过系统理解 AI 辅助研发工作流。

核心流程：

1. 模拟进入系统。
2. 在客户列表按条件筛选。
3. 进入客户详情，查看基础信息、账户信息、风险等级、最近评级记录。
4. 管理人员修改客户信息。
5. 触发风险重新计算。
6. 系统记录操作日志。
7. 前端刷新展示最新风险结果。

## 4. 第一阶段单体架构

第一阶段使用一个 Spring Boot 3.x 单体应用，内部按模块分层：

```text
Vue 3 前端
  -> Spring Boot 单体后端
      -> common 模块
      -> customer 模块
      -> account 模块
      -> risk 模块
      -> operation-log 模块
      -> openapi 配置
  -> MySQL 8
```

第一阶段包含：

- REST API。
- 参数校验。
- 统一响应。
- 统一异常处理。
- 基础操作日志。
- OpenAPI 文档。
- Flyway 初始化表和样例数据。
- 单元测试、接口测试。
- 本地 Docker Compose 数据库。

第一阶段不做：

- 真实登录认证。
- 复杂权限模型。
- 分布式事务。
- MQ。
- 缓存集群。
- 真实风控模型。
- 真实银行系统对接。

## 5. 第二阶段微服务演进架构

第二阶段用于教学“为什么拆、怎么拆、拆完如何验证”：

```text
Vue 3 前端
  -> gateway-service
      -> customer-service
          -> customer / account 数据
      -> risk-service
          -> risk_record / 风险计算
  -> Nacos 注册与配置
  -> MySQL 8：先共享库分表，进阶再演示按服务独立库
```

建议只拆 3 个服务：

- `gateway-service`：统一入口、路由、跨域、简单鉴权占位。
- `customer-service`：客户资料、账户查询。
- `risk-service`：风险等级查询、风险重算、评级记录。

第二阶段不拆 `account-service`、`audit-service`，避免入门课程复杂度失控。操作日志先保留在各服务本地表或作为扩展题。

## 6. 模块职责

第一阶段单体模块：

- `common`：统一响应、错误码、异常、分页对象、日志工具。
- `customer`：客户查询、新增、修改、详情聚合。
- `account`：账户余额、账户状态、账户列表。
- `risk`：风险等级展示、风险重算、风险记录。
- `operation-log`：记录查询、修改、重算等操作。
- `openapi`：接口文档配置。

第二阶段服务：

- `gateway-service`：API 网关、路由转发、统一入口。
- `customer-service`：客户和账户主数据查询。
- `risk-service`：风险评级规则、风险记录、重算接口。
- `frontend`：企业后台式查询与演示界面。

## 8. REST API 清单

客户：

- `GET /api/customers`：分页查询客户列表。
- `GET /api/customers/{customerNo}`：客户详情。
- `POST /api/customers`：新增客户。
- `PUT /api/customers/{customerNo}`：修改客户。

账户：

- `GET /api/customers/{customerNo}/accounts`：查询客户账户列表。

风险：

- `GET /api/customers/{customerNo}/risk`：查询当前风险等级。
- `GET /api/customers/{customerNo}/risk-records`：查询历史评级记录。
- `POST /api/customers/{customerNo}/risk/recalculate`：触发风险重算。

日志：

- `GET /api/operation-logs`：查询操作日志。

支撑：

- `GET /actuator/health`：健康检查。
- `GET /v3/api-docs`：OpenAPI JSON。
- `GET /swagger-ui`：Swagger UI。

## 9. 统一响应、异常、日志和安全

统一响应：

```json
{
  "success": true,
  "code": "000000",
  "message": "success",
  "data": {},
  "requestId": "demo-request-id",
  "timestamp": "2026-06-21T10:00:00+08:00"
}
```

错误码：

- `000000`：成功。
- `400001`：参数校验失败。
- `404001`：客户不存在。
- `409001`：业务状态冲突。
- `500000`：系统异常。

异常：

- 参数异常返回 400。
- 资源不存在返回 404。
- 业务规则不满足返回 409。
- 未预期异常返回 500，但不暴露堆栈给前端。

日志：

- 每次请求生成 `requestId`。
- 记录接口、耗时、结果、关键业务号。
- 不打印手机号、证件号、完整账号、密钥、数据库连接串。
- 错误日志保留排查信息，但敏感字段必须脱敏。

安全：

- 第一阶段使用固定操作人或 `X-Operator` 请求头模拟。
- 第二阶段在网关层增加鉴权占位。
- 不接入真实认证中心。
- 配置文件只放本地样例，不提交真实密钥。

## 10. 前端页面结构

- 首页 / 仪表盘：客户总数、风险等级分布、账户余额概览。
- 客户列表页：筛选区、表格、风险等级标签、查看详情按钮。
- 客户详情页或抽屉：基础信息、账户列表、当前风险、历史评级记录。
- 客户维护弹窗：新增 / 修改客户基础信息。
- 风险重算确认弹窗：展示重算结果。
- 操作日志页：按操作人、类型、时间筛选。

视觉方向：企业后台、克制、清晰、可演示。避免营销页和炫技页。

## 11. 测试策略

后端：

- Service 单元测试：客户查询、风险计算、异常分支。
- Controller 接口测试：参数校验、正常响应、错误响应。
- Repository 测试：基础查询和分页。
- 风险规则边界测试：低 / 中 / 高等级。

前端：

- 页面渲染测试可选。
- 重点做手工验收：筛选、详情、重算、错误提示。

接口：

- 使用 OpenAPI、Postman 或 HTTP 文件保存可复现请求。
- 每个接口准备成功、参数错误、资源不存在三类样例。

验收：

- AI 验收：新会话审查需求、设计、代码、测试结果。
- 人工验收：讲师按清单手动操作。
- 安全验收：检查样例数据、日志、配置、接口返回是否泄露敏感信息。

## 12. 课程录制拆分

1. 需求分析：明确角色、功能边界、不做什么。
2. 架构设计：单体优先、微服务后置、设计评审。
3. 后端最小服务：客户列表、详情、统一响应。
4. 数据库：表结构、Flyway、脱敏样例数据。
5. 风险评级：规则计算、历史记录、测试。
6. 前端页面：列表、详情、风险展示、联调。
7. 微服务演进：Gateway、Nacos、customer-service、risk-service。
8. 验收复盘：AI 审查、人工验收、安全检查、课程方法沉淀。

## 13. 风险点

- 业务设计过重，导致入门课变成银行核心系统课。
- 微服务拆分过早，学员还没理解单体链路就进入治理复杂度。
- 日志、样例数据、配置不注意脱敏。
- 风险评级规则被误解成真实风控模型。
- 前端页面做成营销页或炫技页，不适合企业后台演示。

## 14. 后续可派发子任务

- 数据库设计子窗口：细化 MySQL DDL、Flyway 脚本和样例数据。

- 前端设计子窗口：设计 Vue 3 企业后台界面和联调计划。
- 测试验收子窗口：生成测试矩阵、验收清单和 HTTP 请求样例。
- 讲稿子窗口：把本架构设计转成第 4 课录屏脚本。
