# P1 内部试讲组织方案

## 1. 试讲目标

本方案用于组织 3 到 5 人小范围内部试讲，验证 AI 编程入门课程是否能被目标学员听懂、跟做、复盘，并形成进入正式推广前的改进清单。

试讲不是正式发布，也不是完整生产培训。它的目标是：

- 验证课程主线是否连贯：课程导入、工作流、环境、需求、设计、后端、联调、验收能否形成闭环。
- 验证学员能否理解并复现关键步骤，尤其是环境配置、AI 任务拆分、POC 跟做和验收。
- 验证讲师是否能稳定讲清公司价值、团队效率、研发质量和风险控制。
- 收集试讲反馈，判断是否进入 P2 正式发布版打磨。

## 2. 适合参与的人群

建议 3 到 5 人，角色尽量有差异：

| 角色 | 建议人数 | 选择标准 | 主要观察点 |
| --- | --- | --- | --- |
| 技术骨干 | 1 到 2 人 | 熟悉 Java、前后端或项目交付流程 | 判断技术路线、代码质量、验收方式是否站得住 |
| AI 使用经验较少的研发同学 | 1 到 2 人 | 有开发经验，但不常用 Agent 工具 | 判断课程是否易懂、是否能跟做 |
| 项目或团队负责人 | 0 到 1 人 | 关注团队推广和风险控制 | 判断价值表达、推广成本和组织可行性 |
| 质量或测试同学 | 0 到 1 人 | 关注测试、验收和缺陷复盘 | 判断验收材料是否可执行 |

不建议第一轮试讲人数过多。人数超过 5 人后，环境问题和讨论分歧会明显增加，不利于快速收敛。

## 3. 试讲前准备

### 3.1 讲师准备

- 确认本轮试讲只使用脱敏样例数据和本地 POC，不展示真实客户、真实账号、内部系统地址、密钥或生产配置。
- 准备课程材料路径：`docs/lesson-00-recording-material.md`、`docs/lesson-01-workflow-recording-material.md`、`docs/lesson-02-setup-recording-material.md`、`docs/lesson-03-requirement-analysis-recording-material.md`、`docs/lesson-04-design-review-recording-material.md`、`docs/lesson-05-backend-recording-material.md`、`docs/lesson-06-recording-material.md`、`docs/lesson-09-acceptance-recording-material.md`。
- 准备验收材料：`docs/test-acceptance-material.md`、`docs/fullstack-integration-acceptance.md`、`docs/http/customer-risk-poc.http`。
- 复核 Claude Code 和 DeepSeek 当前官方安装方式，确认环境变量名称、base URL、模型名和公司网络可用性。
- 准备一套讲师本机可运行环境，优先使用后端 test profile + H2 和前端 Vite。
- 准备失败兜底截图或历史验收记录，避免现场环境问题中断课程。
- 打印或共享 `docs/pilot-feedback-form.md`。

### 3.2 学员准备

- 自带电脑，提前确认 Node.js、npm、Java、Maven、Git、终端工具是否可用。
- 如果使用 Windows，优先准备 WSL2；无法使用 WSL2 的同学准备 PowerShell 路径。
- 准备个人可用的模型访问方式。试讲中不收集、不展示、不共享真实 token。
- 提前阅读 `README.md`、`docs/course-blueprint.md` 和 `docs/lesson-plan.md` 的前半部分即可，不要求完整读完所有录屏稿。
- 明确试讲反馈目标：不是评价讲师个人，而是帮助课程进入可推广状态。

## 4. 推荐课程顺序

P1 内部试讲建议按以下顺序组织。若时间不足，可以拆成两次，每次 2 到 3 小时。

| 顺序 | 课程 | 材料路径 | 建议试讲时长 | 试讲目标 | 验收点 |
| --- | --- | --- | --- | --- | --- |
| 1 | 第 0 课：课程导入和预期管理 | `docs/lesson-00-recording-material.md` | 15 到 20 分钟 | 让参与者理解课程定位、团队价值、交付物和安全边界 | 参与者能说清课程不是算法课、不是工具炫技，而是 AI 辅助研发工作流 |
| 2 | 第 1 课：AI 编程基础概念和工作流 | `docs/lesson-01-workflow-recording-material.md` | 25 到 30 分钟 | 建立提示词、上下文、Agent、主控/子会话和验收意识 | 参与者能复述“先分析、再设计、设验收、再开发” |
| 3 | 第 2 课：Claude Code + DeepSeek 环境搭建 | `docs/lesson-02-setup-recording-material.md`、`docs/setup-faq.md` | 30 到 45 分钟 | 验证学员能理解安装、API 配置、网络限制和安全保管 | 至少 70% 学员能完成基础检查，未完成者能定位卡点 |
| 4 | 第 3 课：用 AI 做需求分析 | `docs/lesson-03-requirement-analysis-recording-material.md` | 30 到 35 分钟 | 演示从粗需求到角色、边界、流程、验收标准 | 学员能写出一段不含敏感信息的需求分析提示词 |
| 5 | 第 4 课：用 AI 做系统设计和评审 | `docs/lesson-04-design-review-recording-material.md` | 35 到 40 分钟 | 把需求转成单体 POC 设计，并识别过度设计和安全遗漏 | 学员能指出第一阶段必须做、第二阶段再做、明确不做的内容 |
| 6 | 第 5 课：后端最小服务开发 | `docs/lesson-05-backend-recording-material.md` | 40 到 50 分钟 | 看 AI 如何辅助 Spring Boot 后端实现，同时由人控制契约、事务和测试 | 能理解统一响应、错误码、`X-Request-Id`、风险重算事务三项效果 |
| 7 | 第 6 课：前端页面和接口联调 | `docs/lesson-06-recording-material.md` | 25 到 30 分钟 | 跑通仪表盘、客户列表、详情、风险重算、操作日志和错误处理 | 学员能按页面主流程解释请求、响应、状态和日志链路 |
| 8 | 第 9 课：AI 验收、人工验收和安全复盘 | `docs/lesson-09-acceptance-recording-material.md`、`docs/test-acceptance-material.md` | 35 到 45 分钟 | 把 AI 审查、自动化测试、HTTP 复验、浏览器验收、安全检查合成质量闸门 | 学员能按清单判断 POC 是否通过录屏前验收 |

## 5. 两种试讲排期

### 5.1 一次性半天试讲

适合领导已经认可方向，只需要快速打通 P1：

| 时间 | 内容 |
| --- | --- |
| 00:00-00:10 | 开场、角色说明、反馈表说明 |
| 00:10-00:30 | 第 0 课 |
| 00:30-01:00 | 第 1 课 |
| 01:00-01:40 | 第 2 课环境安装和问题记录 |
| 01:40-01:50 | 休息 |
| 01:50-02:25 | 第 3 课需求分析 |
| 02:25-03:05 | 第 4 课设计评审 |
| 03:05-03:45 | 第 5 课后端开发重点 |
| 03:45-04:15 | 第 6 课联调演示 |
| 04:15-04:50 | 第 9 课验收复盘 |
| 04:50-05:20 | 填写反馈表、现场讨论、确认改进项 |

### 5.2 两次试讲

适合环境差异较大或需要学员课后跟做：

| 场次 | 内容 | 目标 |
| --- | --- | --- |
| 第一次，2.5 小时 | 第 0、1、2、3 课 | 建立认知、跑通工具、学会需求分析 |
| 第二次，3 小时 | 第 4、5、6、9 课 | 看设计、开发、联调、验收闭环 |

两次之间建议间隔 2 到 5 个工作日，让学员完成环境安装和一个低风险需求分析练习。

## 6. 讲师准备清单

试讲前 1 天：

- [ ] 复核课程顺序、每节课材料路径和试讲时长。
- [ ] 复跑后端 `mvn test` 或准备最近一次通过记录。
- [ ] 复跑前端 `npm run build` 或准备最近一次通过记录。
- [ ] 启动后端 test profile + H2，确认健康检查可访问。
- [ ] 启动前端 Vite，确认 `/dashboard`、`/customers`、`/customers/CUST100002` 可访问。
- [ ] 准备 `docs/http/customer-risk-poc.http`，确认健康检查、客户详情、风险重算、404、409、400 请求路径清楚。
- [ ] 检查截图、终端、浏览器、HTTP 请求中没有真实 token、内部地址、生产配置和真实客户数据。
- [ ] 准备环境失败、token 不可用、前后端启动失败时的备用讲解路径。

试讲当天：

- [ ] 先声明安全边界和反馈方式。
- [ ] 不临时扩大课程范围，不现场连接真实系统。
- [ ] 每节课结束时记录 1 到 3 个反馈点。
- [ ] 对环境问题单独记录，不让单个学员机器拖垮主线。
- [ ] 试讲结束后收齐反馈表。

## 7. 学员准备清单

- [ ] 明确本次试讲使用本地 POC 和脱敏样例数据。
- [ ] 不在提示词、截图或共享屏幕中输入真实客户信息、内部系统地址、密钥或生产配置。
- [ ] 准备开发环境：Java、Maven、Node.js、npm、Git、终端。
- [ ] 准备可访问模型的方式；如没有 token，先参加旁听和流程理解。
- [ ] 试讲前阅读 `README.md` 和 `docs/course-blueprint.md`。
- [ ] 试讲中记录看不懂、跟不上、无法复现、觉得有风险的点。

## 8. 试讲当天流程

1. 讲师开场：说明试讲目标、课程边界、反馈方式和当天安排。
2. 参与者自报背景：开发方向、AI 使用经验、当前环境是否已准备。
3. 按课程顺序试讲，每节课控制一个核心目标。
4. 环境问题统一登记：现场只处理共性问题，个别机器问题课后单独处理。
5. POC 演示使用固定脱敏样例客户，例如 `CUST100002`，不引入真实数据。
6. 每节课后用 2 分钟确认：是否听懂、是否能跟做、是否有阻断问题。
7. 最后 20 到 30 分钟填写反馈表并讨论是否进入下一轮。
8. 讲师当天整理问题清单，区分 P0、P1、P2。

## 9. 试讲后反馈收集和复盘方式

试讲后当天完成：

- 收集 `docs/pilot-feedback-form.md` 中的学员反馈。
- 讲师填写反馈表中的讲师自评区。
- 把问题按类型归类：课程理解、环境安装、POC 跟做、AI 工作流、验收安全、推广建议。
- 把问题按优先级归类：
  - P0：阻断试讲或存在安全风险，必须立即修复。
  - P1：影响内部试讲效果，正式推广前应修复。
  - P2：优化项，可放到正式版或后续 FAQ。

试讲后 2 个工作日内完成：

- 输出一页复盘结论：是否建议进入正式推广准备。
- 更新课程问题清单和 FAQ。
- 明确下一轮修订范围，避免把 P2 大改提前塞回 P1。

## 10. 是否进入正式推广的判断标准

建议满足以下条件后，再进入 P2 正式发布版打磨：

| 维度 | 进入条件 |
| --- | --- |
| 价值认可 | 领导或技术骨干认可课程方向，认为值得继续投入 |
| 听懂程度 | 试讲参与者中至少 80% 能说清课程主线和 AI 工作流边界 |
| 环境可行 | 至少 70% 参与者能完成基础环境检查；失败项有明确 FAQ 或兜底路径 |
| POC 可演示 | 讲师本机能稳定跑通后端 test profile、前端页面和主流程演示 |
| 跟做可行 | 大多数学员能完成需求分析或验收清单练习，复杂开发跟做可分阶段推进 |
| 验收完整 | 自动化测试、HTTP 复验、浏览器验收、日志追踪和安全脱敏检查有清单 |
| 风险可控 | 未发现真实数据、密钥、生产地址、内部系统细节进入课程材料或录屏 |
| 反馈闭环 | P0 问题清零，P1 问题有明确修订计划，P2 问题不阻断推广准备 |

不建议进入正式推广的情况：

- 安全边界讲不清，参与者误以为可以直接上传真实项目信息。
- 环境安装大面积失败，且无法判断是安装包、网络、token 还是模型访问问题。
- POC 主流程无法稳定演示。
- 学员普遍只记住“让 AI 写代码”，没有理解需求、设计、验收和人工把关。

## 11. 风险和兜底方案

| 风险 | 表现 | 兜底方案 |
| --- | --- | --- |
| 环境装不上 | Node、npm、Claude Code、Java、Maven 或 WSL2 安装失败 | 先区分安装包下载、权限、路径、网络和版本问题；现场只处理共性问题，个别机器课后按 `docs/setup-faq.md` 跟进 |
| DeepSeek token 不可用 | API key 缺失、base URL 配错、模型访问失败、网络不通 | 讲师使用占位符讲配置方法，不展示真实 token；现场用讲师环境演示，学员问题登记后单独排查 |
| 后端起不来 | 端口占用、MySQL 未启动、profile 用错、依赖未下载 | 默认使用 test profile + H2；MySQL 作为可选路径；端口占用优先关闭占用进程，不临时改端口影响课程 |
| 前端联调失败 | Vite 启动失败、代理错误、后端健康检查不通、页面 Network Error | 先查后端健康检查，再查 Vite 代理和请求路径；必要时展示 `docs/fullstack-integration-acceptance.md` 的既有验收记录 |
| 学员基础差异大 | 有人能跟做，有人还在安装或概念理解阶段 | 分层要求：初学者完成理解和简单提示词练习，骨干完成 POC 验收和代码审查；不要让所有人同时做完整开发 |
| 领导只关心价值不关心细节 | 技术细节过多导致价值感下降 | 先用第 0 课和第 6 课展示价值与闭环；第 5、9 课作为技术骨干材料，领导只看结论和风险控制 |
| AI 输出不稳定 | 现场生成内容跑偏或质量差 | 使用已整理录屏材料和提示词模板；现场强调 AI 输出是草案，讲人工收敛和验收，不把现场随机输出作为唯一证据 |
| 触碰敏感边界 | 误贴真实数据、内部地址、密钥、生产配置 | 立即停止展示并清理；复盘时列为 P0；后续录屏只使用脱敏样例和占位符 |
