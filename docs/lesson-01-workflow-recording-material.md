# 第 1 课录屏材料：AI 编程基础概念和工作流

## 1. 标题与目标

标题：第 1 课：AI 编程基础概念和工作流

一句话目标：让学员理解 AI 编程工具的能力边界，并掌握一套“先分析、再设计、拆任务、设验收、再开发”的安全可控研发工作流。

## 2. 本课要解决的问题

第 0 课已经讲清楚课程定位和预期管理。第 1 课要继续往前走一步：回答“为什么不能一上来就让 AI 写完整系统”。

本课要解决四个问题：

- AI 到底是什么：它不是懂业务全貌的项目负责人，而是基于上下文生成结果的通用助手。
- AI 能做什么：辅助理解需求、整理文档、生成代码、补测试、查问题、做初步审查。
- AI 不能替代什么：不能替代需求确认、架构决策、代码审查、测试验收、安全判断和上线责任。
- AI 怎么安全进入研发流程：必须放在需求分析、设计评审、计划拆分、测试验收和人工把关组成的工程流程里。

本课核心口径：

```text
AI 是研发搭档，不是最终负责人。
真正的质量和安全，仍然由人和工程流程承担。
```

## 5. 基础概念通俗解释

### 5.1 AI / 大模型 / LLM

在这门课里，可以把 AI 理解为“能根据输入上下文生成文字、代码、计划和检查意见的助手”。

大模型是支撑这个能力的模型。它从大量文本和代码中学到语言、结构和模式，因此能回答问题、补全代码、解释报错、生成文档。

LLM 是 Large Language Model，也就是大语言模型。我们不在入门课里深挖训练算法，只需要知道：LLM 的核心工作方式是读入上下文，然后生成它认为最合理的后续内容。

对 Java / 银行项目研发同学来说，可以先这样理解：

- 它像一个很快的助理，能读很多材料，也能生成初稿。
- 它不等于业务负责人，不知道你们项目里所有历史约束。
- 它会犯错，尤其在信息不完整、上下文太乱、验收标准不清时更容易犯错。

### 5.2 提示词不是咒语

提示词不是“神奇口令”，不是写一句“请帮我写一个完整系统”就能稳定得到好结果。

好的提示词更像一份小型任务说明，至少包括：

- 背景：为什么做这个任务，项目处在什么阶段。
- 目标：这次到底要产出什么。
- 输入材料：需要阅读哪些文档、代码、接口或日志。
- 输出格式：希望按什么结构返回。
- 约束边界：不能改什么，不能泄露什么，不要做哪些扩展。
- 验收标准：做到什么程度才算完成。

可以对学员这样说：

```text
提示词不是让 AI 猜你的心思，而是把任务、上下文、约束和验收标准讲清楚。
```

### 5.3 上下文窗口是什么，为什么会跑偏

上下文窗口可以理解为 AI 当前能“看见”的材料范围。

它包括：

- 当前会话里你说过的话。
- 你贴进去的需求、代码、报错和文档。
- 工具读取到的文件内容。
- AI 自己前面生成过的计划和结论。

上下文窗口不是无限大的。会话变长以后，早期信息可能被压缩、遗漏或理解变形；任务混得太多以后，AI 也容易把前一个任务的规则带到下一个任务里。

常见跑偏表现：

- 已经切换主题了，AI 还在按上一个任务回答。
- 明明只让写讲义，它开始建议改后端代码。
- 明明要求用脱敏样例，它开始编真实感很强的数据。
- 多轮修复后，它开始重复同一套错误方案。

处理原则：

- 一个会话只保留一条主线任务。
- 上下文变脏时，不要硬聊，开新会话。
- 复杂审查用干净会话，让 AI 重新站在审查者角度看问题。

### 5.4 Agent 可以理解为工作流

Agent 不要讲得太玄。入门阶段可以这样解释：

```text
Agent 可以理解为一个会分步骤、会调用工具、会检查结果的 AI 工作流。
```

普通聊天更像“你问一句，它答一句”。Agent 模式下，AI 往往会做这些事：

- 先读文件，理解项目现状。
- 给出计划，拆分步骤。
- 修改文件或生成材料。
- 运行命令或测试。
- 根据结果继续修正。
- 最后总结改了什么、验证了什么。

但 Agent 会调用工具，不代表它就一定可靠。工具让它更接近真实研发流程，同时也带来更高责任要求：文件有没有改错、命令有没有跑对、测试结果是否可信，都需要人验收。

### 5.5 工具调用、文件读写、命令执行、浏览器验收

工具调用：AI 不只是聊天，还能通过工具读取文件、搜索代码、运行命令、操作浏览器或访问外部资料。工具越强，越要明确边界。

文件读写：AI 可以读取项目文档和代码，也可以创建或修改文件。读写前要明确范围，避免它改到无关模块。

命令执行：AI 可以运行测试、构建、搜索、格式化等命令。命令输出是验收依据之一，但不能只看“命令跑过”，还要看跑的是不是正确命令、覆盖的是不是关键路径。

浏览器验收：对前端和联调任务，AI 可以打开页面、点击操作、截图或检查接口请求。浏览器验收能发现很多构建命令发现不了的问题，但仍然需要人工确认业务含义。

本项目里可以对应到：

- 读文件：读取 `docs/api-design.md`、`docs/poc-architecture.md`、`docs/master-task-board.md`。
- 写文件：新增课程材料、任务提示词、验收记录。
- 执行命令：后端 `mvn test`、前端 `npm run build`。
- 浏览器验收：打开仪表盘、客户列表、详情页、风险重算和操作日志。

## 6. 正确的 AI 辅助研发工作流

本课程反复强调的流程是：

```text
需求分析 -> 设计和评审 -> 拆分计划 -> 测试和验收标准 -> 开发 -> AI 验收 -> 人工验收 -> 安全运维验收
```

### 6.1 需求分析

目标：先让 AI 理解问题，而不是直接写代码。

应该产出：

- 需求摘要。
- 用户角色。
- 业务流程。
- 功能边界。
- 非功能要求。
- 待确认问题。
- 明确不做什么。

讲课时可以强调：如果需求没有澄清，AI 只能靠猜。它猜得越像真的，风险越高。

### 6.2 设计和评审

目标：把需求转成可开发设计，并在开发前发现明显风险。

应该产出：

- 架构设计。
- 模块边界。
- 数据库设计。
- API 设计。
- 异常、响应码和日志设计。
- 安全和权限设计。
- 测试策略。

银行项目里尤其要关注金额、日期、状态、异常、日志、敏感字段和幂等边界。AI 可以出设计草案，但架构取舍必须由人把关。

### 6.3 拆分计划

目标：把大任务拆成可以单独执行和单独验收的小任务。

好计划的特点：

- 每一步只处理一个主题。
- 每一步有明确输入和输出。
- 每一步能单独验证。
- 失败时能定位是哪一步失败。

例如不要说“帮我做一个客户风险评级系统”，而要拆成需求分析、架构设计、数据库设计、API 设计、后端实现、前端实现、联调验收、录屏材料。

### 6.4 测试和验收标准

目标：先定义怎么算完成，再开始开发。

验收标准可以包括：

- 自动化测试是否通过。
- API 是否符合契约。
- 页面是否能完成关键操作。
- 错误场景是否可复现。
- 日志是否可追踪。
- 配置和样例数据是否安全脱敏。
- 文档是否和实现一致。

核心口径：

```text
AI 写完不是完成，验收通过才是完成。
```

### 6.5 开发

开发阶段建议：

- 一次只让 AI 做一个小任务。
- 修改前先让 AI 说明影响范围。
- 修改后看 diff。
- 关键修改后跑测试。
- 不满意时回到计划，不要在代码里乱修。

对学员要讲清楚：AI 开发不是“放手让它写”，而是“人在流程里控制输入、边界和验收”。

### 6.6 AI 验收

AI 验收建议开干净会话，让另一个 AI 站在审查者角度看需求、设计、代码摘要和测试结果。

要求它按严重程度输出：

- P0：必须修复，否则不能验收。
- P1：建议本轮修复。
- P2：后续优化。
- 已通过项。
- 需要人工确认的问题。

AI 验收的价值是补充视角，不是替代人工审查。

### 6.7 人工验收

人工验收由讲师、开发者或评审人根据清单手动确认。

重点看：

- 业务流程是否真的符合需求。
- 页面能不能正常操作。
- 接口返回是否符合预期。
- 错误提示是否合理。
- 日志是否可排查。
- 测试结果是否能复现。

### 6.8 安全运维验收

银行项目里，安全和运维边界必须单独验收。

至少检查：

- 没有真实客户数据。
- 没有真实手机号、证件号、账号、客户姓名。
- 没有生产地址、内部系统地址、密钥、token、私钥。
- 日志不打印完整敏感字段。
- 错误响应不暴露 SQL、堆栈和连接串。
- 配置文件只包含本地 demo 或脱敏样例。

## 7. 主控会话和子会话分工

### 7.1 为什么要拆会话

复杂项目如果全塞进一个会话，会出现三个问题：

- 上下文越来越长，AI 容易忘记早期约束。
- 多个任务混在一起，AI 容易串线。
- 验收困难，不知道一个问题到底来自需求、设计、代码还是测试。

所以本课程采用“主控会话 + 子会话”的方式。

### 7.2 主控会话负责什么

主控会话像项目经理加总架构师，负责：

- 维护课程目标和边界。
- 拆分子任务。
- 生成子窗口提示词。
- 维护路线图和任务看板。
- 验收子窗口产出。
- 判断是否返工、通过或进入下一轮。
- 最后提交和同步已验收成果。

本项目对应材料：

- `docs/course-completion-roadmap.md`
- `docs/master-control.md`
- `docs/master-task-board.md`
- `docs/subsession-prompts.md`

### 7.3 子会话负责什么

子会话只负责一个单点产出，例如：

- POC 架构设计。
- 数据库设计。
- API 设计。
- 后端代码实现。
- 前端页面设计。
- 前后端联调验收。
- 测试验收材料。
- 第 0 课、第 1 课、第 6 课录屏材料。

子会话的结束条件要清楚：产出指定文件，说明覆盖范围，列出风险和验收清单，回传给主控。

### 7.4 什么时候新开会话

出现以下情况就应该新开会话：

- 当前任务已经换主题。
- 一个会话里塞了多个不相关任务。
- AI 开始反复误解或重复错误。
- 需要切换角色，例如从后端开发切到课程编导。
- 需要独立审查，避免前面开发过程影响判断。

一句话：

```text
上下文跑偏时，不要硬聊，开新会话。
```

## 8. 本项目真实例子

本项目本身就是第 1 课最好的例子，因为课程材料、POC 设计、代码实现和验收都按主控 / 子会话方式推进。

### 8.1 POC 架构设计子窗口

主控先派发 POC 架构设计子窗口，要求设计客户风险评级与账户查询 POC。

产出包括：

- 业务场景和功能边界。
- 第一阶段单体架构和第二阶段微服务演进。
- 数据库表设计方向。
- API 清单。
- 异常、响应码、日志和测试策略。
- 安全和脱敏注意事项。

这一步解决的是“先想清楚要做什么系统，不急着写代码”。

### 8.2 数据库设计子窗口

数据库设计子窗口专注表结构、初始化数据和迁移脚本草案。

主控要求它只做数据库设计，不顺手改后端、不扩展真实环境。

这一步解决的是：银行场景里的金额、日期、状态、风险记录、操作日志和样例数据要有明确结构。

### 8.3 API 设计子窗口

API 设计子窗口专注接口契约。

产出包括：

- 客户列表、客户详情、账户、当前风险、风险历史、风险重算、操作日志。
- 统一响应结构。
- 错误码。
- 分页和请求头。
- 404、409、参数校验等失败场景。

这一步让后端、前端和测试有共同基线，避免开发时各写各的。

### 8.4 后端 / 前端 / 联调 / 录屏材料子窗口

后端子窗口负责 Spring Boot 最小实现和测试。

前端子窗口负责 Vue 3 页面设计和实现。

联调验收子窗口负责浏览器闭环：仪表盘、客户列表、详情页、风险重算、操作日志。

录屏材料子窗口负责把技术过程改写成适合讲课的口播稿、录屏步骤和讲义。

每个子窗口都只处理一类任务，这样主控才能清楚验收。

### 8.5 主控如何验收和提交

主控窗口不会只看“文件生成了没有”，还会检查：

- 是否符合课程主线。
- 是否只做了本轮任务。
- 是否能用于录屏或讲义。
- 是否与 POC 设计、API 契约和测试材料一致。
- 是否没有真实客户数据、密钥和生产地址。
- 是否有明确的验证命令或验收清单。

通过后再更新任务看板、路线图，并提交已验收变更。

## 9. 好提示词的结构

可以给学员一个通用结构：

```text
背景：
说明项目、业务场景、当前阶段、已有材料。

目标：
说明这次只完成什么，不完成什么。

输入材料：
列出必须阅读的文档、代码、接口、日志或截图。

输出格式：
说明希望输出哪些章节、表格、清单、文件。

约束边界：
说明不能改哪些文件、不能引入哪些依赖、不能使用真实数据。

验收标准：
说明怎样判断完成，例如测试通过、页面可操作、文档可录制、安全检查通过。
```

结合本项目的示例：

```text
你现在是 AI 教程项目的第 1 课工作流材料子窗口。
请只负责整理第 1 课录屏材料、口播稿和讲义，
不改后端代码、不改前端代码、不调整数据库脚本。

请先阅读 README.md、docs/lesson-plan.md、docs/ai-dev-workflow.md 等材料。

新增 docs/lesson-01-workflow-recording-material.md，
内容需要覆盖基础概念、正确工作流、主控 / 子会话分工、提示词结构、常见错误、录屏建议、口播稿、讲义和主控验收清单。

风格要求通俗、克制、可信，不夸大 AI，不引入真实敏感信息。
```

这类提示词的关键不是“措辞漂亮”，而是任务边界清楚、输入材料清楚、输出验收清楚。

## 10. 常见错误

### 10.1 一句话让 AI 做完整系统

错误示例：

```text
帮我做一个客户风险评级系统，前后端和数据库都写好。
```

问题：

- 需求没澄清。
- 架构没评审。
- 数据库和 API 没契约。
- 测试和验收没标准。
- 安全边界没定义。

改法：先让 AI 做需求分析和待确认问题，再进入设计和拆分计划。

### 10.2 一个会话塞太多任务

例如同一个会话里同时让 AI 做后端、前端、数据库、PPT、领导汇报和 bug 修复。

结果通常是：

- 上下文混乱。
- 输出越来越泛。
- 约束互相污染。
- 主控难以验收。

改法：一个会话只做一个主线任务，复杂任务拆给子会话。

### 10.3 不跑测试就相信 AI

AI 说“已完成”“逻辑正确”“可以运行”，都不是验收结论。

必须看：

- 测试是否真的跑过。
- 构建是否通过。
- 接口是否可访问。
- 页面是否可操作。
- 错误路径是否验证。

### 10.4 上传真实数据、token、生产地址

这是高风险错误。

禁止把以下内容发给外部模型或放进课程材料：

- 真实客户数据。
- 真实账号、手机号、证件号。
- 内部系统地址。
- 生产数据库地址。
- API key、token、secret、私钥。
- 真实日志和报错中包含的敏感字段。

本课程只使用 `CUST10000*`、`SAMPLE-*`、`masked`、`sample` 语义样例。

### 10.5 上下文跑偏仍然硬聊

如果 AI 已经明显误解任务，多轮纠正仍然不稳定，不要继续堆更多解释。

更好的做法：

- 新开会话。
- 只放必要背景。
- 重新给清楚任务边界。
- 让它先复述目标和计划。

## 12. 半口语口播稿

大家好，这节是第 1 课，AI 编程基础概念和工作流。

第 0 课我们已经讲过，这套课不是单纯展示工具，而是要用 AI 走完一个小型研发闭环。那从这一节开始，我们先把最关键的认知问题讲清楚：为什么不能一上来就让 AI 写完整系统。

很多同学第一次用 AI 编程工具时，很自然会这样提需求：帮我做一个客户风险评级系统，前后端、数据库、页面都写好。这个想法可以理解，因为大家都想看看 AI 到底有多强。但在真实研发里，这种用法风险很高。

原因很简单：AI 不是项目负责人，它不知道我们项目里的全部背景、历史约束、安全要求、测试口径和上线责任。它能根据上下文生成看起来很完整的内容，但“看起来完整”和“工程上可交付”是两回事。

我们先把几个基础概念讲清楚。

在这门课里，AI 可以先理解成一个基于上下文生成结果的通用助手。大模型和 LLM 是支撑这个能力的模型。它能读需求、解释代码、生成方案、补测试、整理文档，也能帮我们做初步审查。但它不是业务负责人，也不是架构评审委员会，更不是最终审批人。

所以提示词也不是咒语。不是说我们找到一句神奇的话，AI 就能稳定产出好系统。好的提示词更像一份小型任务说明：背景是什么，目标是什么，输入材料有哪些，输出格式是什么，哪些事情不能做，做到什么程度才算完成。

再说上下文。上下文窗口可以理解为 AI 当前能看见的材料范围。你在会话里说过什么，贴了什么文档，AI 读了哪些文件，它都会基于这些内容继续生成。但上下文不是无限大的，而且会话越长、任务越杂，越容易跑偏。

比如一开始你让它做后端，后来又让它写讲义，再后来让它修前端 bug，它可能就会把不同任务的规则混在一起。这个时候继续硬聊，通常只会越来越乱。更好的做法是新开会话，只给当前任务需要的背景。

Agent 这个词也可以讲得简单一点。入门阶段先把它理解为一个会分步骤、会调用工具、会检查结果的 AI 工作流。它可以读文件、改文件、跑命令、看浏览器页面。但工具越强，越需要边界。AI 能改文件，不代表它应该随便改；AI 能跑命令，不代表测试就一定覆盖充分；AI 能打开浏览器，不代表业务验收已经通过。

这就引出本课程最重要的一条工作流：复杂任务不能直接让 AI 开写代码。正确流程应该是：需求分析，设计和评审，拆分计划，测试和验收标准，开发，AI 验收，人工验收，安全运维验收。

第一步是需求分析。先让 AI 帮我们整理需求摘要、用户角色、业务流程、功能边界、非功能要求、待确认问题和明确不做什么。这个阶段不写代码。因为需求没清楚时，AI 只能猜，而它猜出来的内容经常还挺像真的，这反而危险。

第二步是设计和评审。比如架构怎么分，模块边界在哪里，数据库表怎么设计，API 契约怎么定，异常码和日志怎么处理，测试策略是什么。AI 可以给草案，但架构取舍要人判断，尤其是银行项目里的金额、日期、状态、敏感字段和日志边界。

第三步是拆分计划。一个大任务要拆成能单独执行、单独验证的小任务。我们这个项目就是这么做的：先有 POC 架构设计子窗口，再有数据库设计子窗口、API 设计子窗口、后端子窗口、前端子窗口、联调验收子窗口，最后还有录屏材料子窗口。

第四步是测试和验收标准。先定义怎么算完成，再开始开发。后端至少要能跑测试，前端至少要能构建和打开关键页面，接口要符合契约，错误路径要能验证，日志和配置不能泄露敏感信息。

开发阶段，AI 可以提高效率，但建议一次只做一个小任务。每次修改前先看计划，修改后看 diff，关键路径跑测试。AI 说“完成了”不等于完成，测试、审查和人工验收通过才算完成。

AI 验收也很有用。我们可以开一个干净会话，让它站在审查者角度看需求、设计、代码摘要和测试结果，按 P0、P1、P2 输出问题。但注意，AI 验收是补充视角，不是替代人工审查。

最后是人工验收和安全运维验收。人要确认业务流程是不是真的对，页面能不能操作，接口返回是否符合预期，错误提示是否合理，日志能不能排查问题。安全上要确认没有真实客户数据、生产地址、密钥、token，也不要把真实日志直接丢给外部模型。

接下来我们看一下本项目是怎么实践这个方法的。

这里打开 `docs/master-task-board.md`。大家可以看到，主控窗口维护了整个任务状态：POC 架构设计、数据库设计、API 设计、后端实现、前端实现、联调验收、测试验收材料、第 0 课材料、第 1 课材料。每个任务都有产出和主控验收状态。

主控会话的职责不是把所有代码和材料都写完，而是维护目标、拆分任务、定义验收标准、接收子窗口回传、判断是否通过。子会话只负责单点产出，比如数据库设计就只做数据库，讲稿子窗口就只写录屏材料，不顺手改代码。

这也是大家后面使用 AI 编程工具时很重要的习惯：一个会话不要塞多个不相关任务。上下文跑偏时，新开会话。需要审查时，用干净会话。

我们再看 `docs/subsession-prompts.md`。这里的模板其实就是好提示词的结构：先说明项目背景，再说明当前任务，列输入材料，给输出要求，最后说明验收清单和要带回主控的信息。

所以写提示词的重点不是追求华丽，而是把背景、目标、输入、输出、约束和验收讲清楚。

最后总结一下这一课。

AI 编程最容易踩的坑，是把 AI 当成一个可以独立交付系统的开发者。更稳妥的方式，是把它放进我们熟悉的工程流程里：先分析需求，再做设计评审，再拆分任务，再定义验收，然后开发、测试、审查和人工验收。

AI 是研发搭档，不是最终负责人。真正的质量、安全和交付责任，仍然由人和工程流程承担。

这一课学完后，大家不需要记住很多术语，但要记住三个习惯：第一，提示词是任务说明，不是咒语；第二，一个会话只做一条主线；第三，AI 写完不算完成，验收通过才算完成。

下一课我们会进入 Claude Code + DeepSeek 环境搭建，让工具先跑起来。但在装工具之前，先理解今天这套工作流，会让后面的每一步都稳很多。

## 13. 讲义精简版

### 13.1 核心概念

AI / 大模型 / LLM：可以理解为基于上下文生成文字、代码、计划和审查意见的通用助手。它能辅助研发，但不承担最终责任。

提示词：不是咒语，而是任务说明。好提示词要包含背景、目标、输入材料、输出格式、约束边界和验收标准。

上下文窗口：AI 当前能看见和使用的信息范围。会话太长、任务太杂时容易跑偏。

Agent：可以理解为会分步骤、会调用工具、会检查结果的 AI 工作流。

工具调用：包括读写文件、执行命令、运行测试、操作浏览器等。工具能力越强，越需要明确边界和验收。

### 13.2 推荐工作流

```text
需求分析 -> 设计和评审 -> 拆分计划 -> 测试和验收标准 -> 开发 -> AI 验收 -> 人工验收 -> 安全运维验收
```

重点：

- 需求不清楚，不写代码。
- 设计没评审，不进入大规模开发。
- 任务不拆分，不交给 AI 一口气做。
- 没有验收标准，不算完成。
- AI 输出必须经过测试、审查和人工验收。

### 13.3 会话拆分

主控会话负责目标、拆分、验收、路线图和任务状态。

子会话负责单点产出，例如架构设计、数据库设计、API 设计、后端实现、前端实现、联调验收、录屏材料。

一个会话不要塞多个不相关任务。上下文跑偏时新开会话。

### 13.4 好提示词结构

```text
背景：项目和当前阶段是什么。
目标：本次只完成什么。
输入材料：需要阅读哪些文档、代码、日志或截图。
输出格式：希望输出哪些文件、章节、表格或清单。
约束边界：不能改什么，不能使用什么，不能泄露什么。
验收标准：怎么判断完成。
```

### 13.5 常见错误

- 一句话让 AI 做完整系统。
- 一个会话塞太多任务。
- 不跑测试就相信 AI。
- 上传真实客户数据、token、密钥、生产地址。
- 上下文跑偏仍然硬聊。

### 13.6 本课最重要的一句话

```text
AI 是研发搭档，不是最终负责人；AI 写完不算完成，验收通过才算完成。
```

## 14. 课后练习

请选择一个真实但低风险、非敏感的小需求，把它改写成适合 AI 执行的任务说明。

要求：

- 不包含真实客户数据、真实账号、手机号、证件号、生产地址、密钥或 token。
- 不要求 AI 一次完成完整系统。
- 按“背景、目标、输入材料、输出格式、约束边界、验收标准”六段来写。
- 明确本次不做什么。
- 至少写出 3 条验收标准。

示例题目：

```text
把“帮我加一个客户查询功能”改写成适合 AI 执行的任务说明。
```

参考结构：

```text
背景：
当前项目是一个教学用客户风险评级 POC，已有客户列表和客户详情接口设计。

目标：
请只整理客户查询功能的需求说明和验收标准，本轮不写代码。

输入材料：
请阅读 docs/api-design.md 中客户列表和客户详情相关接口。

输出格式：
输出需求摘要、用户角色、查询条件、正常流程、异常场景、不做范围、验收标准。

约束边界：
不使用真实客户数据；不修改后端、前端和数据库；不引入登录权限设计。

验收标准：
1. 能说明客户号、客户名称样例、风险等级、客户状态等查询条件。
2. 能覆盖无结果、非法参数、客户不存在等异常场景。
3. 所有样例数据使用 CUST10000*、SAMPLE-*、masked 或 sample 语义。
```

## 15. 失败兜底方案

如果学员觉得概念太多：

可以强调只需要先记住三件事：提示词是任务说明；会话要拆；AI 输出要验收。

如果学员问“那 AI 到底能不能写代码”：

可以回答：能，而且效率很高。但要先有需求、设计、任务拆分和验收标准。越是复杂和高风险的代码，越不能跳过这些步骤。

如果学员问“会话拆太细会不会麻烦”：

可以回答：短期看多一步，长期看更稳。拆清楚以后，每个任务可复用、可验收、可回滚，也更容易定位问题。

如果学员问“AI 验收能不能代替人工评审”：

可以回答：不能。AI 验收是补充视角，人工仍然要看业务合理性、测试结果、安全边界和项目规范。

如果学员问“真实项目能不能上传日志给 AI”：

可以回答：不能直接上传真实敏感日志。必须先脱敏，并遵守公司安全规范和项目要求。
