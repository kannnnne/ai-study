# 第19章 企业 AI SOP 模板库

> **目标读者**：Team Leader、研发经理、质量负责人，正在推动 AI 工具在团队中规范落地。
> **本章回答的核心问题**：怎么让 AI 从"个人随意使用"变成"团队标准化流程"？每个环节的操作步骤、验收标准、责任人怎么定义？

---

## 19.1 为什么需要 AI SOP

AI 工具在团队推广时常遇到两个瓶颈：

1. **用了但用不好**：每个人用法不同，产出质量波动大。有人把 AI 当搜索引擎，有人把 prompt 写成一篇文章，有人从不 review AI 输出直接提交——结果是团队对 AI 的信任度持续下降。
2. **知道该用但没人推动**：TL 在会上说"大家多用 AI"，但没有具体指令，没有人知道"多"是什么意思、怎么算"用好了"。

SOP（Standard Operating Procedure，标准作业程序）解决的就是这两个问题。它不是限制创造力，而是把已验证有效的操作步骤固定下来，让新人三天上手，让老手一致产出，让管理者有据可查。

本章提供 10 份完整的 AI SOP 模板，覆盖企业 Java/Spring Boot 团队的日常高频场景。每份 SOP 可直接使用或按团队情况定制。

### SOP-01: AI 辅助需求分析 SOP

#### 1. 目的

用 AI 将模糊的业务需求转化为结构化的需求文档，减少需求理解偏差，提前暴露遗漏和矛盾。

#### 2. 适用范围

- 场景：新功能需求分析、需求变更分析、需求评审前准备
- 角色：BA（业务分析师）、产品经理、Tech Lead

#### 3. 前置条件

- 已完成与业务方的初步沟通，有原始需求素材（文字描述、会议录音、聊天记录均可）
- 已配置 AI 工具（Claude Code / ChatGPT / Cursor）
- 团队有 PRD 模板或需求文档标准章节

#### 4. 输入材料

- 业务方原始需求描述（一句话到一段话均可）
- 相关系统的现有文档（可选，帮助 AI 理解上下文）
- 团队 PRD 模板（可选，让 AI 按模板格式输出）

#### 5. 操作步骤

1. **需求录入**（BA / 5分钟）：将业务方的原始需求描述整理为一段通顺的文字，贴入 AI 工具。如果素材是会议录音，先用 AI 转写后再贴入。

2. **AI 结构化展开**（AI / 自动）：使用以下 prompt 让 AI 展开需求：
   ```
   你是一位资深业务分析师。以下是业务方提出的原始需求，请做以下分析：

   1. 用一段话概括需求的核心目标（一句话说清要解决什么问题）
   2. 列出所有涉及的用户角色和他们的使用场景
   3. 列出核心业务流程（主流程 + 至少3个异常分支）
   4. 列出功能性需求（按优先级：P0必须有、P1应该有、P2锦上添花）
   5. 列出非功能性需求（性能、安全、可用性、可扩展性）
   6. 列出边界条件和约束（如：仅限内部用户、需要审批流、数据保留期限等）
   7. 列出需求中的模糊点和矛盾点，给出追问建议

   原始需求：
   [粘贴原始需求]
   ```

3. **人工审核追问**（BA / 15分钟）：逐条检查 AI 输出的模糊点和追问建议，判断是否确实是遗漏项。将有效追问整理后向业务方确认。

4. **补充业务上下文**（BA / 10分钟）：AI 不知道的信息需要人工补充，包括：
   - 监管合规要求（如金融行业的数据留存期限）
   - 历史遗留约束（如"这个字段不能改，因为下游系统依赖"）
   - 业务优先级（如"审批流先做简单版，复杂的下期"）

5. **生成 PRD 草稿**（AI / 自动）：将确认后的需求分析结果和补充信息一起给 AI，指定输出格式为团队 PRD 模板格式。

6. **Review 与定稿**（BA + Tech Lead / 20分钟）：在 Git 上 PRD 以 MR 形式提交，Tech Lead 逐节 review，重点关注：
   - 功能描述是否完整无歧义
   - 验收标准是否可测试
   - "不做什么"是否明确列出

#### 6. 输出物

- 结构化需求分析文档（包含角色、场景、流程、功能列表、非功能需求、边界条件）
- 业务方追问清单及确认结果
- PRD 草稿（按团队模板格式）

#### 7. 验收标准

- [ ] 所有用户角色的使用场景已列出
- [ ] 主流程和至少 3 个异常分支已覆盖
- [ ] 所有模糊点已向业务方确认并获得回复
- [ ] 非功能性需求有明确的量化指标（如"QPS > 1000"而不是"性能要好"）
- [ ] 功能需求按 P0/P1/P2 标注了优先级
- [ ] "不做什么"章节已明确列出

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| AI 生成的需求看起来很完整，但遗漏了关键业务约束 | 必须由熟悉业务的人员逐条审核，不可仅凭"看起来专业"即通过 |
| AI 生成的追问清单过于发散，浪费时间 | BA 先筛选，只保留确实关键的追问（建议不超过 10 条） |
| 业务方给的原始需求过于模糊（一句话需求） | 先向业务方做一轮基础澄清，获得足够素材后再启动 AI 分析，不要用 AI "猜"业务意图 |
| AI 不了解行业监管要求 | 团队维护一份"行业合规检查清单"，需求分析时人工对照检查 |

#### 9. 责任人

- **BA / 产品经理**：对需求分析结果的完整性和准确性负最终责任
- **Tech Lead**：审核技术可行性和非功能需求合理性

#### 10. 相关模板

- PRD 模板（团队维护）
- 行业合规检查清单（团队维护）
- 非功能需求量化指标参考表（见本书附录）

### SOP-02: AI 辅助技术设计 SOP

#### 1. 目的

用 AI 辅助生成技术设计方案，快速产出多方案对比，减少设计阶段盲区。

#### 2. 适用范围

- 场景：新功能技术方案设计、技术选型评估、架构变更方案
- 角色：Tech Lead、高级开发工程师

#### 3. 前置条件

- 已完成需求分析，有明确的需求文档 (PRD)
- 了解现有系统架构和约束
- AI 工具已接入项目代码库（通过 CLAUDE.md / .cursorrules 提供项目上下文）

#### 4. 输入材料

- PRD 或需求分析文档
- 项目 CLAUDE.md（包含技术栈、架构约定、代码规范）
- 相关模块的现有代码（可选，供 AI 理解现状）
- 性能要求和非功能需求指标

#### 5. 操作步骤

1. **描述设计场景**（Tech Lead / 10分钟）：用自然语言向 AI 描述：
   - 要解决什么问题
   - 现有的系统上下文（是什么系统、什么模块、什么技术栈）
   - 关键约束条件（如"不能引入新的中间件""必须兼容现有的 API 格式""数据量级约 100 万条"）

2. **AI 生成多方案对比**（AI / 自动）：使用以下 prompt：
   ```
   你是一位资深 Java 架构师。针对以下需求和约束，请设计 2-3 个技术方案，每个方案包含：

   1. 方案概述（一段话说明核心思路）
   2. 架构图描述（用文字描述组件和交互关系，可用 Mermaid）
   3. 核心数据流（请求从入口到响应的完整路径）
   4. 关键代码结构（包结构、核心类/接口定义）
   5. 数据库变更（如有，列出 DDL 变更要点）
   6. 优缺点对比（用表格列出各方案在复杂度、性能、可扩展性、维护成本上的对比）
   7. 潜在风险和坑点
   8. 推荐方案及理由

   需求描述：[粘贴需求]
   技术栈：Java 17, Spring Boot 3.x, MySQL 8.x, Redis, RabbitMQ
   约束条件：[列出约束]
   ```

3. **团队评审选型**（技术团队 / 30分钟）：召开技术评审会，将 AI 生成的方案作为讨论起点，逐一评估：
   - 方案是否与现有架构一致
   - 团队技术能力是否能落地
   - 运维成本是否可接受
   - 是否有未考虑的边界条件

4. **深化选定方案**（Tech Lead + AI / 20分钟）：选定方案后，让 AI 深化细节：
   - 详细的接口定义（请求/响应结构、状态码、错误码）
   - 数据库表结构 DDL
   - 核心业务逻辑伪代码
   - 异常处理策略

5. **补充团队特定内容**（Tech Lead / 15分钟）：AI 不知道的信息需人工补充：
   - 上线方案和回滚策略
   - 数据迁移方案（如涉及）
   - 监控指标和告警阈值
   - 测试策略

6. **生成设计文档**（AI / 自动）：将全部内容汇总，让 AI 输出为团队设计文档模板格式。

7. **文档评审与归档**（Tech Lead / 15分钟）：在设计文档 MR 中 review，确认后合入项目文档库。

#### 6. 输出物

- 多方案对比分析文档
- 选定方案的详细技术设计文档
- 接口定义（含请求/响应示例）
- 数据库 DDL 变更脚本（草稿）

#### 7. 验收标准

- [ ] 至少产出 2 个方案的对比分析
- [ ] 选定方案有明确的推荐理由
- [ ] 核心接口定义完整（路径、方法、参数、返回值、错误码）
- [ ] 数据库变更清晰（新建表/修改表/数据迁移方案）
- [ ] 异常流程处理策略已明确
- [ ] 上线方案和回滚方案已包含
- [ ] 设计方案已通过技术评审会

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| AI 不了解现有系统的技术债务和隐性约束 | 必须在 prompt 中明确列出约束条件；设计方案必须由熟悉系统的开发人员审核 |
| AI 倾向推荐"理想架构"而非"现实可行的架构" | 评审时优先考虑团队技术能力和运维成本的匹配度，不要因为方案"看着漂亮"就选 |
| AI 可能在方案中引入项目未使用的组件或框架 | 在约束条件中明确"不引入新的第三方组件"或列出允许的组件清单 |
| 设计文档中的时序图/架构图可能不准确 | AI 生成的 Mermaid 图仅作参考，关键流程图应人工验证逻辑正确性 |

#### 9. 责任人

- **Tech Lead**：对技术方案的合理性、可行性负最终责任
- **高级开发工程师**：审核实现细节，确认可落地

#### 10. 相关模板

- 技术设计文档模板（团队维护）
- 技术评审会 checklist（见附录）
- 项目 CLAUDE.md（技术上下文）

### SOP-03: AI 辅助接口开发 SOP

#### 1. 目的

用 AI 生成 Spring Boot 接口的标准分层代码（Controller、Service、DAO、DTO），保证代码风格一致，减少重复劳动。

#### 2. 适用范围

- 场景：标准 CRUD 接口开发、简单业务接口开发、接口代码重构
- 角色：Java 后端开发工程师

#### 3. 前置条件

- 接口设计文档已完成（请求/响应结构、业务规则、异常场景已明确）
- 项目具备完整的分层代码模板或参考实现
- AI 工具已配置项目代码风格（Checkstyle 规则 / CLAUDE.md 代码规范）
- 数据库表已存在或 DDL 已确定

#### 4. 输入材料

- 接口设计文档（含请求参数、响应结构、业务规则、错误码）
- 数据库表结构 DDL
- 项目现有同类接口代码（1-2 个作为风格参考）
- 项目 CLAUDE.md 或代码规范文档

#### 5. 操作步骤

1. **准备上下文**（开发工程师 / 5分钟）：
   - 将项目 CLAUDE.md 放入 AI 工具上下文
   - 将 1-2 个现有完整接口代码作为风格参考提供给 AI
   - 将数据库 DDL 贴入 AI 上下文

2. **生成 Entity 和 DTO**（AI / 自动）：先让 AI 根据表结构生成：
   - Entity 类（含 JPA 注解、字段注释、关系映射）
   - Request DTO（含 `@Valid` 校验注解）
   - Response DTO（含 Swagger 注解）
   - 转换工具类（Entity <-> DTO 转换，推荐 MapStruct）

3. **人工审核 Entity/DTO**（开发工程师 / 5分钟）：检查：
   - 字段映射是否正确（特别注意枚举类型、日期格式、金额精度）
   - 校验注解是否完整（`@NotNull`、`@NotBlank`、`@Size`、`@Min`、`@Max` 等）
   - Swagger 注解描述是否准确

4. **生成 DAO/Repository**（AI / 自动）：生成：
   - Repository 接口（Spring Data JPA / MyBatis-Plus）
   - 自定义查询方法（根据业务查询需求）
   - SQL 或 JPQL 语句

5. **生成 Service 层**（AI / 自动）：分业务复杂度处理：

   **简单业务（纯 CRUD）：**
   - 让 AI 直接生成完整 Service 实现
   - 包含参数校验、事务管理、异常处理

   **复杂业务（多步骤、有状态机）：**
   - 先让 AI 生成业务流程伪代码
   - 人工确认业务流程逻辑正确后，再让 AI 生成完整代码

   Prompt 示例：
   ```
   根据以下接口设计和业务规则，生成完整的 Service 实现类。

   要求：
   - 使用 @Transactional 管理事务
   - 所有异常使用统一的业务异常类 BusinessException
   - 参数校验在 Controller 层通过 @Valid 完成，Service 层做业务校验
   - 关键操作记录操作日志
   - 遵循项目现有的命名规范和代码风格

   接口设计：[粘贴接口设计]
   业务规则：[粘贴业务规则]
   ```

6. **生成 Controller 层**（AI / 自动）：生成包括：
   - REST 路径和方法映射
   - `@Valid` 参数校验
   - 统一响应格式封装
   - Swagger/OpenAPI 注解

7. **编译验证与修复**（开发工程师 + AI / 10分钟）：
   - 运行 `mvn compile` 检查编译错误
   - 如有编译错误，将错误信息贴给 AI 修复
   - 修复后再次编译确认通过

8. **人工强化关键逻辑**（开发工程师 / 时间视复杂度）：AI 生成的代码中，以下部分必须人工逐行 review 和增强：
   - 事务边界（`@Transactional` 的传播行为和回滚策略是否正确）
   - 并发控制（是否存在并发安全问题，是否需要分布式锁）
   - 幂等性处理（重复请求是否安全）
   - 敏感数据处理（日志中是否打印了敏感字段，是否需要脱敏）
   - 金额计算（是否使用 `BigDecimal`，精度是否正确）

9. **运行单元测试**（开发工程师 / 5分钟）：运行 AI 同步生成的单元测试（详见 SOP-05），确保核心逻辑通过。

#### 6. 输出物

- 完整的分层代码文件（Entity、DTO、DAO、Service、Controller）
- 编译通过的代码（`mvn compile` 无错误）
- 通过的单元测试（核心方法覆盖率 > 80%）

#### 7. 验收标准

- [ ] Controller 层参数校验注解完整且正确
- [ ] Service 层事务管理合理，异常处理统一
- [ ] DTO 和 Entity 之间的转换逻辑正确
- [ ] 所有接口有对应的 Swagger 文档注解
- [ ] 关键业务逻辑包含日志记录
- [ ] 无编译警告（或项目允许的警告范围内）
- [ ] 代码通过 Checkstyle / SpotBugs 检查
- [ ] 人工 review 关键逻辑（事务、并发、幂等、金额、敏感数据）通过

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| AI 生成的代码在简单场景下看起来正确，但边界条件处理不足 | 必须人工补充边界条件（null、空集合、超大值、并发）的测试用例和处理逻辑 |
| AI 可能使用过时的 API 或废弃的方法 | 编译验证可拦截大部分；IDEA 的 deprecated 警告也需要检查 |
| AI 不了解项目自定义的工具类和基础类 | 在 CLAUDE.md 中维护"项目基础工具类索引"，每次生成前提供给 AI |
| 批量生成多个接口时质量下降（上下文被稀释） | 每次只处理 1-2 个接口，完成后清理上下文再开始下一批 |

#### 9. 责任人

- **开发工程师**：对代码的正确性、安全性、可维护性负最终责任
- **Code Reviewer**：在 MR review 中按此 SOP 的标准逐项检查

#### 10. 相关模板

- 项目代码规范文档 (CLAUDE.md)
- 业务异常类规范
- 统一响应格式规范
- 日志规范文档

### SOP-04: AI 辅助代码 Review SOP

#### 1. 目的

用 AI 作为代码 review 的第一道关卡，在 MR 提交前自动发现低层次问题，让人专注于架构和业务逻辑的审查。

#### 2. 适用范围

- 场景：MR/PR 提交前的自查、Code Review 的第一轮
- 角色：开发工程师（自检）、Code Reviewer（辅助审查）

#### 3. 前置条件

- 项目已配置静态分析工具（Checkstyle、SpotBugs、SonarQube 任选）
- AI 工具可读取 git diff（Claude Code 的 `/review` 或 `/code-review` 命令）
- 项目有明确的代码规范文档

#### 4. 输入材料

- Git diff（当前分支 vs 目标分支）
- 相关需求文档或接口设计文档（帮助 AI 判断实现是否符合预期）
- 项目代码规范文档

#### 5. 操作步骤

1. **开发自检**（开发工程师 / 5分钟）：
   - 在提交 MR 前，先对本地 diff 运行 AI review
   - 命令示例：Claude Code 中使用 `/code-review` 或直接贴 diff 给 AI
   - 修复 AI 指出的明确问题（命名不规范、缺少 null 检查、日志级别不当等）

2. **AI Review 执行**（AI / 自动）：使用以下 review 检查维度：

   ```
   请 review 以下 git diff，从以下维度逐项检查：

   1. 代码风格：是否符合团队规范（命名、格式、注释）
   2. 空指针风险：是否有未处理的 null 情况
   3. 异常处理：异常是否被正确捕获和处理，是否有吞异常的情况
   4. 资源管理：数据库连接、文件流、HTTP 客户端是否正确关闭
   5. 事务安全：@Transactional 的使用是否正确，事务边界是否合理
   6. 日志规范：关键操作是否记录日志，日志级别是否恰当，是否打印敏感信息
   7. SQL 安全：是否存在 SQL 注入风险，是否有 N+1 查询问题
   8. 并发安全：共享变量是否有线程安全问题
   9. 性能：是否有明显的性能隐患（循环内查库、大事务、未使用批量操作等）
   10. 可维护性：代码是否清晰易懂，是否有过于复杂的逻辑需要拆分

   对于每个问题，请指出：
   - 严重程度：🔴 阻断 / 🟡 建议 / 🟢 优化
   - 具体位置：文件名 + 行号
   - 问题描述和建议修复方案

   Git Diff：
   [粘贴 diff]
   ```

3. **开发修复**（开发工程师 / 时间视问题数量）：处理 AI 标记的问题：
   - 🔴 阻断：必须修复后才能提交 MR
   - 🟡 建议：修复或在 MR 描述中说明不修复的原因
   - 🟢 优化：可在后续迭代中处理，创建 tech debt ticket

4. **提交 MR 并附带 AI Review 摘要**（开发工程师 / 3分钟）：
   - 在 MR 描述中附上 AI review 的摘要
   - 标注哪些问题已修复、哪些问题有意保留及原因

5. **人工 Code Review**（Reviewer / 20分钟）：Reviewer 在 AI review 的基础上聚焦：
   - 业务逻辑是否正确（AI 无法判断的部分）
   - 架构决策是否合理（AI 无法理解全局的部分）
   - 是否有过度设计或设计不足
   - 测试覆盖是否充分

#### 6. 输出物

- AI Review 报告（问题清单，含严重程度和修复建议）
- 开发工程师的修复说明
- 人工 Review 意见

#### 7. 验收标准

- [ ] 所有 🔴 阻断问题已修复
- [ ] 所有 🟡 建议问题已修复或有明确的不修复说明
- [ ] MR 描述中包含 AI Review 摘要
- [ ] 人工 Review 已完成且无未解决的讨论

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| AI 可能误报（false positive），将正确代码标为问题 | Reviewer 需要快速过一遍 AI 报告，过滤明显误报；开发工程师有权标记为"已知误报" |
| AI 漏报业务逻辑错误（AI 看不出"钱算错了"） | 人工 Review 必须覆盖业务逻辑，不能因为"AI 过了"就跳过 |
| 团队可能过度依赖 AI Review，导致人工 Review 流于形式 | 明确分工：AI 负责低层次问题，人负责业务和架构问题；定期抽查 Review 质量 |
| AI 对大型 diff 的 review 质量下降 | 控制 MR 大小：建议每次 MR 不超过 400 行变更；大功能拆分为多个小 MR |

#### 9. 责任人

- **开发工程师**：对提交代码的质量负第一责任（AI review 只是辅助工具）
- **Code Reviewer**：对 Review 的有效性负责（不能仅依赖 AI report）

#### 10. 相关模板

- Code Review Checklist（团队维护）
- 常见问题分类和严重程度定义表
- MR 描述模板（含 AI Review 摘要章节）

### SOP-05: AI 辅助单元测试 SOP

#### 1. 目的

用 AI 批量生成单元测试用例，提高测试覆盖率，减少手动编写重复测试代码的时间。

#### 2. 适用范围

- 场景：新代码的单元测试编写、遗留代码补充测试、测试覆盖率提升专项
- 角色：Java 后端开发工程师

#### 3. 前置条件

- 项目使用 JUnit 5 + Mockito（或团队统一的其他测试框架）
- 被测代码已通过编译
- 项目定义了覆盖率目标（建议：行覆盖率 > 80%，分支覆盖率 > 70%）
- JaCoCo 或其他覆盖率工具已配置

#### 4. 输入材料

- 被测类的完整源代码
- 被测类所依赖的接口定义（用于 mock）
- 项目的测试基类和测试工具类（如有）
- 业务规则说明（如有复杂业务逻辑）

#### 5. 操作步骤

1. **被测代码分类**（开发工程师 / 5分钟）：将被测类分为三类：
   - A 类（纯逻辑类）：Util、Converter、Validator、Domain Service 中的纯计算方法 —— AI 生成，人工抽查
   - B 类（编排类）：Controller、Facade、Application Service —— AI 生成框架，人工补全关键场景
   - C 类（复杂业务类）：涉及状态机、复杂计算、事务编排 —— 人工编写核心用例，AI 补全边界用例

2. **AI 生成测试用例**（AI / 自动）：使用以下 prompt：

   ```
   请为以下 Java 类生成完整的 JUnit 5 + Mockito 单元测试。

   要求：
   - 使用 @ExtendWith(MockitoExtension.class)
   - 使用 @Mock 注解 mock 所有依赖
   - 使用 @InjectMocks 注入被测对象
   - 覆盖以下场景：
     a) 正常路径（Happy Path）：输入正常参数，验证返回值和依赖调用
     b) 异常路径：依赖抛出异常时，验证异常处理逻辑
     c) 边界值：null、空集合、空字符串、0、负数、极大值
     d) 业务异常：触发业务校验失败的场景
   - 每个测试方法名使用 should_xxx_when_xxx 命名风格
   - 使用 given/when/then 结构组织测试代码
   - 断言使用 AssertJ 的 assertThat...isEqualTo() 风格
   - 不要测试框架行为（如不要测试 Spring 的依赖注入是否正常）

   被测代码：
   [粘贴源代码]

   依赖接口定义：
   [粘贴依赖接口]
   ```

3. **编译运行测试**（开发工程师 / 3分钟）：
   - 将 AI 生成的测试代码放入正确的测试目录
   - 运行 `mvn test -Dtest=XXXTest`
   - 检查编译和运行结果

4. **修复失败用例**（开发工程师 + AI / 10分钟）：
   - 分析失败原因：是测试写错了还是被测代码有问题
   - 如果是测试写错了（mock 行为不正确、断言值不对），修正测试
   - 如果是被测代码有问题（实际行为不符合预期），标记为 bug 并修复

5. **人工补全关键场景**（开发工程师 / 15分钟）：
   - 对 B 类和 C 类代码，人工确认以下场景已覆盖：
     - 数据库事务提交和回滚场景
     - 异步处理的回调场景
     - 缓存命中/未命中场景
     - 幂等性验证场景
   - 补充 AI 遗漏的业务场景测试

6. **覆盖率验证**（开发工程师 / 3分钟）：
   - 运行 `mvn test jacoco:report`
   - 检查覆盖率报告，确认达到团队标准
   - 对未覆盖的分支，判断是否需要补充（部分防御性代码可接受不覆盖）

#### 6. 输出物

- 完整的单元测试类文件
- JaCoCo 覆盖率报告
- 未覆盖分支说明（如有意跳过）

#### 7. 验收标准

- [ ] 所有 AI 生成的测试用例通过（绿色）
- [ ] Happy Path 和异常路径都有覆盖
- [ ] 边界值场景覆盖（null、空集合、极值）
- [ ] 行覆盖率 > 80%，分支覆盖率 > 70%
- [ ] 测试类本身代码清晰，每个用例意图明确
- [ ] 无"伪绿色"测试：每个测试都有有效断言（不能只调用方法不验证结果）

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| AI 生成"伪绿色"测试：测试跑了但不做有效断言 | Code Review 时检查每个测试的 assert 语句是否验证了关键结果 |
| AI 不理解业务含义，边界值选得不合理（如"年龄 = -1"在业务上无意义） | 人工审查边界值用例，删除无业务意义的测试 |
| Mock 过度导致测试脆弱：内部重构（没改行为）测试大面积失败 | 优先 mock 外部依赖（数据库、RPC、消息队列），少 mock 内部类 |
| AI 生成的测试可能漏掉并发相关的场景 | 并发测试需要人工编写，不在本 SOP 覆盖范围 |
| 测试代码本身也可能有 bug | 刻意让被测方法返回错误结果，确认测试能捕获（Mutation Testing 思路） |

#### 9. 责任人

- **开发工程师**：对测试质量和覆盖率负最终责任
- **Code Reviewer**：审核测试用例的有效性和合理性

#### 10. 相关模板

- 测试基类模板（团队维护）
- JaCoCo 覆盖率配置
- Mockito 最佳实践文档

### SOP-06: AI 辅助生产排障 SOP

#### 1. 目的

用 AI 加速生产问题的定位和分析，但严格限制 AI 的直接操作权限，确保生产安全。

#### 2. 适用范围

- 场景：生产环境告警响应、接口超时排查、异常日志分析、SQL 慢查询分析
- 角色：值班开发工程师、SRE、Tech Lead

#### 3. 前置条件

- 具备生产日志查看权限（ELK / Grafana Loki / 阿里云 SLS / 日志文件）
- 具备监控平台查看权限（Grafana / Prometheus / 云厂商监控）
- 有一条具体的告警或问题描述（不能是"系统慢了"这种模糊问题）
- AI 工具已开启安全模式（不得有生产环境的写权限）

#### 4. 输入材料

- 告警信息（时间、指标、阈值、持续时间）
- 错误日志片段（**必须先脱敏**，去掉客户 PII、密钥、Token、内网 IP）
- 相关接口的调用链路信息（TraceID / Skywalking / Jaeger）
- 近期发布记录（是否有新版本上线）

#### 5. 操作步骤

1. **问题定界**（值班工程师 / 5分钟）：手动确认以下信息后将问题传递给 AI：
   - 问题发生的时间范围（精确到分钟）
   - 影响范围（单个用户/全部用户、单个接口/全局、单节点/全部节点）
   - 是否有近期变更（代码发布、配置变更、流量变化）
   - 关键监控数据（CPU、内存、QPS、RT、错误率的变化趋势）

2. **日志脱敏**（值班工程师 / 5分钟）：**绝不可跳过**——将日志中的敏感信息替换为占位符：
   - 手机号 → `138****1234`
   - 身份证号 → `3201**********1234`
   - Token/API Key → `sk-****`
   - 内网 IP → `10.x.x.x`
   - 客户姓名 → `[CUSTOMER_NAME]`

3. **AI 日志分析**（AI / 自动）：使用以下 prompt：

   ```
   你是一位资深后端排障工程师。以下是生产环境的问题信息，请帮助分析根因。

   问题概况：
   - 时间：[时间]
   - 影响：[影响范围]
   - 近24小时变更：[变更记录]
   - 关键指标：[CPU/内存/QPS/RT/错误率的变化]

   错误日志（已脱敏）：
   [粘贴脱敏后的日志]

   调用链路信息：
   [粘贴 TraceID 下的调用链]

   请分析：
   1. 根因推断（最可能的原因是什么，为什么）
   2. 证据链（日志中哪些信息支持你的推断）
   3. 影响范围评估
   4. 建议的止血方案（只读操作优先）
   5. 建议的排查方向（如果需要更多信息）
   6. 这个问题的严重程度评估和升级建议
   ```

4. **验证 AI 分析**（值班工程师 / 10分钟）：对 AI 的分析逐条确认：
   - 根因推断是否与监控数据一致
   - 证据链是否完整
   - 止血方案是否安全（**禁止 AI 建议的操作直接在生产执行**）

5. **执行止血**（值班工程师 / 时间视情况）：
   - AI 建议的止血方案需人工在变更平台审批后执行
   - **绝不允许 AI 生成脚本后直接在生产执行**
   - 执行前确认回滚方案

6. **问题复盘**（值班工程师 + AI / 15分钟）：
   - 问题解决后，将处理过程整理给 AI
   - 让 AI 生成复盘报告（5 Why 分析、改进措施、预防方案）

#### 6. 输出物

- 日志脱敏后的分析材料
- AI 分析报告（根因推断、证据链、止血建议）
- 问题复盘报告（事后）

#### 7. 验收标准

- [ ] 问题根因已确认（非"可能""也许"的猜测）
- [ ] 止血操作已完成且有效
- [ ] 复盘报告已归档（包含 5 Why 和改进措施）
- [ ] 相关告警规则已评估是否需要调整
- [ ] 敏感信息未泄露给 AI 系统

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| **最高风险：生产数据泄露** | 日志脱敏是强制步骤，不能跳过。建议团队维护日志脱敏脚本，自动化替换敏感字段 |
| **高风险：AI 建议的脚本直接执行** | 建立规则：AI 生成的所有命令和脚本必须经过人工审核和变更审批后才能在生产执行 |
| AI 可能给出看似合理但方向错误的分析 | 必须用监控数据交叉验证。AI 基于日志文本推断，可能忽略系统级问题（如网络抖动、磁盘慢） |
| AI 可能建议重启服务这种"万能但粗暴"的方案 | 将重启服务设为最后手段，先尝试不影响服务的排查手段（如动态调整日志级别、Arthas 诊断） |
| 排障过程本身产生大量日志，可能触发二次告警 | 排障操作前评估可能的影响，必要时先静默相关告警 |

#### 9. 责任人

- **值班工程师**：对生产安全负第一责任，对排障过程中的所有操作负责
- **Tech Lead**：对复杂问题的根因判断和解决方案负最终责任

#### 10. 相关模板

- 日志脱敏脚本 / 规则配置
- 问题复盘报告模板
- 生产变更审批流程文档

### SOP-07: AI 辅助项目周报 SOP

#### 1. 目的

用 AI 根据项目管理系统数据和代码提交记录自动生成周报，减少信息收集和格式整理时间。

#### 2. 适用范围

- 场景：团队周报、个人周报、项目进度汇报
- 角色：Team Leader、项目经理、开发工程师

#### 3. 前置条件

- 项目使用 Git（可通过 git log 获取提交记录）
- 项目使用项目管理工具（Jira / TAPD / 飞书项目 / Linear 等）
- AI 工具已接入

#### 4. 输入材料

- 本周 Git 提交记录：`git log --since="7 days ago" --oneline --author="<name>"`
- 项目管理工具中本周完成/进行中/计划中的任务（导出或截图转文字）
- 上周周报中"下周计划"部分（用于对比）
- 团队周报模板（如团队有统一格式要求）

#### 5. 操作步骤

1. **收集原始数据**（TL / 项目经理 / 10分钟）：
   - 从项目管理工具导出本周任务列表（完成、进行中、计划）
   - 拉取团队成员的 git log（可按模块或负责人筛选）
   - 准备需要注意的风险事项和阻塞点

2. **AI 生成周报**（AI / 自动）：使用以下 prompt：

   ```
   请根据以下原始数据，生成一份专业的项目周报。

   周报格式要求：
   ## 本周关键成果（3-5 条，每条一句话）
   ## 重点事项进展
   ### 已完成
   （列出本周完成的任务，标注负责人和完成时间）
   ### 进行中
   （列出进行中的任务，标注进度百分比、负责人、预计完成时间）
   ### 下周计划
   （列出下周计划启动的任务）
   ## 风险与阻塞
   （列出需要关注的风险和阻塞项，标注严重程度和应对方案）
   ## 数据指标
   （如有：代码提交次数、MR 数量、Bug 数量、测试覆盖率变化等）

   原始数据：
   Git 提交记录：
   [粘贴 git log]

   项目管理任务：
   [粘贴任务列表]

   需要关注的事项：
   [粘贴风险事项]
   ```

3. **人工审核和修正**（TL / 项目经理 / 10分钟）：
   - 检查关键成果是否准确反映项目进展
   - 补充 AI 无法感知的信息（如跨团队沟通进展、客户的非正式反馈）
   - 修正不准确的进度百分比
   - 添加需要向上汇报的"坏消息"（AI 倾向过于乐观）

4. **补充数据指标**（TL / 5分钟）：从项目管理工具和 CI/CD 平台获取量化数据：
   - 本周代码提交次数
   - 本周 MR 合并数量
   - 本周 Bug 新增/关闭数量
   - 测试覆盖率变化
   - 如有线上问题，记录 P1/P2 事故数量

5. **发布周报**（TL / 项目经理 / 2分钟）：发布到团队沟通渠道（飞书/钉钉/邮件）。

#### 6. 输出物

- 格式化的项目周报文档

#### 7. 验收标准

- [ ] 关键成果描述准确，无夸大或遗漏
- [ ] 各任务进度百分比与实际一致
- [ ] 风险和阻塞项如实反映（不做美化）
- [ ] 数据指标与实际数据一致
- [ ] 周报在约定时间前发布（如每周五 17:00 前）

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| AI 可能美化进展，把"卡住了"写成"有序推进中" | TL 必须在审核时确保"坏消息"被如实传达，不要使用 AI 的乐观修辞 |
| AI 可能漏掉不体现在 Git/Task 上的工作（如技术调研、跨团队协调） | TL 在审核时手动补充这类工作 |
| 纯依赖 AI 生成周报，TL 对项目进展失去直观感受 | 周报审核过程本身是 TL 重新审视项目进展的机会，不能跳过 |
| 数据指标异常时 AI 不会主动分析原因 | TL 发现数据异常（如 Bug 突然增多）时，人工补充原因分析 |

#### 9. 责任人

- **TL / 项目经理**：对周报内容的准确性和完整性负最终责任

#### 10. 相关模板

- 团队周报模板
- 项目管理工具查询快捷方式
- 周报发布沟通模板

### SOP-08: AI 辅助知识库问答 SOP

#### 1. 目的

用 AI 作为团队知识库的统一查询入口，减少在文档海洋中搜索答案的时间。

#### 2. 适用范围

- 场景：查询技术方案、系统架构、业务规则、历史决策、故障复盘记录
- 角色：团队全体成员

#### 3. 前置条件

- 团队知识库已建立（文档统一存放在 Git 仓库 / Confluence / 语雀等）
- 知识库内容已索引（AI 工具可访问）
- 知识库保持了基本的更新习惯（非陈旧过时内容）

#### 4. 输入材料

- 用户的问题描述（自然语言）
- 知识库上下文（AI 工具通过项目 CLAUDE.md 或 MCP 获取）

#### 5. 操作步骤

1. **问题描述**（团队成员 / 2分钟）：用自然语言清楚描述问题，包含：
   - 你要了解什么（如"订单超时取消的逻辑是怎样的？"）
   - 为什么需要了解（如"我在对接订单模块"）
   - 已知信息（如"我知道超时时间是30分钟，但不知道取消后的退款流程"）

2. **AI 检索与回答**（AI / 自动）：AI 从知识库中检索相关文档，给出回答。确保 AI 被要求：
   - 注明信息来源（哪个文档、哪个章节）
   - 区分"文档中有明确答案"和"基于文档的推断"
   - 如果找不到相关信息，明确说"知识库中未找到"

3. **信息验证**（团队成员 / 3分钟）：
   - 如果回答标注了来源，点击链接查看原文确认
   - 如果回答是基于推断（非文档直接信息），向相关同事确认
   - 如果回答内容与你已知事实矛盾，优先相信事实并向文档维护者反馈

4. **反馈与更新**（团队成员 / 2分钟）：
   - 如果发现知识库内容过时或有误，提交文档修复 MR
   - 如果问题答案不在知识库中，考虑是否需要补充文档

#### 6. 输出物

- AI 对问题的回答（含来源引用）
- 知识库更新建议（如有）

#### 7. 验收标准

- [ ] 问题得到了回答（不一定是想要的答案，但 AI 做出了合理的检索和回复）
- [ ] 回答标注了信息来源
- [ ] 结果的准确性已被确认（至少做了来源验证）

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| 知识库内容过时，AI 检索出错误信息 | 团队定期（每月）清理和更新知识库；关键文档标注"最后验证时间" |
| AI 的推断听起来合理但实际错误 | 建立"重要决策必须查阅原文"的文化，不凭 AI 回答做决策 |
| 团队成员查 AI 而非去问直接负责人，可能错过隐性知识 | AI 知识库回答是入口而非终点。对于复杂或关键问题，AI 回答后仍需与负责人确认 |
| 知识库内容非结构化，AI 检索效果差 | 推广知识库规范：每个文档有标题、有标签、有更新时间、有负责人 |

#### 9. 责任人

- **知识库维护者**：对知识库内容的准确性和时效性负责
- **问题提问者**：对 AI 回答的使用负责（验证后再采用）

#### 10. 相关模板

- 知识库文档模板
- 知识库目录结构规范
- 文档 Review 日程表

### SOP-09: AI 辅助新人培训 SOP

#### 1. 目的

用 AI 加速新人的技术上手过程，减少老人对新人的持续打断，让新人能自助解决大部分基础问题。

#### 2. 适用范围

- 场景：新员工入职培训、实习生技术上手、跨团队转岗成员适应期
- 角色：新人、Mentor、Tech Lead

#### 3. 前置条件

- 团队知识库完整（包含系统架构文档、代码规范、开发环境搭建指南、常见问题 FAQ）
- AI 工具已配置项目 CLAUDE.md（包含技术栈、架构约定、开发流程）
- 新人已获得必要的系统权限（代码仓库、数据库只读、日志查看等）

#### 4. 输入材料

- 新人背景信息（技术栈熟悉程度、项目经验、需要上手的模块）
- 团队知识库全部内容
- 新人培训计划
- 新人任务列表（从简单到复杂的任务梯度）

#### 5. 操作步骤

1. **培训环境准备**（Mentor / 30分钟，一次性）：
   - 配置新人的开发环境（按开发环境搭建指南执行）
   - 在 CLAUDE.md 中补充新人专属信息（需要熟悉的模块优先级、代码阅读路径）
   - 为新人创建一份"第一个任务"文档（AI 辅助生成）

2. **新人自助学习**（新人 / 第1-3天）：
   - 新人带着以下 prompt 让 AI 生成一份学习路径：

     ```
     我是一名新加入团队的 Java 开发工程师，之前熟悉 [技术栈背景]。
     团队使用 [项目技术栈]。

     请根据团队知识库，帮我梳理：
     1. 系统架构概览（一句话说清核心业务，5分钟了解模块划分）
     2. 推荐代码阅读顺序（从简单到复杂，每步标注预计阅读时间和关键文件）
     3. 开发环境验证清单
     4. 新人常见坑点（从历史复盘文档中提取）
     5. [第一个任务名称] 的完成指南
     ```

3. **Mentor 定期 Check-in**（Mentor / 每天 15 分钟，第1-2周）：
   - 每天问新人三个问题：今天学到了什么？遇到了什么障碍？AI 帮到你什么？
   - 根据新人反馈调整 AI 辅助策略
   - 补充 AI 回答不了的知识（隐性知识、团队文化、历史上下文）

4. **新人完成任务**（新人 / 第1-4周）：
   - 任务梯度：读代码 → 改配置 → 修简单 Bug → 写小功能 → 参与正常迭代
   - 每完成一个任务，让 AI 生成完成总结，Mentor review

5. **培训评估**（Mentor + Tech Lead / 第2、4周）：
   - 第2周评估：新人能否独立完成简单任务？AI 辅助效果如何？
   - 第4周评估：新人能否独立参与正常迭代？是否需要调整培训计划？

#### 6. 输出物

- 新人个性化学习路径文档
- 新人任务完成记录
- 培训评估报告（第2周 + 第4周）
- 知识库补充建议（新人视角发现的文档不足）

#### 7. 验收标准

- [ ] 新人第1周内完成开发环境搭建和第一个简单任务
- [ ] 新人第4周能独立承担正常迭代任务（不再需要 Mentor 每日 check-in）
- [ ] 新人了解项目核心业务流程和系统架构
- [ ] 新人在遇到问题时，先查询 AI 知识库后提问的习惯已建立
- [ ] 培训过程中的知识空白已补充到团队知识库

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| 新人过度依赖 AI，不主动思考 | Mentor 在 check-in 时通过提问评估理解深度，发现"复制粘贴 AI 答案"的情况及时纠正 |
| AI 给出的学习路径不适合新人的技术背景 | 学习路径必须经过 Mentor 审核和个性化调整 |
| 新人通过 AI 获取的信息可能过时 | 新人遇到"和代码实际行为不一致"的情况时，鼓励直接向 Mentor 确认 |
| Mentor 因为有了 AI 辅助而减少对新人的关注 | AI 辅助不能替代 Mentor 对新人的关怀和融入支持。技术问题找 AI，人文关怀找 Mentor |

#### 9. 责任人

- **Mentor**：对新人的成长进度和融入情况负主要责任
- **Tech Lead**：对新人整体的培训计划和终极评估结果负责
- **新人**：对自身的学习主动性和执行负个人责任

#### 10. 相关模板

- 开发环境搭建指南
- 新人培训 checklist
- 新人任务梯度表（团队维护）

### SOP-10: AI 输出验收 SOP

#### 1. 目的

建立统一的 AI 输出验收标准，确保所有 AI 生成的内容在使用前经过适当级别的验证。

#### 2. 适用范围

- 场景：所有使用 AI 工具生成内容的场景
- 角色：团队全体成员

#### 3. 前置条件

- 团队已了解 AI 的能力边界和常见错误模式
- 团队对不同类型的 AI 输出有明确的验收级别定义

#### 4. 输入材料

- AI 生成的内容
- 该内容所属的验收级别和对应的验收 checklist
- 相关领域的专家评审（如需）

#### 5. 操作步骤

1. **分类 AI 输出**（使用者 / 1分钟）：根据输出物的风险和影响，将其归入以下三级之一：

   | 级别 | 定义 | 示例 | 验收要求 |
   |------|------|------|----------|
   | L1 - 低风险 | 出错不直接影响系统功能或数据 | 周报草稿、会议纪要、代码注释、变量命名建议 | 快速浏览确认无常识性错误 |
   | L2 - 中风险 | 出错会引入 Bug 但不影响核心业务或数据 | 单元测试、CRUD 代码、配置文件、日志语句 | 人工逐条审核 + 自动验证（编译/运行测试） |
   | L3 - 高风险 | 出错会导致资金损失、数据损坏、安全事故 | 资金计算逻辑、数据库变更 DDL、生产配置、安全相关代码 | 人工逐行审核 + 专家评审 + 测试环境验证 + 灰度发布 |

2. **对照 checklist 验收**（使用者 / 时间视级别）：

   **L1 验收 checklist：**
   - [ ] 内容无明显事实错误
   - [ ] 格式符合团队要求
   - [ ] 无明显的逻辑矛盾

   **L2 验收 checklist：**
   - [ ] 编译通过（适用于代码）
   - [ ] 单元测试通过（适用于代码）
   - [ ] 逻辑覆盖了正常路径和至少 2 个异常路径
   - [ ] 边界条件有处理（null、空值、极值）
   - [ ] 没有引入新的警告或静态分析问题
   - [ ] 兼容现有接口和数据格式

   **L3 验收 checklist：**
   - [ ] 全部 L2 检查项
   - [ ] 逐行人工 review 通过
   - [ ] 相关领域专家 review 通过（如 DBA 审核 DDL、安全工程师审核安全代码）
   - [ ] 测试环境验证通过（覆盖正常路径 + 全量异常路径）
   - [ ] 性能测试通过（如适用）
   - [ ] 灰度发布验证通过（如适用）

3. **记录验收结果**（使用者 / 2分钟）：
   - 对于 L2 和 L3 级别，在 MR 描述中记录验收结论
   - 对于发现的 AI 问题，记录到团队 AI 问题知识库

4. **反馈问题模式**（Tech Lead / 每两周）：汇总这段时间 AI 输出中的常见问题模式，更新 CLAUDE.md 或 AI 使用指南，帮助团队避免重复踩坑。

#### 6. 输出物

- 验收通过的 AI 输出物
- 验收记录（L2/L3 级别）
- AI 问题反馈（如有）

#### 7. 验收标准

- [ ] AI 输出的风险级别已正确分类
- [ ] 对应级别的所有验收项已完成
- [ ] L2/L3 级别的验收记录已存档
- [ ] 发现的问题已反馈

#### 8. 风险与注意事项

| 风险 | 应对 |
|------|------|
| 使用者对 AI 输出过于信任，跳过验收步骤 | 通过 CI 自动化强制验收（如 AI 生成代码必须通过编译和测试才能合入）；团队 Leader 定期抽查 |
| 验收流于形式，清单打勾但实际没有认真检查 | 定期回顾因 AI 输出导致的问题，追溯验收环节的不足，更新 checklist |
| AI 输出质量提升后，使用者放松警惕 | AI 的输出质量提升不等于零错误。AI 的"自信"和"正确率"是两回事。始终保持 L2/L3 级别的验收不放松 |
| 验收标准太松或太严，影响实际使用效率 | 每季度回顾验收标准，根据团队 AI 使用经验和问题数据调整 |

#### 9. 责任人

- **AI 输出使用者**：对验收的完整性和有效性负第一责任
- **Tech Lead**：对团队验收标准的制定和执行情况负责
- **Code Reviewer（L2/L3）**：对验收结果负连带责任

#### 10. 相关模板

- AI 输出验收 checklist（按级别分类）
- AI 问题反馈模板
- 团队 AI 使用经验知识库

## 19.3 SOP 使用指南

### 19.3.1 如何定制 SOP

以上 10 份 SOP 是通用模板，在实际落地时需要根据团队情况做定制。定制方向：

**1. 技术栈适配**
- 将 SOP 中的技术术语替换为团队实际使用的技术栈。例如团队使用 MyBatis-Plus 而非 JPA，则将 DAO 层的生成 prompt 中的"Spring Data JPA"替换为"MyBatis-Plus"。
- 将工具名称替换为团队实际使用的工具。例如团队使用 GitLab CI 而非 Jenkins，则对应修改 CI 相关内容。

**2. 团队规模适配**
- 小型团队（3-5 人）：SOP 中的审批环节可以简化。例如 SOP-02（技术设计）中的技术评审会可以简化为"Tech Lead 审核通过即可"。
- 中型团队（10-20 人）：基本可以直接使用完整 SOP。
- 大型团队（30 人以上）：增加跨团队评审环节，增加合规审核环节（尤其金融、医疗行业）。

**3. 团队 AI 成熟度适配**
- 初阶（刚开始用 AI）：先从 SOP-05（单元测试）和 SOP-07（周报）开始，这两个 SOP 风险低、见效快、信心建立容易。
- 中阶（AI 已日常使用）：推 SOP-01（需求分析）、SOP-03（接口开发）、SOP-04（代码 Review），覆盖核心开发流程。
- 高阶（AI 深度集成）：推 SOP-02（技术设计）、SOP-06（生产排障）、SOP-08（知识库问答）、SOP-09（新人培训），覆盖更复杂的决策和分析场景。
- SOP-10（AI 输出验收）建议从第一天就执行。

**4. 行业特性适配**
- 金融/银行行业：增加合规审核节点。SOP-06（生产排障）中需要增加"操作前获得合规审批"的步骤。SOP-10 中增加"数据合规检查"验收项。
- 医疗行业：增加数据隐私检查。所有涉及数据的 SOP 中增加"患者信息脱敏确认"步骤。
- 互联网行业：可放宽验收标准，降低流程复杂度以提高速度。但 SOP-10 的 L3 验收（资金相关）不可降级。

### 19.3.2 如何推动 SOP 执行

SOP 写得再好，不执行等于零。以下是推动执行的具体策略：

**1. Leader 先行**
- TL 自己先用 AI 配合 SOP 产出几份高质量的工作成果（一份技术设计、一份代码 review、一份周报），在团队内分享。
- "我自己这么用了，效果是这样的，你们试试"——比"公司要求大家用 AI"有效十倍。

**2. 嵌入工作流而不是额外负担**
- 把 SOP 的检查要点融入 MR template：MR 描述模板中包含"AI 辅助开发声明"章节，开发者勾选使用了哪些 AI SOP。
- 把 SOP-04（代码 Review）作为 MR 合入的前置条件：机器人自动检查 MR 中是否附带 AI review 报告。

**3. 用数字说话**
- 记录以下基线数据（使用 AI SOP 前）：
  - 平均每个接口开发时间
  - 单元测试覆盖率
  - 代码 review 中发现的问题数
  - 新手上手时间
- 使用 AI SOP 一个月后对比数据，用数字争取团队信任和领导支持。

**4. 允许"违规"但要求"解释"**
- 不强求 100% 遵守 SOP（有些紧急情况确实不适用）。
- 但违反 SOP 时要求明确说明原因（在 MR 描述中写"因紧急修复，跳过 AI review 步骤"）。

**5. 定期回顾和迭代**
- 每两周在团队周会上花 10 分钟回顾"这周 AI SOP 使用中的坑"。
- 每月更新一次 SOP 内容，将团队经验沉淀进去。

### 19.3.3 如何更新 SOP

SOP 不是一成不变的。随着 AI 工具能力的提升和团队经验的积累，SOP 需要持续迭代。

**更新触发条件：**
- AI 工具能力重大升级（如模型版本更新后能力显著变化）
- 团队积累了 3 个以上的同类型问题经验（如多次发现 AI 在某个环节重复产生相同错误）
- 团队技术栈变更（如框架版本升级、引入新组件）
- 每季度的定期审视

**更新流程：**
1. 提出人记录问题：在什么 SOP、什么步骤、遇到了什么问题、建议怎么改。
2. Tech Lead 评估：改动的影响范围有多大，是否与团队实际情况匹配。
3. MR 更新 SOP 文档：像改代码一样改 SOP，通过 MR 提交，团队 review。
4. 周会同步：告知团队 SOP 变更内容和原因。

**版本管理建议：**
- SOP 文档放在团队 Git 仓库中，通过 Git 管理版本历史。
- 每次更新在文档顶部标注更新日期和变更摘要。
- 维护一个 SOP changelog，方便追溯。

## 19.4 本章小结

SOP 是 AI 从"个人玩具"变成"团队武器"的关键一步。没有 SOP，AI 的使用质量取决于每个人的个人能力；有了 SOP，团队就有了统一的质量底线。

实践建议的优先级：

1. **立即启动**（本周）：SOP-10（AI 输出验收），这是所有 SOP 的基础保障。
2. **第一优先级**（第 1-2 周）：SOP-05（单元测试）+ SOP-04（代码 Review），这两个 SOP 风险低、见效快，是最容易让团队感受到 AI 价值的切入点。
3. **第二优先级**（第 3-4 周）：SOP-03（接口开发）+ SOP-07（周报），覆盖日常工作中耗时最大的两个环节。
4. **第三优先级**（第 1-2 月）：SOP-01（需求分析）+ SOP-02（技术设计），覆盖项目前期工作。
5. **持续迭代**（第 2-3 月）：SOP-06（生产排障）+ SOP-08（知识库问答）+ SOP-09（新人培训），覆盖运维和团队协作场景。

记住：SOP 的目的不是增加流程，而是减少混乱。如果某份 SOP 让你的团队做的事情比之前更多了但产出没有提高，说明这份 SOP 需要调整或者不适合你的团队。
