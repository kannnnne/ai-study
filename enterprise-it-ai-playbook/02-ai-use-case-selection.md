# 第2章 如何选择适合 AI 落地的场景

> **目标读者**：Java 后端 Tech Lead，正在评估团队哪些工作可以率先用 AI 提效。
> **本章回答的核心问题**：几十个候选场景摆在你面前，哪个先做？用什么标准判断？

---

## 2.1 场景选择判断框架：十维评分模型

靠直觉选场景容易踩坑。下面这套十维评分模型，给每个候选场景逐项打分，总分直接告诉你优先级。

### 2.1.1 评分表

| 维度 | 权重 | 评分标准（1-5分） | 说明 |
|------|------|------------------|------|
| 重复性 | 高 | 1=一次性任务，5=每天多次重复 | AI 擅长模式复用，高频场景摊薄学习成本 |
| 输入明确性 | 高 | 1=全靠经验判断，5=输入格式固定 | 输入模糊则 AI 猜不准，质量波动大 |
| 输出可验收性 | 高 | 1=纯主观判断，5=可自动验证 | 能自动验收才能迭代优化 prompt |
| 风险可控性 | 高 | 1=出错后果严重（资金/安全），5=出错影响小 | 必须考虑"AI 搞砸了会怎样" |
| 业务判断需求 | 中 | 1=深度业务理解不可替代，5=纯技术操作 | 业务判断越多，AI 越容易翻车 |
| 数据敏感度 | 高 | 1=核心敏感数据（客户/PII），5=公开数据 | 涉及敏感数据需要额外审批和脱敏 |
| 标准化程度 | 中 | 1=每次都不一样，5=高度标准化 | 标准化程度高意味着 pattern 多，AI 学得快 |
| 工具可接入性 | 中 | 1=无 API/CLI 接口，5=API 完善 | 得能通过工具调用把 AI 嵌入工作流 |
| ROI 明确性 | 中 | 1=难以量化收益，5=可精确计算节省时间 | 向上汇报需要数字，不是"应该有用" |
| 团队接受度 | 中 | 1=强烈抵触，5=主动要求 | 再好的工具，没人用等于零 |

### 2.1.2 各维度详解（附 Java 后端实例）

**重复性（高权重）**

- 1 分示例：半年一次的框架版本升级评估，流程每次都不同
- 5 分示例：每次 MR 都要写的单元测试，每天 3-5 次

**输入明确性（高权重）**

- 1 分示例："用户反馈订单列表加载慢，排查一下"——输入只有一句话，要排查的东西千变万化
- 5 分示例：给一个 Controller 类生成 Swagger 文档——输入是固定的 Java 代码，格式确定

**输出可验收性（高权重）**

- 1 分示例：AI 写一份技术方案，好不好全靠 TL 主观评价
- 5 分示例：AI 生成的 DDL 丢进 MySQL 跑一遍，语法错马上知道；生成的单测跑 `mvn test`，覆盖率达标就算合格

**风险可控性（高权重）**

- 1 分示例：AI 修改资金结算逻辑，少一个零就是事故
- 5 分示例：AI 根据日志生成故障复盘草稿，原文还在，写错了也只是草稿质量差

**业务判断需求（中权重）**

- 1 分示例：判断一个需求"值不值得做"，需要权衡商业价值、技术成本、排期冲突
- 5 分示例：把 MySQL 表结构转成 Java Entity 类，纯体力活

**数据敏感度（高权重）**

- 1 分示例：包含真实客户姓名、身份证号、银行卡号的日志分析
- 5 分示例：开源项目的 README 翻译

**标准化程度（中权重）**

- 1 分示例：来自不同客户的各种奇葩格式的导入文件
- 5 分示例：公司内部标准的 RESTful API 接口，统一用 Swagger 2.0 规范

**工具可接入性（中权重）**

- 1 分示例：老旧内部系统，只提供 Web 页面操作，没有 API
- 5 分示例：GitLab CI 有完善的 API 和 Webhook，AI 可以直接提交 MR comment

**ROI 明确性（中权重）**

- 1 分示例："提高代码质量"——怎么衡量？
- 5 分示例：团队 5 人，每人每天写单测 1 小时，AI 生成后验收 15 分钟，每天节省 5 × 45 = 225 分钟

**团队接受度（中权重）**

- 1 分示例：团队成员普遍认为"AI 写的代码不行"，拒绝使用
- 5 分示例：团队主动在群里问"能不能搞个 AI 帮我们写单测"

### 2.1.3 如何使用评分表

对每个候选场景，逐项打 1-5 分，高权重项乘 1.5 系数（中权重乘 1.0），加总得到加权总分。满分约为 60（10个维度，高权重6项×5×1.5 + 中权重4项×5×1.0 = 45 + 20 = 65）。

建议打印成 Excel 表格，团队一起打分——每人打自己的，取中位数，避免个人偏见。

## 2.2 场景分级：从"放心干"到"别碰"

根据十维评分，我把常见的 Java 后端场景分成了四个梯队。

### 2.2.1 适合优先落地（总分 ≥ 40）

**特征**：输入固定、输出可验证、出错影响小、高频重复。满足这四个条件就很稳。

#### 单元测试生成（典型得分：48-52）

| 维度 | 得分 | 理由 |
|------|------|------|
| 重复性 | 5 | 每个类都要写，永远写不完 |
| 输入明确性 | 5 | 被测类代码是固定输入 |
| 输出可验收性 | 5 | `mvn test` 跑过 + JaCoCo 覆盖率报告 |
| 风险可控性 | 5 | 生成错了测不过，不会上线 |
| 数据敏感度 | 4 | 业务代码一般不含生产数据（Mock 层解决） |
| 标准化程度 | 5 | JUnit + Mockito 模式高度统一 |

**落地建议**：从一个模块开始，配好 prompt 模板（包含 JUnit 版本、Mockito 风格、命名规范），CI 里加一个 "AI 生成测试 → 人工审核 → 合并" 的流水线。

#### 代码 Review（典型得分：42-46）

| 维度 | 得分 | 理由 |
|------|------|------|
| 重复性 | 5 | 每个 MR 都要 Review |
| 输入明确性 | 5 | diff 是结构化输入 |
| 输出可验收性 | 3 | 需要人工判断建议是否合理 |
| 风险可控性 | 5 | AI 只给建议，不直接合代码 |
| 数据敏感度 | 4 | 内部代码 |

**落地建议**：CI 中接入 "AI Review → 在 MR 下留言"，人工 Review 时参考 AI 意见。AI 擅长发现空指针、忘记关资源、命名不规范这类机械问题。

#### API 文档生成（典型得分：44-48）

| 维度 | 得分 | 理由 |
|------|------|------|
| 重复性 | 4 | 每个接口写一遍 |
| 输入明确性 | 5 | Controller 代码 + DTO 类 |
| 输出可验收性 | 4 | Swagger UI 渲染无误即可 |
| 风险可控性 | 5 | 文档错了不会影响运行 |
| 标准化程度 | 5 | 公司内部 API 规范统一 |

#### 日志分析（典型得分：40-44）

| 维度 | 得分 | 理由 |
|------|------|------|
| 重复性 | 5 | 每次排障都在做 |
| 输入明确性 | 4 | 日志格式相对固定 |
| 输出可验收性 | 3 | 分析结论需要人工验证 |
| 风险可控性 | 4 | 分析错了只是多花点时间 |
| 工具可接入性 | 4 | ELK/Kibana 有 API |

**落地建议**：把异常栈粘贴给 AI，让 AI 定位根因并给出修复建议。故障复盘时用 AI 生成初版时间线。

### 2.2.2 适合试点（总分 30-39）

**特征**：能省时间但需要较多人工把关，建议小范围验证后再推广。

#### 接口开发：从接口文档生成 Controller + Service + DTO（典型得分：34-38）

- 输入固定（接口文档），但输出需要较多人工校正（业务逻辑是否正确）
- 风险可控——代码编译不通过或测试不过就发现了
- 建议：先用在简单 CRUD 接口，复杂业务逻辑手写

#### 数据库 DDL 生成（典型得分：32-36）

- 输入是 ER 图描述或字段列表，输出是 DDL SQL
- 优势：语法检查零成本，跑一遍 MySQL 就知道对不对
- 风险：索引设计、字段类型选择需要 DBA 复核
- 建议：AI 出初稿 → DBA 审核 → 合并，不要跳过审核

#### 技术方案初稿（典型得分：30-34）

- 输入是需求描述，输出是方案文档
- 最大的问题：输出质量完全依赖人工判断，评分低主要在"输出可验收性(2)"和"业务判断需求(2)"
- 建议：AI 出框架（背景、备选方案对比、风险点）→ 人写关键决策和架构权衡

### 2.2.3 适合个人先试（总分 20-29）

**特征**：不影响团队产出，个人悄悄用了再说。

#### 需求文档整理（典型得分：24-28）

- 痛点：需求方给的文档格式混乱，需要重新梳理成标准模板
- AI 干：把口语化需求描述转成结构化需求文档（用户故事、验收标准、异常流程）
- 风险：整理错了也只是个人用，不影响团队
- 不建议团队推广的原因："输出可验收性(2)"和"业务判断需求(2)"分太低，质量波动大

#### 技术调研（典型得分：22-26）

- AI 干：收集候选方案、对比功能特性、整理社区活跃度数据
- 人不干不行的事：POC 跑一遍看实际性能、判断社区质量（不是 Star 数多就好）
- 风险：AI 可能给出过时信息或幻觉方案

### 2.2.4 暂不适合（总分 < 20）

**特征**：出了事兜不住，或者 AI 根本干不好。

#### 核心金融交易逻辑（典型得分：12-16）

| 维度 | 得分 | 理由 |
|------|------|------|
| 风险可控性 | 1 | 少算一分钱都可能是生产事故 |
| 业务判断需求 | 1 | 交易规则需要资深业务专家判断 |
| 数据敏感度 | 1 | 涉及真实资金数据 |
| 输出可验收性 | 2 | 边界条件太多，测试用例写不完 |

**结论**：不要在资金、交易、对账等核心链路用 AI 生成代码。可以在这些场景的**辅助环节**用（比如生成测试用例、写操作手册）。

#### 安全策略决策（典型得分：10-14）

- 安全策略需要全局视野和攻击者思维，AI 目前不具备
- 误判的后果：该拦的没拦（漏报）或正常的被拦（误报），都是事故
- 可以在辅助环节用：生成安全规则文档、整理漏洞情报

## 2.3 快速判断三问法

不是每次都要打十维表。日常遇到一个任务，用这三个问题快速判断能不能交给 AI。

### 问题 1：输入是否能用文字描述清楚？

**决策树**：

```
输入能否用文字描述？
├── 能，且格式固定 → 非常适合
│   例子：一段 Java 代码、一份 JSON Schema、一个 Stack Trace
├── 能，但需要额外解释背景 → 可以做，但需要写好上下文
│   例子："这个接口是支付回调，当用户支付成功后被调用"
└── 不能，依赖大量隐性知识 → 不适合
    例子："你看了这个业务就应该知道怎么处理"（AI 不知道你脑子里的东西）
```

**自检清单**：

- [ ] 如果让一个刚入职的同事看这个输入，他能理解任务吗？
- [ ] 输入包含的所有术语和缩写，AI 能查到含义吗？
- [ ] 输入有没有隐含的组织架构、历史背景、系统间关系？

### 问题 2：输出是否正确有客观标准？

**决策树**：

```
输出质量如何验证？
├── 自动化验证（编译、测试、lint）→ 非常适合
│   例子：生成的代码编译通过、单测跑过
├── 半自动验证（需要人看一眼）→ 可以做，但要多留 Review 时间
│   例子：技术方案让人扫一遍逻辑
└── 纯主观判断 → 不适合
    例子："这个设计好不好"——10 个人有 11 种看法
```

**自检清单**：

- [ ] 能不能写一段判断逻辑来自动检查 AI 输出是否正确？
- [ ] 如果检查不了，人工判断需要多长时间？
- [ ] 错误率的容忍上限是多少？（5%? 10%?）

### 问题 3：出错后能否及时发现和修复？

**决策树**：

```
AI 出错会怎样？
├── 被自动化流程拦住 → 放心用
│   例子：代码编译不过 CI 直接挂；DDL 语法错数据库报错
├── 人工 Review 能发现 → 可以用，但要配 Review
│   例子：Review 流程中发现 AI 写的文档有不合理的地方
├── 上线后才发现 → 谨慎
│   例子：AI 生成的业务逻辑有隐藏 bug
└── 发现时已造成损失 → 绝对不要用
    例子：AI 建议的安全配置有漏洞，被攻击后才暴露
```

**自检清单**：

- [ ] 出错后的"发现时间"预计多久？（编译时？测试时？上线后？）
- [ ] 发现后修复成本多高？（改一行代码？回滚整个版本？）
- [ ] 有没有灰度机制？（先在小范围用，没问题再推广）

### 三问结果速判

| 问题1 输入 | 问题2 验收 | 问题3 容错 | 结论 |
|-----------|-----------|-----------|------|
| 清晰 | 自动 | 兜得住 | 立刻用 AI |
| 清晰 | 自动 | 兜不住 | 加 Review 后用 |
| 清晰 | 人工 | 兜得住 | 试点，收集反馈 |
| 清晰 | 人工 | 兜不住 | 个人先用 |
| 模糊 | 任意 | 任意 | 先搞清楚输入再谈 AI |

## 2.4 企业场景推荐矩阵

不同企业类型对 AI 的接受度和约束条件差异巨大。下面给出按企业类型的场景切入建议。

### 2.4.1 矩阵总览

| 企业类型 | 推荐起点 | AI 介入深度 | 关键约束 | 主要风险 |
|---------|---------|------------|---------|---------|
| 银行/金融 | 单元测试 + 代码 Review | 浅（辅助层） | 数据不出内网、代码不能上传外部 | 合规红线、数据泄露 |
| 互联网 | API 文档 + 单测 + Review | 深（开发全流程） | 关注社区反馈、迭代速度快 | 团队过度依赖 |
| 传统企业 IT | 日志分析 + 文档整理 | 中（运维+管理） | 系统老旧、API 不全 | 工具链不兼容 |
| 外包/人力派遣 | 接口开发 + 单测生成 | 深（交付层） | 利润率压力大、项目周期短 | 质量不可控导致返工 |

### 2.4.2 各类型详解

#### 银行/金融

**核心矛盾**：收益最高的场景（交易、风控、对账）风险也最高。

**推荐切入口**：
- **第一优先**：单元测试生成、代码 Review（纯内部，数据不出仓库）
- **第二优先**：接口文档生成、日志分析（脱敏后使用）
- **绝对禁止**：任何涉及真实客户数据的场景、核心交易链路代码生成

**落地路径**：
1. 搭建内网 AI 网关（私有化部署或内网 API）
2. CI 中集成 AI 测试生成 + AI Review Comment
3. 制定 AI 使用红线文档（什么场景禁止用 AI，贴在团队 Confluence 首页）
4. 每季度审计一次 AI 使用情况

**关键红线**：
- 涉及 C2 以上密级数据的场景，AI 零接触
- AI 生成的代码必须走完整 Review 流程，不得直接合并
- AI 工具必须支持审计日志（谁、什么时候、生成了什么）

#### 互联网/科技公司

**核心矛盾**：进度压力大，容易一上来就全面铺开。

**推荐切入口**：
- **第一优先**：API 文档、单元测试、代码 Review（见效快，阻力小）
- **第二优先**：接口开发辅助、技术方案初稿（节省核心开发时间）
- **渐进探索**：日志分析 → 故障诊断 → 自动化运维脚本

**常见陷阱**：
- 实习生或新人过度依赖 AI，跳过学习曲线
- 安全策略配置让 AI 决定（不行）
- 技术方案完全不经过人工判断直接采用（幻觉风险）

#### 传统企业 IT

**核心矛盾**：系统老旧、技术栈杂，AI 工具链适配成本高。

**推荐切入口**：
- **第一优先**：日志分析、故障排查辅助
- **第二优先**：运维脚本生成、文档整理、知识沉淀
- **暂缓**：代码生成（老系统不规范，AI 训练数据少，质量差）

**特殊注意事项**：
- 很多老系统没有标准 API，AI 工具可能需要独立部署为内网服务
- 报表、数据导出这类需求先试——输入输出相对标准化
- 运维侧的收益往往大于开发侧（传统企业的开发需求频率低于运维事件）

#### 外包/人力派遣

**核心矛盾**：AI 提效 = 少用人 = 利润上升，但交付质量必须可控。

**推荐切入口**：
- **第一优先**：标准 CRUD 接口开发 + 单测生成（最能压缩交付人天）
- **第二优先**：API 文档、部署脚本、环境配置
- **警惕**：客户特殊业务逻辑不能靠 AI 猜，必须人工确认

**关键风险**：
- 生成的代码风格不统一，客户验收时发现质量问题
- 新人用 AI 快速出代码但不理解，后续维护困难
- **建议**：制定团队 AI 代码规范（prompt 模板 + 必审项目清单），保证交付质量基线

## 2.5 ROI 评估简易模型

向上级汇报"我们要引入 AI"，需要数字不是感觉。下面是个够用不复杂的模型。

### 2.5.1 公式

```
单场景月节省时间 = (原耗时 - AI辅助耗时) × 月频次 × 参与人数

净收益 = 单场景月节省时间 - 学习成本/摊销月 - 验收成本/月

年化 ROI = 净收益 × 12 × 人均时薪 / AI工具年成本
```

### 2.5.2 计算示例：团队引入 AI 生成单元测试

**假设条件**：

| 参数 | 数值 | 说明 |
|------|------|------|
| 团队人数 | 5 人 | 后端开发 |
| 原耗时/天/人 | 60 分钟 | 手写单测 |
| AI 辅助耗时/天/人 | 15 分钟 | 生成 + Review + 微调 |
| 月工作天数 | 22 天 | |
| 前期学习成本 | 10 人时 | Prompt 调试 + 工具配置 + 流程磨合 |
| 摊销月数 | 6 个月 | |
| 月验收成本 | 3 人时/人 × 5 人 | Review AI 生成代码的额外时间 |
| 人均时薪 | 100 元 | 含社保等 |

**计算过程**：

```
月频次 = 22 天
单场景月节省时间 = (60 - 15) 分钟 × 22 天 × 5 人 = 4950 分钟 ≈ 82.5 小时
学习成本摊销 = 10 小时 ÷ 6 = 1.67 小时/月
月验收成本 = 3 × 5 = 15 小时/月
月净收益 = 82.5 - 1.67 - 15 = 65.83 小时
年节省金额 = 65.83 × 12 × 100 元 = 78,996 元
```

**结论**：一个 5 人团队在单测生成这一个场景上，AI 年化节省约 8 万元人力成本。如果你有 10 个团队，就是 80 万——这个数字够你向上级要一笔 AI 工具预算了。

### 2.5.3 ROI 计算注意事项

- **学习成本是一次性的**：第一个场景从零开始至少 2-3 天，第二个场景可能半天就够了
- **验收成本会递减**：第一个月每个人要 3 小时 Review，第三个月可能降到 1 小时（学会了怎么快速判断 AI 输出的质量）
- **不要忘了 AI 工具本身的成本**：API 调用费、私有化部署的服务器成本、商业 License 费用
- **对比基准是"没做"而非"做得很好"**：有些场景（比如日志分析辅助）原来就没专人做，节省的不是人力而是"排查更快了"，这种收益要换算成 MTTR 下降

## 2.6 常见选型错误

以下是见过的最多的五个坑，每个都在真实项目里发生过。

### 错误 1：从最难的地方开始

**典型症状**："AI 这么强，先把最复杂的交易逻辑自动生成搞定。"

**为什么错**：复杂场景的成功率天花板低，失败后团队的结论是"AI 不行"，从此不再尝试。AI 落地需要正反馈循环——第一个场景成功，团队有信心，才会有第二个。

**正确做法**：从十维评分表里选一个总分最高、风险最低的场景切入。先赢一局，再扩地盘。

### 错误 2：忽视数据安全审批

**典型症状**：开发直接把自己的业务代码贴到公网 AI 工具里，被安全部门发现后项目叫停。

**为什么错**：银行、金融、医疗等行业有明确的数据安全红线。代码本身可能就包含表名、字段名、业务逻辑等敏感信息。

**正确做法**：项目启动前，先和安全/合规部门确认什么数据能传、什么不能传、需要什么审批。做完这一步再选工具（内网部署还是 SaaS）。不要等被发现了再补流程。

### 错误 3：把 AI 输出当最终产物

**典型症状**：AI 生成的代码直接上线，没有 Review，没有测试。

**为什么错**：AI 的幻觉是概率性的——100 次里对 95 次，那 5 次错的可能刚好是边界条件。而且 AI 不知道你系统的上下文约束（比如这个字段虽然叫 status 但绝对不能为 NULL）。

**正确做法**：AI 输出 = 初稿。必须走和人类代码一模一样的 Review + CI + 测试流程。降低的不是质量标准，是写初稿的时间。

### 错误 4：没有建立效果反馈循环

**典型症状**：引入了 AI 工具，用了三个月，没人统计过到底省了多少时间、有没有质量问题、团队满不满意。

**为什么错**：没有数据就没法优化——Prompt 要不要改？工具要不要换？要不要扩场景？全靠感觉拍板。

**正确做法**：从第一个月开始就记录三个指标：
- 生成量（AI 产出了多少行代码/文档/建议）
- 采纳率（人有保留多少）
- 返工率（AI 生成的内容有多少比例被后续修改覆盖）

这三个数字告诉你：AI 有没有在干活（生成量）、干得好不好（采纳率）、是否有隐患（返工率）。

### 错误 5：技术选型脱离团队能力

**典型症状**：TL 调研了一圈，选了一个功能最强的 AI 平台，结果团队没人会用，配置复杂，最后没人用。

**为什么错**：工具不是越强越好，是越容易嵌入现有工作流越好。一个好的 AI 工具，团队的第一次成功体验应该在第一天发生——装好就能跑，跑完就看到效果。

**正确做法**：
- 选型时让团队里最"不热心"的那个人试用——他能跑通才说明工具真的好用
- 优先选 IDE 插件或 CLI 工具，而不是额外的 Web 平台（减少切换成本）
- 第一个场景的完整落地周期（从安装到产出）应该在**一周内**完成

## 2.7 本章行动清单

读完本章，建议你按以下步骤推进：

1. **列出候选场景**（10 分钟）：把你团队日常工作中重复性高、格式化程度高的工作列出来，不要过滤，先列出所有
2. **十维打分**（30 分钟）：对每个场景打分，团队一起打，取中位数
3. **三问速判**（10 分钟）：对总分前 5 的场景，用"三问法"再过一遍
4. **选 1 个起点**（5 分钟）：选总分最高且三问完全通过的那个
5. **算 ROI**（15 分钟）：用 2.5 节的模型算一遍，得出一个数字
6. **启动试点**（1 周内）：搭建环境、写好 Prompt、跑通第一次 AI 产出

第一周的目标不是"省了多少时间"，而是"跑通了一种新的工作方式"。拿到第一个成功案例后，再考虑下一个场景。
