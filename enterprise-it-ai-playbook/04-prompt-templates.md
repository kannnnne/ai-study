# 第4章 企业 IT 常用提示词模板库

> 本章收录 20 个经过生产验证的提示词模板，覆盖需求分析、技术设计、代码生成、测试、审查、运维全流程。每个模板均可直接复制使用，无需填入额外信息。

## 4.0 使用说明

### 模板结构约定

每个模板按以下六段式组织：

| 段落 | 说明 |
|------|------|
| **使用场景** | 什么时候该用这个模板，解决什么问题 |
| **适用工具** | 推荐使用的 AI 工具（Claude Code / ChatGPT / Codex CLI 等） |
| **输入材料** | 使用前需要准备好的文档、代码或上下文 |
| **提示词正文** | 可直接复制粘贴的完整 prompt，无需修改 |
| **期望输出** | AI 应该产出什么、什么格式 |
| **使用注意事项** | 容易踩的坑和最佳实践 |

### 通用约定

1. **提示词正文全部为中文指令**，技术术语（类名、注解、SQL 关键字）保留英文。
2. **所有代码生成类模板均内置安全约束条款**，防止 AI 产出有潜在风险的代码。
3. **模板中的 `{占位符}` 需要替换为你的实际内容**。不带花括号的内容即为可直接使用的固定指令。
4. **在 Claude Code 中使用时**，建议将提示词正文保存为项目内的 Markdown 文件，通过 `claude -p "$(cat prompt.md)"` 或直接在对话中粘贴使用。
5. **迭代原则**：第一轮生成的结果通常需要微调，不要期望一次完美。把模板当作起点，根据实际输出继续追加约束。

## 4.1 需求分析类

### 模板1: 需求澄清提示词

**使用场景**: 产品经理或业务方给了一段模糊需求描述，需要 AI 帮助理清边界、拆解功能点、识别遗漏场景。

**适用工具**: Claude Code / ChatGPT / Codex CLI

**输入材料**: 一段原始需求描述（可以是聊天记录、一句话需求、或者粗略的 PRD 草稿）。

**提示词正文**:

```
你是一位资深业务分析师，擅长需求澄清和功能拆解。

请阅读以下原始需求，完成以下分析任务：

【分析步骤】
1. 识别核心业务目标：用一句话概括这个需求要解决什么业务问题
2. 拆解功能模块：将需求拆解为独立的功能点，每个功能点用一个简短标题描述
3. 用户故事映射：为每个功能点写出标准用户故事（作为<角色>，我希望<功能>，以便<价值>）
4. 边界条件检查：列出至少 5 个可能被忽略的边界场景或异常情况
5. 依赖与约束：识别该需求涉及的外部系统依赖、数据依赖和技术约束
6. 优先级建议：按 P0（必须有）/ P1（应该有）/ P2（锦上添花）对功能点分类
7. 待澄清问题清单：列出需要向需求方确认的 5-10 个问题

【输出格式要求】
- 使用 Markdown 表格组织功能模块与优先级的对应关系
- 每个待澄清问题标注"阻塞级别"：阻塞开发 / 影响体验 / 可后续确认
- 如果某个边界场景可能导致需求返工，用 **!!!** 标记

【原始需求】
{粘贴你的需求描述}
```

**期望输出**:

```markdown
## 需求分析报告

### 核心业务目标
一句话概括...

### 功能模块拆解

| 编号 | 功能模块 | 优先级 | 用户故事 |
|------|---------|--------|---------|
| F1 | ... | P0 | 作为...，我希望...，以便... |

### 边界条件检查
1. **!!!** 并发场景下...（可能造成数据不一致）
2. ...

### 待澄清问题

| 编号 | 问题 | 阻塞级别 |
|------|------|---------|
| Q1 | ... | 阻塞开发 |
```

**使用注意事项**:

- 原始需求越模糊，这个模板的效果越明显。如果需求已经写成了完整的 PRD，可以把步骤 1-3 简化为"评审现有 PRD 的完整性"。
- 第 5 步"依赖与约束"在金融、医疗等强监管行业尤为重要，不要跳过。
- 输出中的 `!!!` 标记项建议在开发启动前逐项确认。

### 模板2: 需求转功能清单提示词

**使用场景**: 有了一份会议纪要或需求访谈记录，需要从中提取出结构化的功能清单，确保没有遗漏讨论中提到的任何需求点。

**适用工具**: Claude Code / ChatGPT

**输入材料**: 会议纪要、需求访谈记录、或者多轮沟通的聊天记录。

**提示词正文**:

```
你是一位系统分析师，负责从非结构化的会议记录中提取功能需求。

请仔细阅读以下会议记录，完成以下任务：

【提取规则】
1. 逐段扫描，标记每一段讨论所涉及的功能点
2. 忽略纯管理类话题（排期、人员安排、预算），只提取与技术实现相关的内容
3. 同一个功能点如果在多处被提及，合并为一条，但记录所有提及的上下文
4. 对于每个功能点，推断并记录以下属性：
   - 功能名称（简短、明确）
   - 触发条件（什么情况下用户会用到）
   - 输入/输出（用户给什么，系统返回什么）
   - 业务规则（讨论中提到的约束和规则）
   - 验收标准（如何判断功能已正确实现）
   - 讨论来源（在哪一段、谁提出的）

【输出格式】
生成三张表：

表1: 功能清单总览
| 编号 | 功能名称 | 优先级推断 | 提出人 | 是否达成共识 |

表2: 功能详情
| 编号 | 功能名称 | 触发条件 | 输入 | 输出 | 业务规则 | 验收标准 |

表3: 待澄清项
| 编号 | 关联功能 | 歧义描述 | 建议澄清方式 |

【特殊处理】
- 对于讨论中出现分歧的功能点，在表3中独立列出
- 对于"技术上可实现但需要评估成本"的讨论，在功能名称后标记 [需评估]
- 如果会议记录中完全没有提到某个关键维度（如权限控制、异常处理、性能要求），在表3中主动提出

【会议记录】
{粘贴你的会议记录或访谈原文}
```

**期望输出**: 三张结构化表格，可直接导入需求管理工具（Jira / 飞书多维表格 / Notion）。

**使用注意事项**:

- 如果会议记录很长（超过 5000 字），建议分段输入，每次输入后用"继续，保持格式一致"追加处理。
- 表3 的"主动提出"部分是关键价值：AI 会发现人类讨论时容易遗漏的非功能性需求。
- 建议将输出导入需求管理工具后，找原始参会人确认表1的优先级和表3的待澄清项。

### 模板3: 功能清单转技术方案提示词

**使用场景**: 功能清单已经评审通过，需要生成技术方案文档，包含架构设计、数据库变更、接口定义、核心流程和风险点。

**适用工具**: Claude Code / ChatGPT

**输入材料**: 已确认的功能清单（可以使用模板2的输出），以及项目现有的技术栈信息（框架、数据库、中间件）。

**提示词正文**:

```
你是一位资深 Java 后端架构师，技术栈为 Spring Boot 2.7+ / MyBatis-Plus / MySQL 8.0 / Redis。

请基于以下功能清单，生成一份技术方案文档。

【方案要求】
1. 架构分层说明：描述该需求涉及哪些模块，模块间的调用关系，并画出文字版架构图
2. 数据库变更：
   - 新建表给出完整 DDL（包含字段注释、默认值、索引设计）
   - 修改表现在 ALTER TABLE 语句
   - 说明每个索引的设计理由（覆盖哪些查询场景）
3. API 设计概要：
   - 每个接口列出：路径、HTTP 方法、请求参数、响应结构、错误码
   - 标注哪些接口需要做幂等处理
   - 标注哪些接口需要做权限控制
4. 核心业务流程：
   - 用文字版时序图描述 2-3 个关键流程
   - 每个流程标注事务边界
   - 标注需要发送消息队列或异步处理的环节
5. 缓存策略：说明哪些数据适合缓存，缓存 key 设计、过期策略
6. 风险识别：
   - 列出至少 3 个技术风险（并发、数据一致性、性能瓶颈等）
   - 每个风险给出缓解方案
   - 标注风险等级（高/中/低）

【代码约束】
- 所有数据库操作必须使用 MyBatis-Plus 的 BaseMapper
- 事务管理使用 @Transactional，标注传播行为
- 敏感数据（手机号、身份证号）必须加密存储，日志脱敏
- 所有接口入参使用 @Valid 校验，不允许在业务代码中手动判空
- 异常统一使用全局异常处理器，不返回 500 给前端

【技术栈上下文】
- 后端框架: Spring Boot 2.7.x
- ORM: MyBatis-Plus 3.5.x
- 数据库: MySQL 8.0
- 缓存: Redis 7.x
- 消息队列: RocketMQ（如有异步需求）
- 注册中心: Nacos

【功能清单】
{粘贴功能清单内容}
```

**期望输出**: 一份完整的技术方案文档，可直接作为技术评审的输入材料。

**使用注意事项**:

- 代码约束部分是防御性设计：AI 有概率生成没有参数校验或没有事务管理的代码，提前在提示词中锁定这些要求。
- 如果项目使用的是 JPA 而非 MyBatis-Plus，替换第 1 行和代码约束中的对应信息。
- 技术方案输出建议由团队 Tech Lead 评审后再进入开发，AI 可能在调用链设计上有遗漏。

## 4.2 技术设计类

### 模板4: 技术方案评审提示词

**使用场景**: 已经写好了技术方案文档，让 AI 从多个维度做一次全面的技术评审，发现潜在问题。

**适用工具**: Claude Code / ChatGPT

**输入材料**: 完整的技术方案文档。

**提示词正文**:

```
你是一位技术评审委员会成员，职责是找出技术方案中的缺陷和风险。请保持批判性思维，不要附和方案中已有的结论。

请按照以下检查清单逐项评审以下技术方案：

【检查清单】

### 1. 架构合理性
- 模块划分是否合理？是否存在循环依赖？
- 服务调用链是否过长（超过 3 层）？
- 是否引入了不必要的中间件或组件？

### 2. 数据库设计
- 索引是否覆盖了所有查询条件？
- 是否存在潜在的死锁场景（并发更新多表且顺序不一致）？
- 字段类型选择是否合理（特别是金额字段、枚举字段）？
- 是否有分表需求但方案中未提及？
- 字符集和排序规则是否明确？

### 3. 接口设计
- 是否存在接口粒度过粗或过细的问题？
- 错误码定义是否完整（特别是业务异常的错误码）？
- 是否有不幂等的写接口缺少幂等设计？
- 响应体是否包含了不必要的数据（过度暴露）？

### 4. 安全风险
- 是否存在 SQL 注入、XSS 风险？
- 敏感数据是否正确处理了加密和脱敏？
- 权限校验是否在所有接口的代码路径上覆盖？
- 文件上传是否有类型和大小限制？

### 5. 性能与可靠性
- 是否存在 N+1 查询风险？
- 事务边界是否合理（是否有长事务、大事务）？
- 缓存方案是否考虑了穿透、击穿、雪崩？
- 降级和熔断策略是否明确？

### 6. 可维护性
- 关键业务逻辑是否有清晰的注释和文档？
- 配置项是否支持动态变更（而非硬编码）？
- 是否有必要的监控和告警埋点？

【输出格式】
- 每个发现标注严重程度：🔴 阻塞 / 🟡 建议修改 / 🟢 可选优化
- 🔴 阻塞项必须给出修改方案
- 🟡 建议项给出推荐做法
- 评审结论单独一段，明确给出"通过"/"有条件通过"/"不通过"的结论

【技术方案】
{粘贴你的技术方案文档}
```

**期望输出**: 一份结构化的评审报告，按严重程度排序。阻塞项必须附带具体的修改方案。

**使用注意事项**:

- 这个模板的设计思路是"不附和方案中已有的结论"——如果 AI 只是复述方案中已经写的内容，说明你的方案已经比较完善。
- AI 评审不能替代人类 Tech Lead 评审，但可以覆盖 80% 的常规检查项，让人类聚焦在架构决策层面。
- 对于 AI 标记为 🟡 和 🟢 的项，不需要全部修改——根据项目实际情况选择性采纳。

### 模板5: API 设计提示词

**使用场景**: 已有明确的业务需求描述，需要设计一套 RESTful API，包括路径、方法、请求/响应体、错误码。

**适用工具**: Claude Code / ChatGPT / Codex CLI

**输入材料**: 业务需求描述（功能清单或 PRD 中的功能说明部分）。

**提示词正文**:

```
你是一位 RESTful API 设计专家，擅长设计符合 OpenAPI 3.0 规范的接口。

请基于以下业务需求，设计完整的 RESTful API。

【设计要求】
1. 为每个接口生成 OpenAPI 3.0 兼容的定义，包含：
   - 路径和方法
   - 请求参数（Path / Query / Body），标注必填和类型
   - 响应体结构（成功和失败两种场景）
   - HTTP 状态码和业务错误码

2. URL 设计规则：
   - 使用复数名词表示资源集合（如 /api/v1/orders）
   - 层级嵌套不超过 2 层（如 /api/v1/orders/{orderId}/items 合法，/api/v1/orders/{orderId}/items/{itemId}/logs 不合法，应改为 /api/v1/order-items/{itemId}/logs）
   - 查询参数用于过滤、排序、分页，不用路径承载

3. 响应体统一结构：
   ```json
   {
     "code": 0,
     "message": "success",
     "data": {},
     "timestamp": 1700000000000,
     "traceId": "uuid-string"
   }
   ```
   code=0 表示成功，业务错误码从 10001 开始编排。

4. 分页统一参数：
   - pageNo: 页码（从 1 开始）
   - pageSize: 每页条数（默认 20，最大 100）
   - 响应体 data 中使用 pageInfo 包裹分页信息

5. 每个接口标注：
   - 是否需要认证（JWT Token）
   - 是否需要特定权限（角色或权限编码）
   - 是否需要幂等处理

【业务需求】
{粘贴业务需求描述}
```

**期望输出**:

```markdown
## API 设计文档

### 统一约定
- 基础路径: /api/v1
- 认证方式: Bearer JWT
- 响应结构: {code, message, data, timestamp, traceId}

### 接口列表

#### 1. 创建订单
- **路径**: POST /api/v1/orders
- **认证**: 需要
- **权限**: order:create
- **幂等**: 需要（通过 X-Idempotent-Key 请求头）
- **请求体**:
  ```json
  {
    "productId": 1001,
    "quantity": 2,
    "remark": "请尽快发货"
  }
  ```
- **成功响应** (201):
  ```json
  {
    "code": 0,
    "message": "success",
    "data": {"orderId": "ORD20240701001", "status": "PENDING"},
    "timestamp": 1700000000000,
    "traceId": "..."
  }
  ```
- **错误码**:
  | 错误码 | 说明 |
  |--------|------|
  | 10001 | 商品不存在 |
  | 10002 | 库存不足 |

（后续接口按相同格式输出）
```

**使用注意事项**:

- 第 2 条"层级嵌套不超过 2 层"是在项目实践中反复验证的约束，超过 2 层的 URL 后期维护成本极高。
- 幂等键建议使用 UUID 由客户端生成，或使用业务唯一键（如外部订单号）。
- 如果项目已有统一的响应体结构和错误码规范，请替换提示词中的 JSON 结构。

### 模板6: 数据库设计提示词

**使用场景**: 基于 API 设计文档或功能需求，生成完整的数据库 DDL，包含索引和约束。

**适用工具**: Claude Code / ChatGPT

**输入材料**: API 设计文档（模板5的输出）或详细的功能需求描述。

**提示词正文**:

```
你是一位 DBA，精通 MySQL 8.0 的数据库设计。请基于以下 API 设计生成完整的 DDL。

【设计规范】
1. 命名规范：
   - 表名: t_{模块缩写}_{业务含义}，如 t_ord_order, t_ord_order_item
   - 字段名: 小写蛇形命名，如 create_time, order_status
   - 索引名: idx_{表缩写}_{字段名}，唯一索引用 uk_{表缩写}_{字段名}
   - 主键: 统一使用自增 BIGINT id

2. 必备字段（每张表必须包含）：
   - id BIGINT AUTO_INCREMENT PRIMARY KEY COMMENT '主键ID'
   - create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间'
   - update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
   - is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除标记:0-未删除,1-已删除'

3. 字段类型约束：
   - 金额字段: DECIMAL(18,2)，不允许使用 FLOAT/DOUBLE
   - 状态枚举: TINYINT + COMMENT 中说明取值含义，或 VARCHAR(32) 字符串枚举
   - 手机号: VARCHAR(20)（需加密存储，存密文）
   - 身份证号: VARCHAR(64)（需加密存储，存密文）
   - 文本内容: 超过 500 字符用 TEXT，超过 65535 字符用 MEDIUMTEXT
   - 布尔标记: TINYINT(1) DEFAULT 0

4. 索引设计：
   - WHERE 条件、JOIN 条件、ORDER BY 字段必须建索引
   - 区分度高的字段优先
   - 联合索引遵循最左前缀原则，等值条件在前、范围条件在后
   - 每个索引写一行 COMMENT 说明覆盖的查询场景
   - 单表索引数量不超过 5 个（含主键）

5. 字符集: utf8mb4，排序规则: utf8mb4_general_ci

6. 引擎: InnoDB

### 模板8: Java Service 生成提示词

**使用场景**: 有明确的业务逻辑描述，需要生成 Service 层代码，包含事务管理、异常处理和业务校验。

**适用工具**: Claude Code / ChatGPT / Codex CLI

**输入材料**: 业务逻辑的文字描述或伪代码，以及相关的数据表结构。

**提示词正文**:

```
你是一位资深 Java 后端开发，精通 Spring Boot 事务管理和业务逻辑设计。

请根据以下业务需求生成 Service 层代码。

【设计原则】
1. Service 层负责业务逻辑编排，不包含 SQL 操作（SQL 放在 Mapper/DAO 层）
2. 复杂业务拆分为多个私有方法，每个方法单一职责
3. 优先使用函数式编程（Optional, Stream）处理集合和判空，禁止深层 if-else 嵌套
4. 状态流转使用枚举 + switch 或状态机模式，禁止魔法值

【事务管理】
- @Transactional(rollbackFor = Exception.class) 放在需要事务的方法上
- 只读查询方法使用 @Transactional(readOnly = true)
- 事务方法内不进行远程RPC调用、不发送MQ消息（在事务提交后再发送）
- 数据库更新操作必须先查再改，禁止使用 UPDATE ... WHERE ... 直接更新后不校验影响行数

【异常处理】
- 业务异常抛出自定义 BusinessException，携带错误码和错误信息
- 调用外部服务时必须 try-catch，包装为业务异常后抛出
- 不允许 catch 后只打日志不处理（吞异常）

【安全与规范】
- 敏感数据（手机号、身份证号）写入前必须加密，读取后在使用点解密
- 金额计算使用 BigDecimal，禁止使用 double/float
- 禁止使用 new BigDecimal(double)，必须使用 new BigDecimal(String)
- 除数为零、空指针、数组越界必须有前置防护
- 所有公开方法必须写 Javadoc（描述功能、参数含义、返回值、异常）
- 关键业务节点写 info 日志，日志中携带可追踪的业务 ID

【业务需求】
{粘贴业务需求和表结构}
```

**期望输出**: 完整的 Service 类，包含所有公开方法和关键私有方法，带 Javadoc 和事务注解。

**使用注意事项**:

- 事务内不发送 MQ 的原则在分布式场景尤为重要：如果事务回滚但 MQ 已发送，会造成数据不一致。解决方案是使用事务消息或本地消息表。
- `new BigDecimal(double)` 的禁止规则是 Java 经典坑：`new BigDecimal(0.1)` 的结果不是 0.1 而是 `0.100000000000000005551115...`，精度丢失来源于 double 的二进制表示。
- 生成后的代码重点检查事务边界是否合理——AI 有时会把只读操作也包在大事务里。

### 模板10: 单元测试生成提示词（JUnit 5 + Mockito）

**使用场景**: 已有完整的 Service 类，需要生成全面的单元测试，覆盖正常路径、异常路径和边界条件。

**输入材料**: Service 类的完整源代码。

**提示词正文**:

```
你是一位资深 Java 测试工程师，精通 JUnit 5 和 Mockito。请为以下 Service 类生成完整的单元测试代码。

【测试要求】
1. 测试框架: JUnit 5 (Jupiter) + Mockito + AssertJ
2. 测试类命名: {ServiceName}Test，放在 src/test/java 对应的包路径下
3. 每个测试方法必须有 @DisplayName("中文描述") 注解
4. 测试覆盖率目标: 行覆盖 >= 80%，分支覆盖 >= 70%

【必须覆盖的测试场景】
对于每个公开方法，至少覆盖以下 4 类场景：

1. 正常路径（Happy Path）：
   - 输入合法参数，验证返回结果的正确性
   - 验证依赖的 Mapper/Service 方法被正确调用

2. 异常路径（Exception Path）：
   - Mock 依赖抛出异常，验证 Service 是否正确处理或向上抛出
   - 模拟数据库操作失败场景

3. 边界条件（Boundary）：
   - 空参数、空集合、null 值
   - 数值为 0、负数、最大值
   - 集合为空、只有一条、超大集合

4. 业务规则覆盖（Business Rules）：
   - 状态流转的所有合法路径和非合法路径
   - 权限校验的通过和不通过场景

【代码模板】
```java
@ExtendWith(MockitoExtension.class)
class XxxServiceTest {

    @Mock
    private XxxMapper xxxMapper;

    @InjectMocks
    private XxxService xxxService;

    // 每个测试方法结构:
    @Test
    @DisplayName("创建订单 - 库存充足，应返回订单ID且状态为待支付")
    void createOrder_whenStockSufficient_shouldReturnPendingOrder() {
        // Given: 准备测试数据和 Mock 行为
        CreateOrderDTO dto = new CreateOrderDTO();
        dto.setProductId(1L);
        dto.setQuantity(2);

        Product product = new Product();
        product.setId(1L);
        product.setStock(100);
        product.setPrice(new BigDecimal("99.00"));

        when(xxxMapper.selectById(1L)).thenReturn(product);
        when(xxxMapper.insert(any(Order.class))).thenReturn(1);

        // When: 执行被测方法
        Result<OrderVO> result = xxxService.createOrder(dto);

        // Then: 验证结果和交互
        assertThat(result.getCode()).isEqualTo(0);
        assertThat(result.getData().getStatus()).isEqualTo("PENDING");
        verify(xxxMapper, times(1)).selectById(1L);
        verify(xxxMapper, times(1)).insert(any(Order.class));
    }
}
```

【Mock 规范】
- 不 Mock 值对象（DTO、VO、Entity），只 Mock 有行为的依赖（Mapper、外部Service）
- 不要使用 Mockito.any() 而不传参数，使用 any(Xxx.class) 明确类型
- verify() 只在确实需要验证调用次数时使用，不过度使用

【禁止事项】
- 禁止在测试中连接真实数据库（使用 @Mock 替代 @SpringBootTest）
- 禁止测试方法之间有依赖关系（每个测试独立运行）
- 禁止在测试中使用 Thread.sleep() 等待异步结果
- 禁止因为覆盖率而写无断言的测试（每个测试必须有 assertThat 或 verify）

【源代码】
{粘贴 Service 类的完整源代码}
```

**期望输出**: 完整的测试类，每个测试方法有清晰的 Given-When-Then 结构和 @DisplayName 中文描述。

**使用注意事项**:

- 要求 AI"不要 Mock 值对象"是一个有效约束：如果测试中 Mock 了 DTO 对象，那测试就失去了验证数据流转的意义。
- Given-When-Then 结构是团队协作的关键——新人通过读测试理解业务逻辑，如果测试结构混乱，这个价值就丢失了。
- 生成后的测试建议先跑一次，AI 可能在 Mock 参数匹配上有细微偏差导致编译不通过。

### 模板12: 代码 Review 提示词

**使用场景**: 提交代码前，让 AI 做一轮全面的代码审查，覆盖安全、性能、Bug、风格四个维度。

**适用工具**: Claude Code / ChatGPT / Codex CLI

**输入材料**: 需要审查的代码文件（可以是一个类，也可以是一个 PR 的 diff）。

**提示词正文**:

```

【审查维度】

### 1. 安全审查（高优先级）
- SQL 注入风险：是否存在 ${} 拼接 SQL、字符串拼接查询条件
- XSS 风险：是否对用户输入做了转义处理
- SSRF 风险：是否存在根据用户输入构造 URL 并请求的情况
- 权限绕过风险：敏感接口是否都有 @PreAuthorize 或等效的权限注解
- 敏感信息泄露：日志中是否打印了密码、Token、手机号、身份证号
- 任意文件读取：文件路径是否来自用户输入且未做路径穿越校验

### 2. Bug 审查（高优先级）
- 空指针风险：Optional 使用不当、未判空的 getter 链式调用
- 并发问题：共享变量的线程安全问题、SimpleDateFormat 的并发使用
- 事务问题：事务方法内调用了异步方法、捕获异常后未回滚
- 精度丢失：金额使用 double/float 而非 BigDecimal
- 死循环风险：while 循环缺少明确的退出条件或超时机制
- 资源泄漏：未关闭的流、未释放的连接

### 3. 性能审查（中优先级）
- N+1 查询：循环内查询数据库
- 大事务：事务内包含 RPC 调用、MQ 发送、文件操作
- 缺失索引提示：新的查询条件可能导致全表扫描
- 不合理的缓存使用：缓存了大对象、过期时间设置不合理
- 大量数据的全量加载：SELECT * + 一次性加载全部到内存

### 4. 代码风格审查（低优先级）
- 命名规范：类名、方法名、变量名是否符合驼峰规范
- 魔法值：硬编码的数字和字符串
- 注释质量：注释是否与代码实际行为一致
- 方法长度：单个方法是否过长（建议不超过 50 行）
- 重复代码：是否有可抽取的公共逻辑

【输出格式】
- 每个发现用以下格式：
  > **[{严重程度}]** {问题描述}
  > **位置**: {文件名}:{行号或方法名}
  > **问题**: {具体说明为什么这是个问题}
  > **建议**: {具体的修改方案，最好给出代码示例}
  > **参考**: {相关的最佳实践或规范文档链接}

- 严重程度定义：
  - 🔴 严重：必须在合并前修复（安全漏洞、数据丢失风险）
  - 🟡 重要：建议在本次迭代中修复（Bug、性能问题）
  - 🟢 建议：后续优化即可（风格、命名）

【代码】
{粘贴需要审查的代码}
```

**期望输出**: 按严重程度排序的发现列表，每个发现包含问题描述、位置、原因、修改建议。

**使用注意事项**:

- AI 代码审查不能替代人类审查，但能覆盖大量机械性检查，让人类审查者聚焦在业务逻辑和架构决策上。
- 对于 🔴 严重问题，AI 的判断可能有误报。例如 AI 标记了一段遍历 Map 的代码为"空指针风险"，但实际代码上下文已经保证了非空。人工确认即可。
- 建议在 CI 流水线中加入这个模板，对每个 PR 自动执行一轮 AI 审查。

### 模板13: Bug 分析提示词

**使用场景**: 线上出现问题，需要根据错误描述和堆栈信息快速定位根因。

**适用工具**: Claude Code / ChatGPT / Codex CLI

**输入材料**: Bug 描述（用户操作步骤、现象）、错误日志（堆栈信息）。

**提示词正文**:

```
你是一位资深后端排障工程师，请分析以下 Bug 的根因。

【分析流程】
请按照以下步骤逐层分析，不要跳跃：

第一层：现象还原
- 用通俗语言描述用户看到了什么现象
- 这个现象影响了哪些用户（所有用户 / 特定用户 / 特定时间段）

第二层：堆栈分析
- 逐行解读堆栈信息：
  - 异常类型是什么（NullPointerException / SQLException / TimeoutException / ...）
  - 异常抛出的确切位置（哪个类、哪个方法、第几行）
  - 调用链是怎样的（从入口到异常点的完整链路）
- 如果是 SQLException，分析 SQL 语句本身是否有问题
- 如果是 NullPointerException，定位哪个变量为空

第三层：根因推测
- 列出 2-3 个最可能的根因，按概率排序
- 对每个可能根因给出推理依据
- 标注是否需要查看更多信息才能确认（如需要，明确指出需要什么信息）

第四层：影响评估
- 该 Bug 影响的业务范围（是否会导致数据错误、资金损失、用户体验下降）
- 紧急程度建议：立即修复 / 24小时内修复 / 下个迭代修复

第五层：修复方案
- 给出具体的代码修改方案（修改哪个文件、哪个方法、怎么改）
- 标注修复的风险点（修改是否会影响其他功能）
- 建议回归测试的范围

【Bug 描述】
{粘贴用户反馈的 Bug 描述}

【错误日志】
{粘贴堆栈信息和相关日志}
```

**期望输出**: 按五层结构组织的分析报告，每层有明确的结论。

**使用注意事项**:

- 如果堆栈信息不完整（例如只有一行异常信息），提示 AI "请根据当前信息做最大限度的推断，并标注哪些部分是推测"。
- 第三层"2-3 个最可能的根因"比单一下结论更重要——AI 不是万能的，多个假设给开发人员提供了排查方向。
- 对于与数据库相关的 Bug，建议同时提供 EXPLAIN 结果和慢查询日志，AI 的分析会更精准。

### 模板14: 日志分析提示词

**使用场景**: 收到一段生产环境日志，需要从中提取异常模式、识别性能瓶颈、发现安全事件。

**适用工具**: Claude Code / ChatGPT

**输入材料**: 日志文件片段（可以是 ELK 导出的搜索结果，或者直接 tail 的生产日志）。

**提示词正文**:

```
你是一位运维工程师，擅长从海量日志中发现异常模式。请分析以下日志片段。

【分析维度】
请按以下优先级逐项分析：

1. 异常识别（最高优先级）：
   - 列出所有 ERROR 和 FATAL 级别的日志，按出现次数排序
   - 对每条异常日志，判断是偶发还是持续（根据时间分布）
   - 标记首次出现时间和最后出现时间
   - 如果是偶发异常，推测触发条件

2. 性能分析（高优先级）：
   - 提取所有接口的响应时间，计算 P50 / P95 / P99
   - 识别响应时间 > 3 秒的慢请求
   - 标注慢请求的时间分布（是否集中在某个时间段）
   - 分析慢请求之间是否有共同的调用链特征

3. 安全事件检测（高优先级）：
   - 统计每个 IP 的请求频率，标记异常高频 IP（> 100 次/分钟）
   - 识别连续失败认证的记录（可能为暴力破解）
   - 检测 SQL 注入、XSS 攻击的特征 payload
   - 标记非业务时段的异常访问（凌晨 2-5 点）

4. 业务模式分析（中优先级）：
   - 统计各接口的调用频率 Top 10
   - 识别业务高峰期的时间段
   - 标注明显的调用量突增或突降的时间点

5. 系统健康检查（中优先级）：
   - GC 日志分析：Full GC 次数和耗时
   - 连接池状态：是否有连接获取超时的记录
   - 线程池状态：是否有任务被拒绝的记录

【输出格式】
每个发现用以下格式：

> **[{类别}]** {一句话描述发现}
> **证据**: {引用具体日志行}
> **影响**: {这个发现可能造成什么后果}
> **建议**: {如何处理}

【日志片段】
{粘贴日志内容}
```

**期望输出**: 按优先级分类的发现列表，每个发现附带具体日志作为证据。

**使用注意事项**:

- 如果日志量很大（数万行），先 grep 过滤出关键信息再输入。例如 `grep -E "ERROR|WARN|timeout|slow" app.log | tail -500`。
- 安全事件检测（第 3 点）在金融、政务行业是刚需——很多团队在渗透测试后才发现已经在日志中暴露了攻击特征。
- 如果公司有 ELK/Grafana，把这个模板的思路融入告警规则，实现自动化的异常检测。

### 模板15: SQL 优化提示词

**使用场景**: 发现一条慢 SQL，需要分析原因并给出优化方案。

**适用工具**: Claude Code / ChatGPT

**输入材料**: 慢 SQL 语句、EXPLAIN 结果、表结构 DDL 和相关索引信息。

**提示词正文**:

```
你是一位 DBA，精通 MySQL 8.0 的 SQL 优化。请分析以下慢 SQL 并提出优化方案。

【输入信息】
1. 慢 SQL 语句
2. EXPLAIN 执行计划
3. 涉及表的 DDL（含索引）
4. 该表的大致数据量

【分析步骤】
请按以下步骤分析：

### 步骤一：EXPLAIN 解读
逐字段解读 EXPLAIN 结果：
- type: 访问类型（ALL > index > range > ref > eq_ref > const）。ALL 和 index 通常需要优化
- key: 实际使用的索引。如果和 possible_keys 不一致，分析原因
- rows: 预估扫描行数。如果 rows 远大于实际返回行数，说明过滤不充分
- Extra: 特别关注 Using filesort（需要排序优化）、Using temporary（需要临时表优化）、Using where（需要索引覆盖优化）

### 步骤二：问题诊断
判断慢 SQL 的具体原因：
- 索引缺失或索引失效（函数包裹字段、隐式类型转换、OR 条件、LIKE '%xx' 前缀模糊）
- 回表过多（需要通过覆盖索引优化）
- JOIN 顺序不合理（小表驱动大表原则）
- 子查询未优化（考虑改写为 JOIN）
- 数据量过大（考虑分页优化、归档）
- 锁竞争（分析是否有行锁、表锁等待）

### 步骤三：优化方案
按以下优先级给出优化建议：
1. 索引优化（最快见效）：
   - 新建索引或调整联合索引的字段顺序
   - 给出具体的 CREATE INDEX 语句
   - 分析新索引对写入性能的影响
2. SQL 改写：
   - 给出改写后的完整 SQL
   - 说明改写前后语义的等价性
3. 表结构调整：
   - 是否需要分区、分表、归档
4. 应用层优化：
   - 是否需要加缓存、异步处理

### 步骤四：验证方案
- 说明如何验证优化效果（对比 EXPLAIN 前后的 rows 变化）
- 建议在哪个环境先测试（不建议直接上生产）

【输入】
SQL: {粘贴慢 SQL}
EXPLAIN: {粘贴 EXPLAIN 结果}
DDL: {粘贴涉及表的 CREATE TABLE 语句}
数据量: {该表大致行数}
```

**期望输出**: 按四步结构组织的优化报告，优化方案优先索引优化，附带可执行的 SQL 语句。

**使用注意事项**:

- 如果只有慢 SQL 没有 EXPLAIN 结果，先执行 `EXPLAIN {慢SQL}` 获取执行计划再输入——没有执行计划的优化分析是盲目的。
- 索引优化的副作用也要关注：每增加一个索引，INSERT/UPDATE/DELETE 的性能都会下降。如果表写入频繁，索引不宜过多。
- AI 建议的索引名可能不符合团队的命名规范，记得替换为项目统一的命名规则。

## 4.5 安全检查类

### 模板16: 安全风险检查提示词

**使用场景**: 对 Java 后端代码做一次全面的安全审计，重点覆盖 OWASP Top 10。

**适用工具**: Claude Code / ChatGPT / Codex CLI

**输入材料**: 需要审计的 Java 代码（建议按模块输入，单次不要超过 3000 行）。

**提示词正文**:

```
你是一位应用安全专家（AppSec Engineer），请对以下 Java/Spring Boot 代码进行安全审计。

【审计范围：OWASP Top 10 2021】

1. A01:2021 - 访问控制失效
   - 检查所有 Controller 方法是否都有明确的权限注解（@PreAuthorize / @Secured）
   - 检查是否存在水平越权风险（用户 A 能否访问用户 B 的数据）
   - 检查是否有接口未做认证即可访问

2. A02:2021 - 加密机制失效
   - 检查敏感数据（密码、手机号、身份证号）的存储和传输是否加密
   - 检查密码是否使用弱哈希算法（MD5、SHA1）
   - 检查密钥是否硬编码在代码中
   - 检查 JWT Token 的签名密钥是否足够强壮

3. A03:2021 - 注入攻击
   - 检查是否存在 SQL 注入（${} 拼接、字符串拼接 SQL）
   - 检查是否存在命令注入（Runtime.exec() 的参数来自用户输入）
   - 检查是否存在 LDAP 注入、XPath 注入

4. A04:2021 - 不安全的设计
   - 检查是否有过度暴露的错误信息（堆栈信息返回给前端）
   - 检查是否有不安全的默认配置（debug=true、trace 开启）

5. A05:2021 - 安全配置错误
   - 检查 CORS 配置是否过于宽松（allowOrigin="*" 且 allowCredentials=true）
   - 检查 CSRF 防护是否启用
   - 检查是否有不安全的 HTTP 方法暴露（TRACE、OPTIONS 返回敏感头信息）

6. A06:2021 - 易受攻击和过时的组件
   - 检查 pom.xml / build.gradle 中的依赖版本是否存在已知漏洞（CVE）
   - 重点检查：Fastjson、Log4j、Spring Framework 的版本

7. A07:2021 - 身份识别和认证失败
   - 检查登录接口是否有暴力破解防护（验证码、限流）
   - 检查密码重置流程是否有安全漏洞
   - 检查会话管理是否安全（Token 过期、退出登录是否失效）

8. A08:2021 - 软件和数据完整性故障
   - 检查是否有不安全的反序列化（ObjectInputStream 使用未校验）
   - 检查文件上传是否校验了文件类型（不只依赖扩展名）

9. A09:2021 - 安全日志和监控故障
   - 检查登录成功/失败是否记录了审计日志
   - 检查敏感操作（资金交易、权限变更）是否记录了操作日志
   - 检查日志中是否记录了敏感信息（密码、Token、身份证号）

10. A10:2021 - SSRF（服务端请求伪造）
    - 检查是否存在用户可控的 URL 被服务端发起请求
    - 检查 RestTemplate / HttpClient 的 URL 来源是否经过白名单校验

【输出格式】
每个发现按以下格式输出：

> **[{OWASP 编号}] {问题标题}**
> **严重程度**: 🔴严重 / 🟡高危 / 🟢中危 / ⚪低危
> **位置**: {文件路径}:{行号或方法名}
> **描述**: {具体的问题描述}
> **攻击场景**: {攻击者如何利用这个漏洞}
> **修复方案**: {具体的代码修改建议}
> **验证方法**: {如何验证修复是否有效}

【代码】
{粘贴需要审计的代码}
```

**期望输出**: 按 OWASP Top 10 分类的安全发现列表，每个发现含攻击场景和具体修复方案。

**使用注意事项**:

- 安全审计的结果需要人工确认——AI 可能将正常业务逻辑标记为漏洞（误报），也可能遗漏真正的漏洞（漏报）。
- A06（依赖版本漏洞）建议配合 `mvn dependency-check:check` 或 `snyk test` 等工具交叉验证。
- 生产环境的安全审计建议每季度执行一次，每次使用项目最新代码重新审计。

### 模板17: 性能风险检查提示词

**使用场景**: 对 Java 后端代码做性能风险识别，发现 N+1 查询、大事务、缺失缓存等常见性能陷阱。

**适用工具**: Claude Code / ChatGPT / Codex CLI

**输入材料**: 需要检查的 Java 代码（通常是 Service 层和 Mapper 层）。

**提示词正文**:

```
你是一位性能工程师，请对以下 Java/Spring Boot 代码进行性能风险评估。

【检查项目】

### 1. 数据库访问
- 循环内查询数据库（N+1 问题）：标注代码行，
  例：for (Order order : orders) { order.setItems(itemMapper.selectByOrderId(order.getId())); }
- 缺少批量操作：逐条 INSERT/UPDATE 应改为 batchInsert/batchUpdate
- SELECT * 查询：是否查询了不必要的字段（特别是大字段 TEXT/BLOB）
- 缺失分页：LIST 查询是否没有分页限制
- 索引使用不合理：WHERE 条件是否会导致索引失效（函数包裹、前置模糊 LIKE）

### 2. 事务管理
- 事务粒度过大：事务内是否包含无关操作（日志记录、通知推送、文件处理）
- 事务内远程调用：@Transactional 方法内是否调用了 RPC / HTTP / MQ
- 长事务：事务边界内是否有用户交互或复杂计算
- 锁竞争：SELECT ... FOR UPDATE 是否合理，是否存在死锁风险

### 3. 缓存策略
- 高频读取无缓存：频繁查询且数据变化不频繁的场景是否缺少缓存
- 缓存 key 设计不当：key 是否包含可变信息导致缓存命中率低
- 缓存不一致风险：更新数据库后是否同步更新/失效缓存
- 缓存穿透风险：查询不存在的数据时是否会每次都查库

### 4. 内存与GC
- 大对象加载：是否一次性从数据库加载大量数据到 List
- 内存泄漏风险：ThreadLocal 使用后是否清理、静态集合是否无限增长
- 字符串拼接：循环内是否使用 + 拼接字符串（应使用 StringBuilder）
- 不必要的大对象创建：BigDecimal 频繁 new 是否可复用常量（BigDecimal.ZERO, BigDecimal.ONE）

### 5. 异步与线程池
- 主线程阻塞：是否有同步等待异步结果的操作
- 线程池配置不合理：核心线程数、队列大小是否与业务量匹配
- 缺少超时控制：外部调用是否设置了合理的超时时间

【输出格式】
每个发现使用以下格式：

> **[{类别}] {问题简述}**
> **位置**: {类名}.{方法名}:{行号}
> **风险等级**: 🔴高 / 🟡中 / 🟢低
> **描述**: {为什么这是个性能问题}
> **触发条件**: {什么场景下会触发（低并发下可能不明显）}
> **优化建议**: {具体的优化方案，附带代码对比}

> **优化前**:
> ```java
> {问题代码}
> ```
>
> **优化后**:
> ```java
> {优化后代码}
> ```

【代码】
{粘贴需要检查的代码}
```

**期望输出**: 按类别分组的性能问题清单，高风险项附带"优化前/优化后"代码对比。

**使用注意事项**:

- 分析代码层面的性能风险不等于分析系统性能——真正的性能瓶颈还需要结合压测数据和监控指标来判断。
- N+1 问题是最高频的性能陷阱，建议将检查 N+1 作为 Code Review 的必检项。
- 优化建议中的代码对比部分最具价值——开发人员可以直接理解"当前写法有什么问题"和"应该怎么写"。

## 4.6 日常提效类

### 模板18: 周报生成提示词

**使用场景**: 每周五需要汇总本周工作产出，生成结构化周报。

**适用工具**: Claude Code / ChatGPT

```

【周报结构】

### 一、本周工作总结
按项目/模块分组，每组列出完成的功能点和修复的问题。
格式：- [类型] 功能描述（例如：- [功能] 新增订单导出Excel功能）

- 完成的任务数
- 修复的 Bug 数

### 五、需要支持的事项
列出需要其他团队或领导协调的事项

【周报风格要求】

【输入材料】
Git Log（本周提交记录）:
{粘贴 git log --author="你的名字" --since="7 days ago" --oneline}

任务记录:
{粘贴 Jira/TAPD/飞书任务的本周完成记录}

工作笔记:

```

**期望输出**: 一份结构化的周报，可直接发送或粘贴到周报系统。

**使用注意事项**:

- `git log --author="你的名字" --since="7 days ago" --oneline` 是快速获取本周提交记录的方法。如果团队使用 squash merge，建议加上 `--no-merges` 过滤合并提交。
- 周报风格要求中的"不写没有具体产出的描述"很重要——AI 在没有明确约束时会生成大量看起来很忙但信息量为零的文字。

### 模板19: 会议纪要整理提示词

**使用场景**: 开完会后有一份录音转文字稿或会议笔记，需要整理成结构化的会议纪要。

**适用工具**: Claude Code / ChatGPT

**输入材料**: 会议录音转文字稿（飞书妙记、通义听悟等工具生成）或手写/打字笔记。

**提示词正文**:

```
你是一位会议记录员，请将以下会议原始记录整理为结构化会议纪要。

【会议纪要结构】

### 会议基本信息
- 会议主题: （从记录中提取，提取不到则写"待补充"）
- 日期: （从记录中提取）
- 参会人: （从记录中提取）
- 记录人: （待填写）

### 核心结论
用 3-5 条要点概括本次会议最重要的结论。每条不超过 30 字。

### 讨论议题与决议

每个议题按以下格式：
**议题 N: {议题名称}**
- 背景: （一句话说明为什么要讨论这个议题）
- 讨论要点:
  - 观点A（提出人）: ...
  - 观点B（提出人）: ...
- 决议: （最终决策是什么，有分歧则标注"未达成一致，待下次讨论"）

### 待办事项 (Action Items)

| 编号 | 任务 | 负责人 | 截止日期 | 优先级 | 备注 |
|------|------|--------|---------|--------|------|

### 遗留问题
列出本次会议未解决、需要后续跟进的问题。

### 下次会议预告
- 建议时间:
- 建议议题:

【整理规则】
1. 去除口语化的语气词（"那个""就是说""然后然后"等）
2. 技术术语保持原样，不做翻译
3. 会议中出现的人名、日期、数字务必准确保留
4. 讨论中如果有明显的错误信息（说错了马上纠正），只保留正确版本
5. 待办事项必须明确到可执行的程度，不允许"进一步讨论""后续跟进"这类模糊描述
6. 如果原始记录中有不完整的地方，用 [待补充] 标记，不要编造

【原始会议记录】
{粘贴会议录音转文字或会议笔记}
```

**期望输出**: 一份按照以上结构组织的会议纪要，待办事项明确到人和时间。

**使用注意事项**:

- 如果原始记录来自语音转文字工具，会有大量识别错误（人名、专业术语），建议先人工修正关键术语再输入。
- 第 5 条"不允许模糊描述"是关键约束——AI 有很强的"套话"倾向，不加这个约束它会生成看似完整实则无法执行的待办事项。
- 会议的"决议"比"讨论要点"更重要——一份好的纪要让没参会的人看完就知道"结论是什么"和"我要做什么"。

### 模板20: Claude Code / Codex 长任务提示词

**使用场景**: 需要 AI 执行一个跨多个文件、多步骤的复杂开发任务，要求按步骤执行、每步检查、不跳步。

**适用工具**: Claude Code / Codex CLI

**输入材料**: 详细的任务描述，包括每个步骤的输入和预期输出。

**提示词正文**:

```
## 任务: {一句话描述本次任务目标}

### 背景
{简要描述任务背景、涉及的模块、期望达成的效果}

### 技术栈
- 后端: Java 17, Spring Boot 2.7.x, MyBatis-Plus 3.5.x
- 数据库: MySQL 8.0
- 测试: JUnit 5, Mockito
- 构建工具: Maven

### 工作原则（必须严格遵守）

1. 按步骤执行：严格按照下方"执行步骤"逐项完成，不允许跳过或合并步骤。
2. 每步检查：每完成一步，输出"✅ 步骤 N 完成：{简述做了什么}"，然后再进入下一步。
3. 验证先行：写代码之前先确认需要创建或修改的文件存在，不存在则先创建。
4. 安全红线：
   - 不修改数据库连接配置、安全配置、环境变量
   - 不删除已有代码（除非明确要求重写）
   - 不修改依赖版本（pom.xml / build.gradle 中的版本号）
   - 不在代码中写入任何硬编码的密钥、密码、Token
5. 中断确认：遇到以下情况停止执行并向我确认：
   - 不确定某个设计决策时
   - 发现已有代码有 Bug 但不在本次任务范围内时
   - 需要修改超过 5 个文件时
6. 代码质量：
   - 所有新增公开方法必须有 Javadoc 注释
   - 所有新增接口入参使用 @Valid 校验
   - 所有数据库操作使用 MyBatis-Plus，不写纯 JDBC
   - 异常不吞，该抛就抛，该包装就包装

### 执行步骤

步骤 1: {描述第一步做什么}
  - 涉及文件: {路径}
  - 输入: {需要什么信息}
  - 产出: {应该产出的文件或改动}
  - 检查: {如何验证这步正确}

步骤 2: {描述第二步做什么}
  - 涉及文件: {路径}
  - 输入: {依赖第一步的产出}
  - 产出: {应该产出的文件或改动}
  - 检查: {如何验证这步正确}

步骤 3: 验证与测试
  - 运行编译命令: mvn compile -q
  - 运行单元测试: mvn test -pl {模块名} -Dtest={测试类名}
  - 如果编译或测试失败，定位问题并修复
  - 确认所有测试通过后输出: ✅ 全部步骤完成

### 完成标准
- [ ] 所有步骤按顺序完成
- [ ] 代码编译通过（mvn compile -q 无错误）
- [ ] 单元测试通过
- [ ] 无新增编译警告
- [ ] 无硬编码密钥或密码

### Git 提交规范
完成任务后，建议使用以下提交信息格式：
```
feat: {简短描述}

- {改动项1}
- {改动项2}

Co-Authored-By: Claude <noreply@anthropic.com>
```
```

**期望输出**: AI 按步骤执行，每步输出完成状态，所有步骤完成后汇报编译和测试结果。

**使用注意事项**:

- 这是本章唯一一个有大量占位符的模板——因为长任务的具体内容每个都不一样。但工作原则和安全红线部分是固定的，不要删除。
- "中断确认"机制是这个模板的核心价值：无限制地让 AI 自动修改代码极易造成灾难，设置合理的边界让 AI 在不确定时停下来问。
- 执行步骤的数量建议控制在 3-7 步。少于 3 步不需要这个模板；多于 7 步建议拆分为多次执行。
- 如果任务涉及前端代码，在技术栈中补充前端框架信息，并在执行步骤中增加对应的检查和验证方法。

## 4.7 模板组合使用指南

### 完整开发流程模板链

以下是一个功能从需求到上线的完整模板使用顺序：

```
模板1 (需求澄清) → 模板2 (功能清单) → 模板3 (技术方案) → 模板4 (方案评审)
    ↓
模板5 (API 设计) + 模板6 (数据库设计)
    ↓
模板7 (Controller) + 模板8 (Service) + 模板9 (DAO)
    ↓
模板10 (单元测试) + 模板11 (接口测试)
    ↓
模板12 (代码 Review) + 模板16 (安全检查) + 模板17 (性能检查)
    ↓
模板18 (周报) + 模板19 (会议纪要)
```

### 不同场景的模板组合

| 场景 | 推荐模板组合 |
|------|-------------|
| 新功能从零开发 | 1 → 2 → 3 → 5 → 6 → 7 → 8 → 9 → 10 → 12 |
| 线上 Bug 修复 | 13 → 14 → 修复代码 → 10 → 12 |
| 重构老代码 | 12 → 16 → 17 → 重构 → 10 → 12 |
| 紧急安全响应 | 14 → 16 → 修复 → 10 → 12 |
| 每周团队工作汇报 | 18 |
| 技术方案设计 | 3 → 4 → 5 → 6 |

### 模板定制建议

1. **安全约束是底线**：代码生成类模板（7/8/9/10）中的安全条款根据团队规范调整，但不要删除——删掉的后果比你想象的大。
2. **响应体结构要统一**：模板5中的 `{code, message, data}` 结构替换为项目实际使用的统一响应类。
3. **技术栈参数化**：把模板中"Spring Boot 2.7 / MyBatis-Plus 3.5"替换为项目的实际版本，存在项目根目录的 `.claude/prompts/` 目录下。
4. **渐进式优化**：先用原版模板跑一遍，看看产出质量，再针对性调整提示词。不要一上来就大改——原版是在 Spring Boot 2.7 + MyBatis-Plus 体系下反复验证过的。

## 4.8 本章小结

20 个模板覆盖了企业 IT 开发的完整流程。关键原则重申：

1. **模板是起点，不是终点**。每个团队的技术栈、编码规范、业务领域不同，模板需要持续迭代。
2. **安全约束不容妥协**。凡是涉及代码生成的模板，都内置了安全条款。在定制模板时保留这些约束。
3. **AI 输出需要人类把关**。模板提升的是效率和一致性，不是准确率。生成代码后跑测试、生成方案后做评审，这些人类把关步骤不能省。
4. **建立团队提示词库**。将团队验证过的提示词存入项目 `.claude/prompts/` 目录，新成员入职就能用，降低上手成本。
