# Review 标准模板

> 使用说明：本文件包含6类 Review 模板。使用时复制对应模板到新文档，按检查项逐项 Review 并打勾。每项检查结果用 Y(通过) / N(不通过) / N/A(不适用) 标记，N 项必须在"问题描述"列说明具体问题和修复建议。

---

## 1. 需求 Review 模板

### 用途

在进入技术方案设计前，对需求文档的完整性、清晰度、可行性、风险进行审查。

### 基本信息

| 字段 | 内容 |
|------|------|
| 需求文档名称 | <填写> |
| 需求文档版本 | <填写> |
| Review 日期 | <填写> |
| 参与 Review 人员 | <填写> |
| 需求来源 | <业务方/产品经理/客户> |

### 审查维度一：完整性

| 编号 | 检查项 | 结果(Y/N/N/A) | 问题描述 | 修复建议 |
|------|--------|---------------|----------|----------|
| RQ-1.1 | 业务背景描述是否清晰 | | | |
| RQ-1.2 | 核心目标是否可量化 | | | |
| RQ-1.3 | 功能范围边界是否明确（做什么+不做什么） | | | |
| RQ-1.4 | 用户角色是否完整列出 | | | |
| RQ-1.5 | 权限要求是否明确 | | | |
| RQ-1.6 | 业务流程是否完整（正常流程+异常流程+边界条件） | | | |
| RQ-1.7 | 外部系统依赖是否明确列出 | | | |
| RQ-1.8 | 非功能需求是否明确（性能/安全/可用性） | | | |
| RQ-1.9 | 验收标准是否定义 | | | |

## 3. 代码 Review 模板

### 用途

对代码变更进行审查，重点关注正确性、安全性、性能、可维护性、代码规范。Java 项目专用，含25+检查项。

### 基本信息

| 字段 | 内容 |
|------|------|
| 项目/模块 | <填写> |
| 分支/PR | <填写> |
| Review 日期 | <填写> |
| Review 人员 | <填写> |
| 变更文件数量 | <N> |
| 代码行数变更 | +<N> -<M> |

### 审查维度一：正确性

| 编号 | 检查项 | 结果(Y/N/N/A) | 问题描述 | 修复建议 |
|------|--------|---------------|----------|----------|
| CR-1.1 | 业务逻辑是否与 Spec/需求文档一致 | | | |
| CR-1.2 | 边界条件是否处理（null/空集合/0值/最大最小值） | | | |
| CR-1.3 | 异常场景是否处理（网络超时/数据库异常/第三方服务异常） | | | |
| CR-1.4 | 状态流转逻辑是否正确（是否只允许合法路径） | | | |
| CR-1.5 | 数值计算是否正确（浮点精度/溢出/除零） | | | |
| CR-1.6 | 日期时间处理是否正确（时区/夏令时/跨天） | | | |
| CR-1.7 | 事务边界是否正确（回滚范围/传播行为/嵌套事务） | | | |

### 审查维度二：安全性

| 编号 | 检查项 | 结果(Y/N/N/A) | 问题描述 | 修复建议 |
|------|--------|---------------|----------|----------|
| CR-2.1 | 是否有 SQL 注入风险（${} 拼接/动态排序字段） | | | |
| CR-2.2 | 输入参数是否校验（@Valid/白名单/长度限制） | | | |
| CR-2.3 | 是否有越权风险（水平越权/垂直越权） | | | |
| CR-2.4 | 敏感数据是否脱敏（日志中/响应体中/异常信息中） | | | |
| CR-2.5 | 是否有 SSRF 风险（根据用户输入发起 HTTP 请求） | | | |
| CR-2.6 | 密码/Token/密钥是否硬编码在代码中 | | | |
| CR-2.7 | 文件操作是否安全（路径遍历/任意文件读取） | | | |

### 审查维度三：性能

### Review 结论

## 5. 上线前 Review 模板

| 字段 | 内容 |
|------|------|

## 附录：Review 操作流程

任何 Review 都遵循以下通用流程：

```
1. Review 准备
   - 确认 Review 对象（文档/代码/测试/上线方案）
   - 确定参与 Review 人员
   - 选定对应的 Review 模板

2. 逐项检查
   - 按模板中的检查项逐项审查
   - 用 Y/N/N/A 标记每项结果
   - N 项必须填写"问题描述"和"修复建议"

3. 问题汇总
   - 汇总所有 N 项
   - 分级：阻塞问题（必须修） / 一般问题（建议修） / 轻微问题（可选修）
   - 严重安全问题自动升级为阻塞问题

4. 结论判定
   - 无阻塞问题 → 通过
   - 有阻塞问题但可边修边继续 → 有条件通过
   - 有阻塞问题且无法继续 → 不通过，修完后重新 Review

5. 跟踪闭环
   - 所有问题录入跟踪系统（JIRA/飞书/钉钉）
   - 修复后 Review 人确认关闭
   - 不通过的 Review 修完后重新走完整流程
```
