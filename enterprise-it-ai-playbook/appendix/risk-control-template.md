# 风险控制模板

> 使用说明：本文件覆盖企业在 AI 辅助开发过程中常见的 9 大类风险。每个风险类别以"风险登记表"形式呈现，每项风险包含：风险描述、可能性、影响等级、预防措施、检测方法和应急响应方案。
>
> 使用方法：将本模板作为项目风险清单的起点，根据项目实际情况增删风险项，调整可能性和影响等级。

---

## 风险等级定义

| 等级 | 可能性 | 影响 | 处理策略 |
|------|--------|------|----------|
| 极高 | > 70% | 核心业务中断、数据安全事故、合规处罚 | 必须消除才能继续推进 |
| 高 | 40%-70% | 重要功能异常、安全漏洞、项目延期超 2 周 | 必须有预防 + 检测 + 应急三重保障 |
| 中 | 10%-40% | 非核心功能异常、性能下降、延期 1-2 周 | 必须有预防和检测措施 |
| 低 | < 10% | 轻微影响、可快速修复 | 记录在案，定期回顾 |

| DSK-004 | AI 将内部代码、文档、数据用于模型训练导致知识产权泄露 | 中 | 高 | 高 | (1) 使用企业版 AI 服务并签署不用于训练的数据处理协议 (2) API 调用时开启"不用于训练"开关 (3) 核心代码与算法使用本地模型处理 | (1) 审查 AI 服务商隐私条款与数据处理协议 (2) 法务定期审核 AI 服务使用合规性 | (1) 立即停止使用存在隐私风险的 AI 服务 (2) 评估已发送数据的敏感性 (3) 联系 AI 服务商确认数据删除 |

| DSK-006 | AI 生成代码时为了演示或测试，使用了弱密码、默认密码或测试密钥 | 中 | 极高 | 高 | (1) 在提示词中声明"不得使用默认密码、测试密码" (2) 默认密码/测试密钥纳入密钥扫描规则 (3) 所有密码必须从外部配置注入 | (1) 密钥扫描工具中加入常见弱密码规则 (2) 配置审计：检查 application.yml 中的默认值 (3) Code Review 中关注密码相关代码 | (1) 立即替换所有弱密码 (2) 排查使用弱密码的系统是否已暴露 (3) 加强密码策略配置 |

| CQR-002 | AI 生成代码与项目现有风格不一致，降低可维护性 | 高 | 中 | 高 | (1) 在 CLAUDE.md 中明确定义编码规范 (2) 使用代码格式化工具统一风格（Spotless、Google Java Format） (3) 提示词要求"遵循项目现有代码风格" | (1) CI 中集成代码风格检查（Checkstyle、SonarQube） (2) Code Review 关注风格一致性 (3) 使用 AI Review 输出的一致性检查清单 | (1) 运行代码格式化工具自动修复风格 (2) 结构性不一致（分层方式等）进行人工重构 |
| CQR-003 | AI 生成代码存在性能隐患：N+1 查询、全表扫描、大事务、未用连接池 | 中 | 高 | 高 | (1) 提示词要求 AI 考虑数据库查询效率和事务范围 (2) 数据访问层要求使用 MyBatis-Plus 分页等标准方案 (3) 禁止循环中执行数据库查询 | (1) Code Review 检查数据库访问模式 (2) 性能测试监控慢 SQL (3) 数据库慢查询日志分析 | (1) 优化性能问题代码（加索引、减少查询、拆分事务） (2) 性能测试验证效果 (3) 影响线上时紧急回滚后修复 |
| CQR-004 | AI 引入不必要的复杂依赖或过度设计 | 中 | 中 | 中 | (1) 在 CLAUDE.md 列出已有依赖，要求优先使用 (2) 新增依赖须在 Spec 或方案 Review 中审批 (3) 提示词声明"简单优先，不过度设计" | (1) Code Review 检查新增依赖合理性 (2) 依赖分析工具检查替代方案 (3) 架构 Review 评估设计复杂度 | (1) 评估新依赖风险（许可证、维护状态、漏洞） (2) 有更简单方案时重构替换 (3) 更新 CLAUDE.md 依赖清单 |
| CQR-005 | AI 使用废弃 API 或不推荐的过时做法 | 中 | 中 | 中 | (1) CLAUDE.md 注明框架版本与推荐做法 (2) 提示词注明"使用项目当前版本的标准做法" | (1) IDE 废弃 API 警告 (2) CI 静态分析工具检测 (3) Code Review 关注 | (1) 替换为当前推荐 API (2) 检查其他文件是否有相同问题 |
| CQR-006 | AI 生成代码缺少异常处理，异常被静默吞掉或处理方式不当 | 高 | 中 | 高 | (1) 提示词要求"每个外部调用、数据库操作必须有异常处理" (2) 使用全局异常处理器兜底 (3) 禁止空 catch 块 | (1) 静态代码分析检查空 catch 块 (2) Code Review 检查异常处理完整性 (3) 代码覆盖率工具检查异常分支 | (1) 补充缺失的异常处理 (2) 审计日志中查找被吞掉的异常 (3) 加强提示词中的异常处理要求 |
| CQR-007 | AI 生成代码存在资源泄露（数据库连接、文件流、线程池未关闭） | 中 | 高 | 高 | (1) 提示词要求"所有资源使用 try-with-resources" (2) 编码规范明确资源管理要求 (3) 使用连接池，禁止手动管理连接 | (1) 静态分析工具检测资源泄露（PMD、SpotBugs） (2) 长时间运行后监控连接数和文件句柄数 (3) Code Review 关注资源关闭 | (1) 添加 try-with-resources 或 finally 关闭逻辑 (2) 排查生产环境资源泄露程度 (3) 严重时重启服务释 |

## 3. 业务逻辑风险

> 覆盖：业务规则错误、边界遗漏、状态异常

| 编号 | 风险描述 | 可能性 | 影响 | 风险等级 | 预防措施 | 检测方法 | 应急响应 |
|------|----------|--------|------|----------|----------|----------|----------|
| BZR-001 | AI 错误理解或遗漏业务规则，导致代码逻辑与实际需求不符 | 中 | 极高 | 高 | (1) 使用 Spec-Driven Development，所有业务规则写入 Spec (2) Spec 中状态流转、异常处理、权限规则必须完整 (3) 关键业务规则在 Code Review 中逐条核对 | (1) 单元测试覆盖所有业务规则路径 (2) 业务验收测试由产品经理或业务方执行 (3) 使用 Spec 的验收标准逐条验证 | (1) 发现后立即修复代码 (2) 检查是否有因错误逻辑产生的脏数据 (3) 评估是否需要数据修复脚本 |
| BZR-002 | AI 遗漏了业务规则中的边界情况（如已取消订单不允许再次取消、已发货订单不允许修改地址） | 高 | 高 | 高 | (1) 在 Spec 异常处理规则中列出所有边界条件 (2) 提示词要求 AI"检查所有状态下的所有操作是否合法" (3) 测试用例覆盖所有非法操作路径 | (1) 单元测试覆盖每个状态的非法操作 (2) 集成测试覆盖边界场景 (3) 上线前业务方验收 | (1) 修复代码添加边界校验 (2) 补充测试用例 (3) 检查已有数据是否被边界问题影响 |
| BZR-003 | AI 对状态流转执行不完整：漏掉后置动作（如取消订单后未释放优惠券、发货后未发短信通知） | 中 | 高 | 高 | (1) Spec 的状态流转矩阵明确列出前置条件和后置动作 (2) 状态变更封装为独立的有事务保障的方法 (3) Code Review 关注状态变更完整性 | (1) 集成测试验证后置动作是否执行 (2) 数据库检查：状态变更后关联数据是否正确 (3) 日志审计中发现异常 | (1) 修复代码补充后置动作 (2) 对历史数据执行补偿脚本（如补发优惠券） (3) 完善 Spec 后置动作描述 |
| BZR-004 | AI 使用错误的数值计算逻辑（金额 float/double 精度问题、税率计算错误） | 中 | 极高 | 高 | (1) Spec 和编码规范明确：金额使用 BigDecimal，单位用分 (2) 提供统一的金额计算工具类 (3) 禁止使用 float/double 处理金额 | (1) 代码静态检查：禁止 float/double 参与金额计算 (2) 单元测试使用精确值验证 (3) 对账系统定期检查金额一致性 | (1) 立即修复计算逻辑 (2) 排查所有受影响的金额记录 (3) 执行数据修复脚本 (4) 通知财务或业务方 |
| BZR-005 | AI 实现了与现有业务流程冲突的逻辑（如自动取消订单与人工审批流程冲突） | 中 | 高 | 中 | (1) 在 Spec 中标注与现有系统的交互点和冲突规则 (2) 跨模块的 Code Review (3) 流程图级别的业务验证 | (1) 集成测试中端到端验证 (2) 业务流程演练 (3) 与已有功能的对账 | (1) 暂停新功能，确定正确的规则优先级 (2) 修复冲突逻辑 (3) 补充跨模块的协作规范 |

## 4. 权限风险

> 覆盖：越权访问、认证绕过、权限提升

|------|----------|--------|------|----------|----------|----------|----------|
| PMR-001 | AI 生成接口时忘记添加权限注解，导致未授权即可访问 | 高 | 极高 | 极高 | (1) 提示词中强调"每个接口必须有权限注解" (2) Spec 权限矩阵列出每个接口所需的角色 (3) Controller 基类强制权限检查 | (1) 自动化扫描：检查所有 @RestController 方法是否有 @PreAuthorize (2) 安全测试：不同角色 Token 调用每个接口 (3) 上线前安全 Review | (1) 紧急添加权限注解 (2) 审计日志查是否已被未授权访问 (3) 已有未授权访问记录时升级为安全事件 |

| PMR-003 | AI 实现权限判断时使用了错误的角色枚举或逻辑运算符（AND/OR 混淆） | 中 | 高 | 高 | (1) Spec 权限矩阵中明确每个接口的角色列表 (2) 使用统一的权限校验框架而非手写判断 (3) 权限枚举统一管理 | (1) 权限自动化测试：每个角色 X 接口组合 (2) Code Review 重点检查权限判断代码 | (1) 修复角色判断逻辑 (2) 回退错误授权导致的操作 (3) 完善权限测试用例 |
| PMR-004 | JWT Token 过期、无效或被篡改时的处理逻辑缺失或错误 | 低 | 中 | 中 | (1) 统一认证过滤器处理 Token 校验 (2) 401 响应由 Spring Security 统一处理 (3) 认证失败返回清晰但不泄露信息的错误提示 | (1) 安全测试：过期 Token、无效 Token、无 Token 调用 (2) 监控 401 响应率异常 | (1) 确认统一认证过滤器配置正确 (2) 测试所有接口的 401 响应行为 |
| PMR-005 | AI 实现认证绕过漏洞：通过参数注入、Header 伪造绕过了认证 | 低 | 极高 | 高 | (1) 认证逻辑统一在 Filter 层，不在 Controller 层判断 (2) 禁止从请求参数中取用户身份（只能从 Token 解析） (3) 安全编码规范写入 CLAUDE.md | (1) 安全渗透测试 (2) 代码审计：检查是否有从请求参数取 userId 的逻辑 (3) OWASP ZAP 自动化扫描 | (1) 立即修复认证绕过漏洞 (2) 全量安全审计受影响模块 (3) 排查通过该漏洞的异常访问记录 |
| PMR-006 | AI 生成的管理后台接口缺少权限校验或校验过于宽松 | 中 | 高 | 高 | (1) 管理后台接口默认需 ADMIN 角色 (2) 提示词中强调"管理接口权限不可遗忘" (3) 管理接口与前台接口分开 Controller 管理 | (1) 安全扫描：管理接口路径集合逐条权限验证 (2) Code Review 重点关注 /admin/ 路径 (3) 定期渗透测试 | (1) 紧急封堵缺失的权限控制 (2) 排查已有未授权管理操作 (3) 管理接口单独加强监控与告警 |

## 风险登记表使用说明

### 初始化步骤

1. **筛选**：以本模板为起点，删除与项目无关的风险项，补充项目特有的风险项
2. **评估**：对保留的每项风险，根据项目实际情况重新评估"可能性"和"影响"等级
3. **确认措施**：确认每项风险的"预防措施"是否已落地，"检测方法"是否已建立
4. **分配责任人**：每项中风险和高风险的应急响应指定具体责任人
5. **定期回顾**：每双周或每个迭代回顾一次风险登记表，更新风险状态

### 风险状态标记

| 状态 | 说明 |
|------|------|
| 活跃 | 风险存在且未触发，预防和检测措施均已到位 |
| 已触发 | 风险已变为实际问题，正在执行应急响应 |
| 已消除 | 风险已不复存在（如已修复相关代码，或已下线相关功能） |
| 已接受 | 经评估决定接受该风险（须记录接受理由和决策人） |

### 风险升级机制

| 触发条件 | 升级动作 |
|----------|----------|
| 同一风险在 1 个迭代内触发 2 次以上 | 升级至高一个等级，重新评估预防措施有效性 |
| 极高等级风险 | 必须通报技术负责人和项目经理，制定专项消除计划 |
| 应急响应超过 4 小时未能解决 | 升级至部门级应急响应 |

### 日常风险控制节奏

| 频率 | 活动 | 负责人 |
|------|------|--------|
| 每日 | 代码提交前验收检查 | 开发者 |

### 快速风险检查卡

在每个 AI 任务开始前，用这张卡片快速评估：

- [ ] 输入是否涉及敏感数据？需要先脱敏
- [ ] 输出是否直接影响生产系统？需要人工复核
- [ ] 输出是否需要人工审批？审批人已确认
- [ ] 使用的 AI 工具是否在白名单内？非白名单工具不使用
- [ ] 任务是否在 AI 适合的范围内？超出范围不提给 AI
- [ ] CLAUDE.md 上下文是否最新？过期则先更新
- [ ] 验收标准是否明确？不明确则先定义

### 模板定制建议

根据项目类型调整重点关注的风险章节：

| 项目类型 | 重点关注的章节 | 说明 |
|----------|--------------|------|
| 金融、支付类 | 1-数据安全 + 3-业务逻辑 + 5-合规 + 9-责任边界 | 资金安全和监管合规是生命线 |
| 医疗健康类 | 1-数据安全 + 5-合规 | 患者隐私保护法规严格 |
| 政务、公共类 | 5-合规 + 9-责任边界 | 审计追溯要求高 |
| SaaS 平台类 | 1-数据安全 + 2-代码质量 + 4-权限 | 多租户数据隔离是关键 |
| 内部工具类 | 2-代码质量 + 7-AI幻觉 + 8-上下文污染 | 安全要求相对较低 |
| 原型、实验类 | 7-AI幻觉 + 8-上下文污染 | 可接受更多风险，流程可适当放宽 |
