# 附录: 验收清单库

> 本附录汇总了企业 IT AI 落地全流程中各环节的验收清单。每一份清单均可直接打印使用，所有检查项均无 TODO、无空行，确保落地即用。

## 使用说明

- **分类标记**：★ 必须通过（不通过则打回）；☆ 建议检查（不通过需说明原因）；⚠ 高风险项（必须人工确认）；👤 人工判断（无法自动化）
- **时机**：每份清单头部标注了推荐使用场景，在对应里程碑节点逐项勾检
- **责任人**：建议每条检查项明确到人（复核人不同于执行人），避免自查自过

## 一、文档验收

### 1.1 需求文档验收清单

**使用场景**: PRD/用户故事/需求规格说明书完成后，进入技术方案设计之前

| # | 检查项 | 分类 | 检查方法 | 通过标准 |
|---|--------|------|----------|----------|
| 1 | 需求背景是否完整描述了业务痛点和目标 | ★ | 文档审查 | 背景段包含当前问题、目标指标、预期收益三个要素 |
| 2 | 功能范围是否明确界定（含 / 不含） | ★ | 逐条核对 | 每个功能点有明确的边界说明，不存在模糊表述 |
| 3 | 非功能需求是否覆盖了性能、安全、可用性 | ★ | 逐项检查 | 至少包含响应时间、并发量、数据量级、安全等级四项 |
| 4 | 验收标准是否可量化、可测试 | ★ | 逐条审查 | 每条验收标准包含具体数值或可验证的真/假条件 |
| 5 | 异常场景是否完整覆盖（输入边界、超时、并发冲突） | ★ | 异常场景矩阵检查 | 每个主要功能至少覆盖 3 种异常路径 |
| 6 | 角色和权限矩阵是否明确列出 | ★ | 矩阵审查 | 每个功能点标注了所需权限，权限定义无歧义 |
| 7 | 数据流图或时序图是否覆盖核心链路 | ☆ | 图表审查 | 核心业务流程有至少一张图解 |
| 8 | 外部系统依赖是否明确列出并标注接口人 | ★ | 依赖清单审查 | 每个外部依赖有系统名称、接口协议、SLA、联系人 |
| 9 | 需求优先级是否清晰排定（P0/P1/P2） | ★ | 优先级审查 | 每个需求点标注优先级，P0 数量不超过总量的 30% |
| 10 | 需求变更历史是否记录了每次变更的原因和日期 | ☆ | 变更日志审查 | 变更记录包含日期、变更人、变更内容、影响评估 |
| 11 | 术语表是否存在且统一（不出现同一概念多个叫法） | ★ | 全文搜索 | 同一概念在全文中使用同一术语，无别名混用 |
| 12 | 用户故事是否包含"作为...我想...以便..."标准格式 | ☆ | 格式审查 | 每个功能需求有对应的用户故事，且验收条件写在故事下方 |
| 13 | 是否所有 open question 都已关闭或明确了决策期限 | ★ | 问题跟踪表审查 | 文档中无未解决的开放式问题标记 |
| 14 | 干系人是否已全员签字确认（或电子审批通过） | ★ | 审批记录审查 | 产品、开发、测试三方负责人均已确认 |
| 15 | 需求是否可拆分为独立的开发任务（颗粒度 ≤ 3 人天） | ☆ | 粒度估算 | 每条需求可拆分为 ≤ 3 人天的开发任务，无巨型需求 |
| 16 | 是否存在与已有功能的冲突分析 | ☆ | 影响面分析审查 | 对于改版需求，包含对现有功能的影响评估 |
| 17 | UI 交互原型是否覆盖了所有页面状态的切换 | ☆ | 原型走查 | 每个页面覆盖正常态、空态、加载态、错误态、无权限态 |

### 1.2 技术方案验收清单

**使用场景**: 技术方案设计文档完成，进入编码之前

| # | 检查项 | 分类 | 检查方法 | 通过标准 |
|---|--------|------|----------|----------|
| 1 | 架构图是否清晰表达系统分层和模块边界 | ★ | 图表审查 | 架构图包含分层关系、模块边界、数据流向三个要素 |
| 2 | 技术选型是否有明确的理由和备选对比 | ★ | 选型理由审查 | 每个关键组件有选型理由，涉及新引入组件时有 2 种以上对比 |
| 3 | 数据库设计是否包含 ER 图和核心表结构定义 | ★ | ER 图和 DDL 审查 | ER 图覆盖核心实体关系，表定义包含字段类型、索引、默认值 |
| 4 | 接口设计是否定义了完整的请求/响应结构（含错误码） | ★ | 接口文档审查 | 每个接口有 URL、METHOD、请求体、响应体、错误码、鉴权方式 |
| 5 | 核心流程的时序图是否覆盖正常路径和关键异常路径 | ★ | 时序图审查 | 核心流程至少覆盖正常流程和 2 条异常流程 |
| 6 | 缓存策略是否明确了缓存键、过期策略、更新机制 | ★ | 缓存设计审查 | 每个缓存场景有 key 设计、TTL、缓存更新策略、缓存穿透/击穿防护 |
| 7 | 消息队列设计是否定义了 topic/queue、消息体、消费幂等策略 | ★ | MQ 设计审查 | 每个消息有 topic 名称、消息体 schema、消费者、重试策略、死信处理 |
| 8 | 安全设计是否覆盖了认证、鉴权、数据加密、审计日志 | ★ | 安全设计审查 | 至少覆盖认证方式、权限模型、传输加密、敏感字段存储加密、操作审计 |
| 9 | 性能预估是否基于数据量和并发量给出了量化的瓶颈分析 | ★ | 性能推算审查 | 给出目标 QPS、数据量级、预估瓶颈点、预案 |
| 10 | 灰度发布和回滚方案是否可执行（非空话） | ★ | 发布方案审查 | 包含灰度策略（比例/白名单）、监控指标、回滚条件、回滚步骤 |
| 11 | 第三方依赖的风险评估是否包含降级方案 | ☆ | 依赖风险审查 | 每个不可控外部依赖有超时设置、熔断阈值、降级兜底逻辑 |
| 12 | 监控和告警设计是否定义了关键指标和告警规则 | ★ | 监控方案审查 | 业务指标、技术指标、告警阈值、通知渠道均已明确 |
| 13 | 数据迁移方案是否覆盖了存量数据兼容和回滚数据恢复 | ★ | 迁移方案审查 | 包含迁移脚本、验证方式、回滚脚本、影响行数预估 |
| 14 | 是否考虑了多环境差异（开发/测试/预发/生产） | ☆ | 环境差异审查 | 配置项、数据源、外部依赖在不同环境下的差异有明确说明 |
| 15 | 是否包含技术债务评估和后续优化计划 | ☆ | 技术债审查 | 标明了已知的取舍（trade-off）和后续优化时间节点 |
| 16 | 代码仓库和分支策略是否明确 | ☆ | 分支策略审查 | 标明仓库地址、主干/开发/发布分支命名规则、合并策略 |

### 1.3 API 文档验收清单

**使用场景**: API 设计或开发完成，联调之前

| # | 检查项 | 分类 | 检查方法 | 通过标准 |
|---|--------|------|----------|----------|
| 1 | 每个接口是否标注了 HTTP 方法和完整的 URL 路径 | ★ | 逐接口审查 | URL 路径无歧义，RESTful 风格一致，不使用动词 |
| 2 | 请求参数是否完整标注了名称、类型、必填/可选、约束条件 | ★ | 参数表审查 | 每个参数标注类型（含格式如 email/uuid）、是否必填、取值范围 |
| 3 | 请求体和响应体是否有 JSON Schema 或完整示例 | ★ | 示例审查 | 每个接口至少提供一个完整请求示例和一个正常响应示例 |
| 4 | 认证和鉴权方式是否在每个接口上明确标注 | ★ | 逐接口检查 | 每个接口标注是否需要认证、所需权限/角色 |
| 5 | 错误码是否完整列举（业务错误码 + HTTP 状态码） | ★ | 错误码表审查 | 每个接口列出了所有可能的业务错误码和对应的 HTTP 状态码 |
| 6 | 分页接口是否统一了分页参数命名和响应结构 | ★ | 一致性审查 | 所有分页接口使用相同的 page/pageSize/total/totalPages 字段 |
| 7 | 时间字段是否统一了格式和时区约定 | ★ | 一致性审查 | 所有时间字段使用 ISO 8601 格式，明确标注时区（UTC 或 Asia/Shanghai） |
| 8 | 枚举字段是否列出了所有合法值 | ★ | 枚举审查 | 枚举类型字段的每个合法值有中文说明 |
| 9 | 废弃字段或接口是否有 @deprecated 标记和替代说明 | ☆ | 废弃标记审查 | 废弃字段注明废弃版本、替代字段、计划移除版本 |
| 10 | 版本管理策略是否明确（URL path / header / query param） | ★ | 版本策略审查 | 文档中明确写了 API 版本策略和升级兼容规则 |
| 11 | 调用频率限制（Rate Limit）是否明确 | ☆ | 限流规则审查 | 标注了每个接口的调用频率上限和超限后的返回格式 |
| 12 | 幂等性要求是否标注（POST/PUT 是否需幂等键） | ★ | 幂等性审查 | 需要幂等的接口标注了幂等键字段和重复调用的返回行为 |
| 13 | 文件上传接口是否标注了大小限制和支持的文件类型 | ★ | 上传接口审查 | 标注了最大文件大小（MB）、MIME 类型白名单、上传超时时间 |
| 14 | 响应中的 ID 字段是否统一了类型（Long 精度/ String/ UUID） | ☆ | 一致性审查 | 全 API 使用统一的 ID 类型，长整型前端安全的使用 String 传输 |
| 15 | 接口文档是否与代码实现保持同步（非手写一次即过期） | ★ | 自动化校验 | 有工具确保文档与代码一致（Swagger/OpenAPI 自动生成或契约测试） |
| 16 | 是否包含调用时序说明（哪些接口需要先调用，依赖关系） | ☆ | 调用流程审查 | 复杂业务流程有接口调用顺序说明 |

## 二、代码验收

### 2.1 Java 代码验收清单（核心）

**使用场景**: 代码提交 Code Review 前自检，或作为团队 CR 标准清单

| # | 检查项 | 分类 | 检查方法 | 通过标准 |
|---|--------|------|----------|----------|
| 1 | 代码是否能无报错编译通过 | ★ | `mvn compile` 或 IDE 构建 | 编译零错误、零警告（deprecation 警告除外但需说明） |
| 2 | 类名、方法名、变量名是否符合团队命名规范 | ★ | 人工审查 + Checkstyle | 类名 UpperCamelCase，方法/变量 lowerCamelCase，常量 UPPER_SNAKE_CASE |
| 3 | 包结构是否符合分层约定（controller/service/dao/model） | ★ | 目录结构审查 | 代码在约定包下，无跨层调用（如 controller 直接调 dao） |
| 4 | 是否存在循环依赖（A→B→A） | ★ | IDE 依赖分析 或 ArchUnit | 模块间和类间无循环依赖 |
| 5 | 接口是否定义了完整的参数校验注解（@NotNull/@NotEmpty/@Valid） | ★ | 逐接口审查 | Controller 层所有入参 DTO 有 JSR-303 校验注解，且有 @Valid 触发 |
| 6 | 业务层是否有重复的参数校验（与 Controller 层重复且不一致） | ☆ | 对比审查 | Service 层不再重复 Controller 已有校验，或使用统一校验器 |
| 7 | 异常处理是否使用了全局异常处理器而非到处 try-catch | ★ | 异常处理审查 | 业务代码不吞咽异常，统一由 @ControllerAdvice 处理 |
| 8 | 自定义异常是否继承合适的基类并携带业务错误码 | ☆ | 异常类审查 | 自定义异常有错误码、有清晰的中英文消息 |
| 9 | 日志是否覆盖了关键分支（入口参数、出口结果、异常栈） | ★ | 日志埋点审查 | 核心方法入口打 INFO 日志含关键参数，异常打 ERROR 含完整堆栈 |
| 10 | 日志中是否包含敏感信息（手机号/身份证/密码/Token） | ⚠ | 日志脱敏审查 + 正则扫描 | 日志中无明文敏感数据，已脱敏或已屏蔽 |
| 11 | SQL 是否使用了参数化查询（PreparedStatement / MyBatis #{}） | ⚠ | SQL 拼装审查 | 无字符串拼接 SQL，MyBatis 中使用 #{} 而非 ${}（除非动态表名/列名且已白名单校验） |
| 12 | MyBatis 动态 SQL 中的 ${} 是否做了白名单校验 | ⚠ | 动态 SQL 审查 | 每个 ${} 使用处有对应的白名单校验代码，且校验在拼接前 |
| 13 | 数据库事务边界是否正确（@Transactional 是否加在 Service 层） | ★ | 事务注解审查 | @Transactional 加在 Service 实现类的方法上，不在 Controller 层 |
| 14 | 事务传播行为和隔离级别是否合理（非默认时必须写注释说明） | ☆ | 事务属性审查 | 使用非 REQUIRED 传播行为时有注释说明原因 |
| 15 | 是否存在大事务（包含 RPC 调用、MQ 发送、文件 IO 在事务内） | ⚠ | 事务范围审查 | 事务内不包含外部 RPC 调用或耗时 IO 操作 |
| 16 | 数据库操作是否有 N+1 查询问题 | ★ | SQL 日志 + 代码走查 | 关联查询使用 JOIN 或批量查询，循环内无单条 SQL |
| 17 | 是否避免了 SELECT * 并明确列出所需字段 | ☆ | SQL 审查 | 查询语句显式指定字段列表 |
| 18 | 分页查询是否有合理的 limit 且无全表扫描风险 | ★ | SQL 执行计划审查 | 分页查询有索引支撑和 limit 限制，禁止无 limit 的全表查询 |
| 19 | 是否存在并发安全问题（共享可变状态未加锁/未用原子类） | ⚠ | 并发代码审查 | 共享可变状态有 synchronized/lock/AtomicXxx/CopyOnWrite 保护 |
| 20 | 线程池使用是否合理（自定义参数，未用 Executors 工厂方法） | ☆ | 线程池审查 | 使用 ThreadPoolExecutor 显式构造，核心/最大线程数有业务依据 |
| 21 | 文件/流/连接是否在 finally 或 try-with-resources 中关闭 | ★ | 资源释放审查 | IO 流、数据库连接、网络连接均有确定性的关闭逻辑 |
| 22 | equals() 和 hashCode() 是否同时重写 | ★ | 实体类审查 | 重写 equals 的类一定重写 hashCode，反之亦然 |
| 23 | 是否存在魔法值（硬编码数字/字符串） | ☆ | 硬编码扫描 | 魔法值为常量替代（业务配置进配置中心/枚举/常量类） |
| 24 | 单元测试是否覆盖了核心业务逻辑 | ★ | 覆盖率报告 | 核心 Service 方法测试覆盖率 ≥ 80%，关键分支有独立用例 |
| 25 | 测试用例是否有断言（非只调方法不校验结果） | ★ | 测试代码审查 | 每个 @Test 方法至少包含一个 assertXxx 语句 |
| 26 | 是否存在明显的性能问题（如循环内字符串 +=、不合理的正则编译） | ☆ | 性能扫描 | 无低效的字符串拼接模式，正则 Pattern 在静态常量中预编译 |
| 27 | 敏感数据是否在接口返回前做了脱敏处理 | ⚠ | 返回体审查 | 手机号、身份证、银行卡号等在序列化时已脱敏 |
| 28 | 权限校验是否在每个接口上生效（非仅在菜单上隐藏） | ⚠ | 权限注解审查 | 每个需要权限的接口有 @PreAuthorize 或等效权限注解 |
| 29 | 跨模块调用是否使用了防腐层（Anti-Corruption Layer） | ☆ | 架构审查 | 调用外部系统时有独立的适配层，外部 DTO 不直接侵入业务层 |
| 30 | 代码中是否有 TODO/FIXME 未关联 JIRA/issue 编号 | ☆ | TODO 扫描 | TODO/FIXME 注释后附 issue 编号和处理计划 |
| 31 | Lombok 使用是否规范（避免 @Data 在 JPA Entity 上，@Builder 与继承混用） | ☆ | Lombok 审查 | JPA Entity 用 @Getter/@Setter 而非 @Data，@Builder 不用于有继承的类 |
| 32 | 金额字段是否使用 BigDecimal 而非 float/double | ⚠ | 金融精度审查 | 所有涉及金额的字段和计算使用 BigDecimal，且用 String 构造器 |
| 33 | 集合返回是否做了空集合而非 null（避免 NPE） | ★ | 返回值审查 | 查询列表无结果时返回 Collections.emptyList() 而非 null |
| 34 | 配置项是否有合理的默认值和校验（@ConfigurationProperties） | ☆ | 配置审查 | 关键配置有默认值，启动时校验必填配置是否存在 |
| 35 | 是否存在明显的 OOM 风险（大对象未限制、缓存无上限） | ⚠ | 内存风险审查 | 本地缓存有大小上限和淘汰策略，大文件/数据采用流式处理 |

### 2.2 数据库 DDL 验收清单

**使用场景**: DDL 脚本提交审核前，或数据库变更上线前

| # | 检查项 | 分类 | 检查方法 | 通过标准 |
|---|--------|------|----------|----------|
| 1 | 每张表是否有主键 | ★ | DDL 审查 | 所有表定义了 PRIMARY KEY |
| 2 | 主键是否使用了自增 ID 或雪花算法（非业务字段） | ★ | 主键策略审查 | 主键无业务含义，不使用身份证号、手机号等作为主键 |
| 3 | 是否有合理的字符集和排序规则（建议 utf8mb4 + utf8mb4_general_ci） | ★ | 建表语句审查 | 表和字符字段指定 CHARSET utf8mb4 |
| 4 | 每个字段是否有 NOT NULL 约束和合理的 DEFAULT 值 | ★ | 字段约束审查 | 必填字段有 NOT NULL，非必填字段有 DEFAULT 值而非允许 NULL |
| 5 | 索引是否覆盖了高频查询条件（WHERE / JOIN / ORDER BY） | ★ | 索引覆盖审查 | 每个核心查询的 WHERE 条件有对应索引 |
| 6 | 是否存在冗余索引（如 (a,b,c) 和 (a,b) 同时存在） | ☆ | 索引冗余审查 | 无前缀重复的复合索引，单表索引数合理（建议 ≤ 5 个） |
| 7 | 是否建了唯一索引防止业务重复数据 | ★ | 业务唯一性审查 | 有业务唯一性要求的字段组合有 UNIQUE 索引 |
| 8 | varchar 字段长度是否有业务依据而非一律 varchar(255) | ☆ | 字段长度审查 | 可变长字段长度与业务规则匹配（如手机号 20，姓名 50，描述 500） |
| 9 | 是否避免了外键约束（生产环境建议应用层保证引用完整性） | ☆ | 外键审查 | 生产表不使用 FOREIGN KEY 约束，关联关系在应用层维护 |
| 10 | text/blob 类型是否单独建表或垂直拆分（避免影响主表查询性能） | ☆ | 大字段拆分审查 | 含大字段的表有对应的拆分策略或明确注释说明不拆的原因 |
| 11 | 是否设置了合理的表注释和字段注释 | ★ | 注释审查 | 每张表和每个字段有 COMMENT，字段注释说明业务含义非纯翻译字段名 |
| 12 | 数据归档和清理策略是否明确（尤其日志表/流水表） | ☆ | 归档策略审查 | 大表（预估超千万行）有明确的分表策略或归档方案 |
| 13 | 是否有默认的 create_time 和 update_time 字段 | ★ | 审计字段审查 | 每张业务表含 create_time（DEFAULT CURRENT_TIMESTAMP）和 update_time（ON UPDATE） |
| 14 | 是否评估了 DDL 对生产环境的影响（锁表、耗时） | ⚠ | 影响面评估 | 大表 DDL 有在线变更方案（pt-online-schema-change/gh-ost），非直接 ALTER |
| 15 | 敏感字段（密码、手机号、身份证）是否有加密或脱敏存储方案 | ⚠ | 敏感数据存储审查 | 密码使用 BCrypt 等单向哈希，身份证等使用可逆加密且有密钥管理方案 |
| 16 | 字段类型选择是否合理（不用 varchar 存数字比较的字段、不用 int 存可能超限的值） | ☆ | 字段类型审查 | 数值比较/计算的字段使用数值类型，金额使用 decimal |

| 14 | 是否有针对性的并发测试用例 | ☆ | 并发测试审查 | 共享状态的代码有并发测试用例验证线程安全性 |

## 四、安全验收

### 4.1 代码安全验收清单（OWASP Top 10 重点）

**使用场景**: 代码安全审计，发布前安全审查

| # | 检查项 | 分类 | 检查方法 | 通过标准 |
|---|--------|------|----------|----------|
| 1 | SQL 注入：所有数据库操作是否使用参数化查询 | ⚠ | 代码静态扫描 + 人工抽查 | 100% 使用 PreparedStatement 或 MyBatis #{}，无字符串拼接 SQL |
| 2 | SQL 注入：动态排序字段/表名是否做了白名单校验 | ⚠ | 动态 SQL 专项审查 | 所有 orderBy/groupBy/tableName 接收前端参数时有白名单校验 |
| 3 | XSS：所有用户输入在输出到页面时是否做了转义 | ⚠ | XSS 防护审查 | 后端输出到前端使用 HtmlUtils.htmlEscape 或模板引擎自动转义 |
| 4 | XSS：富文本输入是否经过白名单标签过滤（如 Jsoup） | ⚠ | 富文本审查 | 富文本内容经过 HTML 清洗，只保留安全标签和白名单属性 |
| 5 | CSRF：状态变更操作（POST/PUT/DELETE）是否有 CSRF Token 校验 | ⚠ | CSRF 防护审查 | 非 GET 请求有 CSRF Token 或 SameSite Cookie 保护 |
| 6 | 权限绕过：每个接口是否在后端做了权限校验（非仅前端隐藏按钮） | ⚠ | 权限校验审查 | 所有接口在后端有权限注解或 AOP 校验，直接调接口无法绕过 |
| 7 | 越权：用户只能访问自己的数据（水平越权防护） | ⚠ | 越权测试 | 更换请求中的用户 ID 后无法访问他人数据，返回 403 或空结果 |
| 8 | 敏感信息泄露：错误信息是否暴露了堆栈、SQL、路径 | ⚠ | 错误信息审查 | 生产环境不返回 stackTrace，自定义错误信息不含内部实现细节 |
| 9 | 敏感信息泄露：API 响应中是否包含密码、密钥、Token | ⚠ | 响应体扫描 | 响应体中无明文密码、无 accessKey/secretKey、无内部 IP |
| 10 | 敏感信息泄露：日志中是否打印了用户敏感数据 | ⚠ | 日志脱敏审查 | 日志中手机号/身份证/密码/Token 已脱敏或未打印 |
| 11 | 不安全的反序列化：是否接受并反序列化了不可信来源的对象 | ⚠ | 反序列化审查 | 不接受外部输入的序列化对象，Fastjson/Jackson 无 enableDefaultTyping |
| 12 | SSRF：服务端发起的 HTTP 请求是否有 URL 白名单或内网 IP 过滤 | ⚠ | SSRF 防护审查 | 用户可控 URL 的请求做了域名/IP 白名单，禁止访问内网地址 |
| 13 | 文件上传：是否有文件类型白名单校验（MIME + 魔数双重校验） | ⚠ | 上传安全审查 | 文件上传校验文件魔数 + MIME 类型 + 扩展名白名单，三者都通过才放行 |
| 14 | 文件上传：上传目录是否禁止脚本执行且文件名随机化 | ⚠ | 上传目录审查 | 上传目录不在 Web 根目录下，配置了禁止脚本执行，文件名使用 UUID 重命名 |
| 15 | 加密：密码是否使用了不可逆哈希算法（BCrypt/Argon2）存储 | ⚠ | 密码存储审查 | 密码使用 BCrypt 或 Argon2，无明文或 MD5/SHA1 存储 |
| 16 | 加密：敏感数据（身份证、银行卡）存储时是否加密 | ⚠ | 数据加密审查 | 敏感数据使用 AES-256-GCM 等算法加密，密钥由 KMS 管理不入代码 |
| 17 | 加密：TLS 版本是否 ≥ 1.2，是否禁用了不安全的加密套件 | ☆ | TLS 配置审查 | 生产环境最低 TLS 1.2，禁用 SSLv3/TLSv1.0/TLSv1.1 |
| 18 | 认证：会话 Token 是否有合理的有效期和刷新机制 | ★ | 认证机制审查 | Access Token 有效期 ≤ 2 小时，Refresh Token ≤ 7 天，有退出失效机制 |

| 20 | 依赖安全：第三方依赖是否有已知高危漏洞（CVE） | ⚠ | 依赖扫描（OWASP Dependency-Check / Snyk） | 无 Critical/High 级别未修复漏洞 |
| 21 | CORS：是否配置了精确的 Access-Control-Allow-Origin（非 *） | ☆ | CORS 配置审查 | CORS 允许的来源为白名单域名，不使用通配符 * |
| 22 | 速率限制：登录、短信验证码等接口是否有频率限制 | ⚠ | 频率限制审查 | 登录接口有 IP 级和账号级频率限制，验证码接口 60 秒内不可重复发送 |

## 清单使用建议

1. **自动化优先**：标注了检查方法的项目，优先通过 Checkstyle / SonarQube / OWASP Dependency-Check / JaCoCo / 契约测试等工具自动检查，人工仅处理 ⚠ 和 👤 项
2. **分层执行**：文档 → 代码 → 测试 → 安全 → AI 输出的顺序执行，前一个环节不通过不进入下一个环节
3. **持续迭代**：每个项目根据实际踩过的坑持续补充清单，清单是活文档而非一次性产物
4. **签名确认**：每个清单完成后，执行人和复核人双签，存档备查
5. **回顾改进**：每次线上事故后对照清单反向检查，如果清单中缺少对应检查项，立即补充
