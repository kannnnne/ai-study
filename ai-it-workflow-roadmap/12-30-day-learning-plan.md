# 第 12 章：30 天学习计划

## 本章要解决的问题

前面 11 章把 AI 时代程序员需要掌握的知识体系拆开讲了一遍。但知道"应该学什么"和"怎么学"是两回事。你大概率遇到过这种情况：收藏了一堆文章、买了几个课程、装了几个新工具，两周后什么都没坚持下来。

本章解决三个核心问题：

1. **优先级问题**：AI 相关技能浩如烟海，先学哪个后学哪个？每个技能投入多少时间合适？
2. **落地问题**：学了之后怎么在工作中用起来？怎么避免"学完就忘"？
3. **反馈问题**：怎么知道自己的学习方向是对的？什么算"学会了"？

本章给出一份按天编排的 30 天学习计划。它不是让你照着课程表打卡——你有一份全职工作，不可能每天抽出 8 小时来学 AI。这份计划是按照**每天 1-2 小时**的投入设计的，核心逻辑是：**先用起来，再补理论；先拿到收益，再深入原理**。

## 学习计划设计原则

在展开具体计划之前，先对齐几条设计原则。这些原则比具体内容更重要——如果你时间冲突需要调整计划，守住原则就不会跑偏。

**原则一：工具先行，原理后补**

传统学习方法论是先学基础理论再动手。在 AI 场景下这条路行不通。LLM 发展太快，等你"学完基础"，工具已经迭代了好几版。正确的顺序是：先用 Claude Code 完成一个任务，感受到它能做什么、不能做什么，再回头理解 Token、Context Window、Temperature 这些概念就很容易——因为你见过它们在实际场景中的表现。

**原则二：每天必须有产出**

"今天我读了某篇文章"不算产出。产出是一段可运行的代码、一份写好的 Spec、一个验证过的结论、一条写进 CLAUDE.md 的规范。每天结束后，你应该有一个具体的东西可以给同事看，或者说："你看，今天我用 AI 把这个接口生成了。"

**原则三：真实场景优先**

不要用 LeetCode 练手。用你的真实工作项目练手。如果你在做一个用户管理模块，就让 AI 帮你生成 CRUD 代码、写单元测试、写接口文档。学习材料和你的工作内容重叠度越高，转化率越高——因为你学完当天就能用，用了就有反馈，有反馈就会加深理解。

**原则四：遇到卡点立刻问 AI**

计划里的每日内容只是一个框架。你在实践过程中遇到的问题——工具怎么配置、命令怎么跑、报错怎么解——直接问 Claude Code 或 ChatGPT。不要憋着，不要"我先自己研究一下"。你是在学怎么跟 AI 协作，不是在学怎么靠自己搞定一切。用 AI 来解决学习过程中的障碍，本身就是训练。

**原则五：第 7、14、21 天必须复盘**

每周最后一天停下来，回顾一周做了什么、哪些技能真的在工作中用上了、哪些只停留在练习层面。让每周的复盘结果驱动下一周的重点。如果某个方向学了一周还没找到实际应用场景，果断降低投入。

#### Day 1：LLM 基础与 Token 机制

**学习内容**

LLM 的工作原理不需要深入到 Transformer 架构、注意力机制那一层。作为使用者，你需要理解三个关键概念：

**Token**：LLM 不是按字符读文本的，是按 Token。一个 Token 大约等于 3/4 个英文单词或 0.5-1 个中文字。代码中，一个关键字、一个变量名、一个括号都可能是 1 个 Token。这意味着：
- API 调用的计费按 Token 算（包括输入和输出）
- 每次对话能塞进去的上下文有限
- 把一段代码扔给 AI 之前，它在 Token 层面就会"吃掉"一部分对话空间

**Context Window（上下文窗口）**：模型一次能"看到"的最大 Token 数。不同模型差异巨大：
- Claude Sonnet：200K tokens（约 15 万英文单词）
- GPT-4o：128K tokens
- 本地小模型：通常 4K-32K tokens

Context Window 是你和 AI 对话的"工作记忆"。超过这个限制，模型会"忘记"最前面说过的话。这在长对话或处理大代码库时是核心瓶颈。

**Temperature（温度）**：控制模型输出的随机性。0 最确定（每次都选概率最高的 Token），1 以上越来越"创造性"。编程场景一般用 0-0.3，创意写作才需要高温度。很多 AI 编程工具默认已经替你设好了合适的值，但理解这个参数有助于你理解为什么同一个 Prompt 可能产生不同结果。

**实践任务**

1. 打开 OpenAI Tokenizer 页面（platform.openai.com/tokenizer），分别输入一段中文、一段英文、一段 Java 代码，观察 Token 的切分方式。记录你发现的规律。

2. 在 Claude Code 中执行以下操作：
   - 用 `/add-dir` 把当前项目目录加入上下文，观察 Token 消耗的变化
   - 尝试让 AI 分析一个超过 2000 行的类文件，看它能否准确理解所有方法

3. 用同一个问题，分别设置 Temperature = 0 和 Temperature = 1，对比输出差异。问题建议："用 Java 实现一个 LRU Cache，包含 get 和 put 方法。"

**产出要求**

- 一份 Token 切分规律的观察笔记（200 字以内即可）
- 一个用 Temperature = 0 和 Temperature = 1 生成的 LRU Cache 代码对比

**验收标准**

- 能用自己的话解释 Token、Context Window、Temperature
- 能估算一段 Java 代码大概消耗多少 Token（误差在 30% 以内即可）

#### Day 2：Prompt Engineering 实战

**学习内容**

Prompt Engineering 不是玄学，是工程方法。核心思想可以归结为一条规则：**你给的信息越精确、越结构化，AI 的输出越可控**。

几个经过验证的有效技巧：

**角色设定**：开头明确告诉 AI 你的身份和它的角色。

```
你是一位资深 Java 后端工程师，代码风格遵循阿里巴巴 Java 开发手册。
我会给你需求描述，你输出可直接运行的代码。
```

**结构化输出要求**：不要让 AI"自由发挥"，明确指定输出格式。

```
输出格式：
1. 核心思路（3 句话以内）
2. 完整代码（只输出 .java 文件内容）
3. 使用说明（包括 Maven/Gradle 依赖）
```

**分步引导（Chain of Thought）**：

```
请按以下步骤完成：
步骤 1：分析输入数据的结构
步骤 2：设计核心算法思路
步骤 3：写出伪代码
步骤 4：生成完整实现
```

**负面示例引导**：告诉 AI 不要做什么，往往比告诉它做什么更有效。

```
要求：
- 不要使用 Lombok
- 不要引入 Spring 依赖（用纯 Java）
- 不要过度抽象，保持在一个类内完成
```

**实践任务**

1. 用三种不同质量的 Prompt 让 AI 生成同一个功能（如"解析 CSV 文件并计算某列的平均值"）：
   - Prompt A：一句话描述（"写一个 Java 程序解析 CSV 文件"）
   - Prompt B：加上角色、输入输出要求
   - Prompt C：加上分步引导、负面示例、结构化输出格式
   
   对比三次生成的代码质量。

2. 找一个你最近写过的、比较复杂的业务方法（50 行以上），把它扔给 AI，用 Prompt 引导它：
   - 先逐行解释逻辑
   - 然后指出潜在 bug
   - 最后给出重构建议

3. 尝试"Few-shot"技巧：给 AI 先看一个你满意的代码示例，再让它按同样风格生成新代码。测试它能否模仿你的编码风格。

**产出要求**

- 三份 Prompt 及其对应生成代码的对比笔记（从格式合格、逻辑正确、风格一致三个维度打分）
- 一个你觉得"写得好"的 Prompt 模板，保存到你的笔记中，后续复用

**验收标准**

- 你写的 Prompt 能稳定让 AI 第一次就给出可用的代码（不需要反复调整）
- 知道什么时候用角色设定、什么时候用分步引导、什么时候用负面示例

**推荐 Prompt 收藏**

以下是一批可以直接复用的 Prompt，调整需求描述后使用：

```text
你是一位 10 年经验的 Java 后端工程师，熟悉 Spring Boot、MyBatis、MySQL。
你的代码遵循以下规范：
- 阿里巴巴 Java 开发手册
- 方法不超过 50 行
- 圈复杂度不超过 10
- 所有 public 方法必须有 Javadoc
- 不使用 Lombok

任务：{在此描述你的需求}

输出格式：
1. 设计思路（3 句话）
2. 完整代码（包含必要的 import 语句）
3. 需要添加的 Maven 依赖（如有）
```

#### Day 3：Agent 概念与工具调用

**学习内容**

昨天你给 AI 一个 Prompt，它返回一段代码。今天要理解的是：AI 可以做更多——它可以**主动调用工具、执行命令、读取文件、搜索网络**。

**Agent 的本质**：Agent = LLM + 工具集 + 执行循环。LLM 决定"下一步该调用什么工具"；工具执行具体操作（读文件、跑命令、发 HTTP 请求）；结果反馈给 LLM，它再决定下一步。这个闭环让 AI 从"一次性回答"变成"多步执行"。

**关键概念**：

- **Function Calling / Tool Use**：模型输出不是自然语言，而是一个结构化的函数调用请求。你的应用代码执行这个函数，把结果返回给模型。
- **ReAct 模式**（Reasoning + Acting）：模型交替进行"思考"和"行动"。思考时分析现状，行动时调用工具获取新信息。
- **MCP（Model Context Protocol）**：Anthropic 提出的开放协议，让 AI 应用通过统一接口连接不同的工具和数据源。你可以理解为"AI 世界的 USB 协议"。

**实践任务**

1. 在 Claude Code 中观察 Agent 行为：执行一个多步骤任务（如"在当前项目中找到所有用到 @Transactional 注解的方法，检查它们的事务传播级别是否合理"），注意看 Claude Code 的执行轨迹——它读了哪些文件、搜了什么、跑了什么命令。

2. 用 Claude Code 的 MCP 功能连接一个外部工具。如果还没配置过，先配一个最简单的——GitHub MCP Server，让 Claude Code 能直接查询你的 GitHub 仓库。

3. 阅读一篇 Agent 相关的技术文章。推荐 Anthropic 的"Building Effective Agents"（搜索中文翻译版如果原文无法访问）。

**产出要求**

- 一份对上述多步骤任务的执行轨迹复盘：AI 做了哪些步骤？每一步的决策合理吗？有没有绕远路？
- 一个已配置且可用的 MCP 工具

**验收标准**

- 能解释 Agent 与传统 Chatbot 的区别
- 能说出至少 3 个 Claude Code 常用的工具（Bash、Read、Edit、Glob 等）
- MCP Server 配置成功且能正常调用

#### Day 4：Claude Code / Codex 深度使用

**学习内容**

前三天你建立了理论基础。今天进入实战——把 AI 编码工具深度嵌入你的日常工作流。

**Claude Code 核心功能清单**：

| 功能 | 用途 | 典型场景 |
|------|------|----------|
| `/add-dir` | 把目录加入上下文 | 让 AI 理解项目结构 |
| `@文件名` | 引用特定文件 | 让 AI 分析或修改特定文件 |
| Plan Mode | 先出方案再动手 | 复杂需求先确认思路 |
| Agent 模式 | 多步骤自动执行 | 跨文件重构、批量修改 |
| MCP 集成 | 连接外部工具 | 查数据库、搜文档、调 API |
| Custom Slash Commands | 自定义快捷指令 | 项目特定的重复操作 |

**Codex CLI**（OpenAI 的编程 Agent 工具）功能对标：
- `codex exec`：一句话执行多步骤任务
- `codex review`：自动 Code Review
- Sandbox 环境：隔离执行，避免影响本地环境

**实践任务**

1. **用 Claude Code 完成一个真实开发任务**。建议选一个你当前工作中确实需要做的小功能：
   - 启动 Claude Code，用 Plan Mode 描述需求
   - 确认方案后，让它在 Agent 模式下生成代码
   - 人工 Review 生成结果
   - 运行测试验证

2. **对比 Claude Code 和 Codex**（如果你两个都能访问）：
   - 同一个需求分别给两个工具
   - 对比生成质量、执行速度、Token 消耗
   - 记录哪个工具在什么场景下表现更好

3. **尝试一个你以前觉得"AI 肯定做不好"的任务**。很多人的心理预设是错的——先试再说。

**产出要求**

- 一个由 AI 生成的、经过你 Review 并通过测试的功能代码
- 一份 Claude Code vs Codex 的对比体验笔记（如有条件）

**验收标准**

- 能在 5 分钟内用 Claude Code 启动一个任务并得出可用的结果
- 能独立判断 AI 生成的代码质量，做出"直接用 / 改改再用 / 重写"的决策
- 知道 Claude Code 当前版本的已知局限性

#### Day 5：Agentic Coding 实践

**学习内容**

Day 3 你理解了 Agent 的概念，今天要在编码场景中刻意练习 Agentic Coding——让 AI 自主完成一个相对复杂的编程任务，你只负责验收和微调。

**Agentic Coding 和普通 AI 辅助编程的区别**：

| 维度 | 普通 AI 辅助 | Agentic Coding |
|------|-------------|----------------|
| 交互方式 | 你问一句，AI 答一句 | 你给目标，AI 自己拆解执行 |
| 你做什么 | 想好每一步怎么写，让 AI 翻译成代码 | 想好"要什么"，AI 想"怎么做到" |
| 代码范围 | 单个方法、单个类 | 跨文件、跨模块 |
| 验收方式 | 每段代码都看 | 跑测试 + 抽查关键逻辑 |

**Agentic Coding 的正确姿势**：

1. **描述"是什么"而不是"怎么做"**。不是"在 UserService 里加一个 findByName 方法"，而是"我需要根据用户名查询用户信息，如果不存在就抛异常，如果存在就返回脱敏后的信息"。
2. **给验收标准，不给实现细节**。不是"用 MyBatis SelectProvider 注解动态拼接 SQL"，而是"查询 SQL 的 WHERE 条件需要根据参数是否为空动态拼接"。
3. **充分提供上下文**。让 AI 知道自己在一个什么项目里、用什么技术栈、遵循什么规范。这就回到了 CLAUDE.md 和 Context Engineering 的重要性。

**实践任务**

1. 找一个需要涉及 2-3 个文件修改的需求（如"给用户管理模块增加分页查询功能，需要改 Controller、Service、Mapper 三层"），用 Agentic Coding 方式完成。

2. 实验：同一个需求，分别用"精确指导"和"目标驱动"两种方式让 AI 实现。对比：哪个更快完成？哪个代码质量更高？哪个更接近你想要的？

3. 记录 AI 做错了什么——这些"错误模式"是你判断 AI 能力边界的关键依据。

**产出要求**

- 一个通过 Agentic Coding 完成的功能（至少涉及 3 个文件）
- 一份"AI 常见错误模式"记录（至少 3 种）

**验收标准**

- 能用"描述目标"而非"描述步骤"的方式让 AI 完成一个跨文件任务
- 能预判 AI 可能在哪类问题上犯错

#### Day 6：Vibe Coding 体验

**学习内容**

Vibe Coding 是 Andrej Karpathy 提出的概念，核心思想是：**你只负责"感觉"，AI 负责"实现"**。你告诉 AI"这个按钮放这里感觉不对"，而不是"把 margin-top 改成 16px"。它是 Agentic Coding 的极端形态——你几乎不写代码，只做决策。

今天用 Vibe Coding 的方式做一个小工具。建议选一个你不熟悉的领域——比如你平时写后端 Java，那就做一个前端页面或 Python 脚本。这样才能真正体验"靠感觉而非技能"的编程方式。

**选题建议**（选一个，1-2 小时内能完成的）：
- 一个 Todo 管理页面（HTML + CSS + JavaScript）
- 一个批量重命名文件的命令行工具（Python）
- 一个简单的 REST API Mock Server（Node.js 或 Python）
- 一个数据可视化页面，展示你日常写代码的时间分布（任何语言）

**实践任务**

1. 选择上述一个选题（或自选一个你不熟悉的领域的小工具），用 Vibe Coding 方式完成。规则：
   - 不要自己写任何代码
   - 所有修改通过自然语言指令完成
   - 遇到报错把报错信息直接扔给 AI，不要自己排查
   - 功能不满意用自然语言描述"感觉哪里不对"

2. 过程中记录：
   - 你发了多少条指令？
   - AI 理解了多少次？误解了多少次？
   - 最终结果可用吗？
   - 你对整个过程的控制感如何？

3. 完成后再思考：如果换成你熟悉的领域（Java 后端），Vibe Coding 的效率会更高还是更低？为什么？

**产出要求**

- 一个由 Vibe Coding 完成的、可运行的小工具
- 一份 Vibe Coding 体验笔记（300 字以内，重点是**你的感受**，不是技术细节）

**验收标准**

- 完成的小工具能正常运行
- 你能说出 Vibe Coding 适合什么场景、不适合什么场景

#### Day 7：复盘与总结

**学习内容**

今天是第 1 周的复盘日。不要继续学新东西——停下来，整理这一周的知识体系。

**复盘清单**：

1. **技能盘点**：这周你学会了什么？列出具体可展示的技能点（不是"我对 AI 有了更深的理解"这种废话，是"我能在 3 分钟内写一份让 AI 生成可用代码的 Prompt"这种可验证的陈述）。

2. **工具盘点**：你现在能熟练使用哪些 AI 工具？每个工具的最强场景是什么？

3. **认知变化**：这周之前你对 AI 编码有什么预设？哪些是对的？哪些是错的？

4. **痛点和卡点**：哪些环节你觉得效率很低？哪些概念你觉得还没理解透？

5. **下周重点**：基于本周的体验，第 2 周的 Spec-Driven Development 学习中你最想解决什么问题？

**实践任务**

#### Day 8：Spec-Driven Development 理论学习

**学习内容**

Spec-Driven Development（SDD）的核心理念：**先写好"做什么"的精确描述，再让 AI 负责"怎么做"的实现**。

这和传统开发流程的镜像关系：

```
传统流程：
  需求文档 → 开发人员理解需求 → 脑子里形成设计 → 写代码

SDD 流程：
  需求文档 → 人写 Spec（精确的任务描述） → AI 生成代码 → 人验收
```

关键变化：人的角色从"翻译需求为代码"转变为"翻译需求为 Spec + 验收 AI 输出"。人的核心价值从代码产量转移到设计判断力。

**Spec 的本质**：

一份好的 Spec 不是需求文档。需求文档写"用户要什么"（what the user needs），Spec 写"代码要做什么"（what the code should do）。区别在于精确度：

> 需求文档："用户需要查看订单历史"（模糊，开发需要自己补充大量细节）
>
> Spec："提供一个 REST GET /orders?userId={id}&page={page}&size={size} 接口，返回分页的订单列表，按创建时间倒序。订单对象包含 id、订单号、金额、状态、创建时间。当用户无订单时返回空列表而非 404。"

**Spec 的结构框架**：

```markdown
# 功能名称

## 目标
一句话说清楚这个功能做什么，给谁用。

## 输入
- 参数列表，类型，是否必填，取值范围或格式
- 依赖的外部数据（数据库表、第三方 API 等）

## 输出
- 返回数据结构
- 成功和失败两种情况的响应
- 边界情况（空输入怎么处理？超长输入怎么处理？）

## 业务规则
- 需要验证的规则（如"金额不能为负"）
- 需要计算的逻辑（如"总金额 = 单价 * 数量 * (1 - 折扣率)"）
- 状态流转（如有）

## 技术约束
- 必须使用的技术栈（Spring Boot 2.7+ / MyBatis / MySQL）
- 不能使用的技术（如不使用 Lombok / 不使用存储过程）
- 性能要求（如有，如"查询响应在 100ms 以内"）

## 验收条件
- 功能验收：具体的可测试的条件
- 代码质量：命名规范、测试覆盖率等
- 边界情况：特殊输入的处理结果
```

2. 找一个你最近做过的功能，对照上面的 Spec 结构，尝试为它补写一份 Spec。注意：这不是重新写一遍需求文档，而是用更精确、更工程化的语言重新描述。

#### Day 9：写第一份 Spec

**学习内容**

昨天学了 Spec 的结构，今天动手写一份真正的 Spec——不是为已有功能补写，而是为一个还没做的功能从零写。

**选题建议**：从你当前工作或 Side Project 中找一个 1-2 天开发量的功能。太小了练不出效果（比如"加一个查询接口"），太大了第一天就卡住了（比如"重构整个支付模块"）。合适的粒度：2-4 个接口、1-2 张表、5-10 个业务规则。

以"员工请假申请功能"为例，一份合格的 Spec 参考：

```markdown
# 员工请假申请功能

## 目标
员工可在线提交请假申请，主管可审批/驳回。替代当前的纸质请假单流程。

## 数据模型

### leave_request 表
| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| applicant_id | BIGINT | 申请人 ID |
| leave_type | VARCHAR(20) | 请假类型：ANNUAL/SICK/PERSONAL |
| start_date | DATE | 开始日期（含） |
| end_date | DATE | 结束日期（含） |
| duration_days | DECIMAL(3,1) | 请假天数，精确到 0.5 |
| reason | VARCHAR(500) | 请假原因 |
| status | VARCHAR(20) | PENDING/APPROVED/REJECTED/CANCELLED |
| approver_id | BIGINT | 审批人 ID |
| approve_comment | VARCHAR(500) | 审批意见 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

## API 设计

POST /api/leave-requests - 提交请假申请
GET  /api/leave-requests?status=PENDING&page=0&size=20 - 查询列表
PUT  /api/leave-requests/{id}/approve - 审批通过
PUT  /api/leave-requests/{id}/reject - 驳回申请

## 业务规则
1. 不允许跨年请假
2. 同一天不能被同一人重复申请（日期重叠检测）
3. 年假需要校验剩余额度
4. 只能审批状态为 PENDING 的申请
5. 审批后不允许修改申请表任何字段
6. 取消只能由申请人自己操作，且仅在 PENDING 状态允许

## 技术约束
- Spring Boot 2.7+, MyBatis, MySQL 8.0
- 所有时间以服务器时区为准，存储为 UTC
- 接口需要做身份认证（通过 JWT 获取当前用户）
- 不使用 JPA/Hibernate，使用 MyBatis XML
```

**实践任务**

1. 选定你的功能，按照上述格式写一份 Spec。核心要求：**另一个程序员拿着这份 Spec 能在不跟你交流的情况下写出完整代码**。如果做不到，说明你的 Spec 不够精确。

2. Spec 写完后做自检：
   - 每个输入字段都有类型和校验规则吗？
   - 每种异常情况都有描述吗？
   - 技术约束明确吗（用什么、不用什么）？
   - 边界条件覆盖了吗（空值、极值、并发）？

3. 把 Spec 给你的 Claude Code 看，问它："你觉得这份 Spec 有歧义或遗漏的地方吗？"

**产出要求**

- 一份完整的功能 Spec（明天会用这份 Spec 驱动 AI 生成代码）

**验收标准**

#### Day 10：用 Spec 驱动 AI 生成代码

**学习内容**

今天用昨天写的 Spec 驱动 AI 生成完整代码。这是 SDD 的核心回路：人写 Spec → AI 生成代码 → 人验收。

**操作流程**：

1. **把 Spec 喂给 AI**。在 Claude Code 中：`/add-dir` 纳入项目上下文，然后把 Spec 文件内容贴过去，或直接 `@Spec文件名.md`。
2. **让 AI 按步骤生成**。不是"根据 Spec 写代码"，而是分步来：
   - 步骤 1：根据数据模型生成建表 DDL
   - 步骤 2：生成实体类和 Mapper
   - 步骤 3：生成 Service 层逻辑
   - 步骤 4：生成 Controller 和 DTO
   - 步骤 5：生成单元测试
3. **中途不要修改 Spec 去迁就 AI 的错误**。AI 理解错了 Spec，不是 Spec 的问题——是你给上下文的方式或 Prompt 需要调整。

**常见问题和应对**：

| 问题 | 应对 |
|------|------|
| AI 生成的代码漏了某条业务规则 | 不是重新生成全部，而是："第 X 条业务规则没实现，请补充" |
| AI 生成的 SQL 有语法问题 | 把报错信息贴给 AI，它会自己修 |
| AI"自由发挥"加了 Spec 里没有的东西 | 明确指出："Spec 里没有要求这个功能，请移除" |
| AI 生成的代码风格不统一 | 在 CLAUDE.md 里补充代码规范，然后让 AI 按规范重构 |

**实践任务**

1. 用昨天写的 Spec 驱动 Claude Code 生成完整的代码实现。
2. 生成过程中记录：AI 在第几次尝试后给出了可用的代码？哪些地方需要你介入纠正？
3. 代码生成完成后，运行生成的测试，看是否全部通过。

**产出要求**

- 一套由 Spec 驱动生成的完整代码（Entity/Mapper/Service/Controller/Test）
- 一份"人机协作记录"：你在哪些环节介入了、为什么介入、各花了多少时间

**验收标准**

- 生成的代码能编译通过、测试通过
- 你对代码质量的判断从"大概能用"提升到"我知道哪些地方好、哪些地方需要改"

#### Day 11：AI 代码验收实践

**学习内容**

昨天 AI 生成了代码，今天你要做系统性验收。AI 代码验收和人类代码 Review 有重叠也有差异。

**重叠部分**（对所有代码都适用的标准）：
- 代码是否能编译通过、测试是否覆盖关键路径
- 逻辑是否正确、边界条件是否处理
- 命名是否清晰、结构是否合理
- 是否有明显性能问题（N+1 查询、大事务等）
- 安全漏洞（SQL 注入、XSS、越权等）

**AI 特有的检查维度**：

1. **幻觉代码**：AI 可能调用不存在的 API、引用不存在的类。检查方法：IDE 里看红色波浪线、编译能否通过、搜项目里有没有这个类。

2. **过度工程**：AI 倾向于"看起来专业"的代码——不必要的抽象层、过多的设计模式。检查方法：问自己"这个功能真的有这个复杂度吗？如果我来写，会写这么多类吗？"

3. **安全盲区**：AI 可能忽略认证授权逻辑，直接在 Controller 里暴露数据接口。检查方法：看每个接口是否有认证检查、参数校验、越权判断。

4. **上下文断裂**：AI 可能忘记项目已有的工具类、约定、配置，自己重新造轮子。检查方法：搜一下项目里是否已有类似的实现。

5. **硬编码和配置缺失**：检查字符串、数字常量，看是否应该提取为配置项。

**实践任务**

对 Day 10 生成的代码进行系统性验收，按以下清单逐项检查：

```markdown
## 代码验收检查清单

### 功能正确性
- [ ] 所有业务规则是否实现？逐条对照 Spec 检查
- [ ] 异常情况处理是否完整？
- [ ] 边界条件是否正确处理？

### 代码质量
- [ ] 类名、方法名、变量名是否清晰？
- [ ] 方法长度是否合理（不超过 50 行）？
- [ ] 是否存在重复代码？
- [ ] 是否存在过度设计？

### 安全性
- [ ] 每个接口是否有认证/鉴权检查？
- [ ] 参数校验是否完整？
- [ ] 是否存在 SQL 注入风险？
- [ ] 敏感信息是否脱敏？

### AI 特有问题
- [ ] 是否存在幻觉代码（引用了不存在的类/方法）？
- [ ] 是否重复实现了项目中已有的功能？
- [ ] 是否有不合理的硬编码？
- [ ] 生成的注释是否准确？

### 测试
- [ ] 测试是否覆盖了核心业务逻辑？
- [ ] 测试是否覆盖了边界情况？
- [ ] 测试是否能独立运行？
```

**产出要求**

- 一份填写完整的验收检查清单
- 一份问题分类统计（人类常见问题 vs AI 特有问题）

**验收标准**

- 对自己验收通过（或修改后通过）的代码有信心上线
- 能识别并描述至少 3 种 AI 生成代码的典型问题模式

#### Day 12：Context Engineering 实践

**学习内容**

前几天的实践里你大概率遇到过这种情况：AI 生成的代码风格跟项目不统一，用了你没有的依赖，或者忽略了项目已有的工具类。根本原因不是 AI 能力不行，而是**它不知道你的项目上下文**。

Context Engineering 就是系统性地解决这个问题——给 AI 提供刚好够用的项目信息，不多不少。

**Context 的层次结构**：

```
层次 1：全局规范（对所有 AI 交互有效）
  ├── ~/.claude/CLAUDE.md（个人全局偏好）
  └── 项目 CLAUDE.md（技术栈、规范、结构说明）

层次 2：任务上下文（针对当前任务）
  ├── 相关代码文件（@引用）
  ├── Spec / 需求文档
  └── 错误日志、数据库 Schema 等

层次 3：运行时信息（任务执行中的动态信息）
  ├── 编译错误
  ├── 测试结果
  └── AI 自己的推理轨迹
```

**CLAUDE.md 优化指南**：

一份好的 CLAUDE.md 应该包含：

```markdown
# 项目名称

## 技术栈
- JDK 17, Spring Boot 3.2, MyBatis 3.0, MySQL 8.0
- 构建工具: Maven 3.9+
- 测试框架: JUnit 5, Mockito

## 代码规范
- 遵循阿里巴巴 Java 开发手册
- Controller → Service → Mapper 三层架构
- DTO 用于接口输入输出，Entity 对应数据库表，禁止混用
- 异常统一使用 BusinessException，通过 GlobalExceptionHandler 处理
- 不使用 Lombok，手写 Getter/Setter/Builder

## 项目结构
- controller/ - REST 接口层，只做参数校验和路由
- service/ - 业务逻辑层
- mapper/ - MyBatis Mapper 接口
- entity/ - 数据库实体
- dto/ - 数据传输对象
- config/ - Spring 配置类
- common/ - 通用工具类和异常定义

## 约定
- 所有接口返回统一格式 Result<T>
- 分页查询统一使用 PageQuery 和 PageResult
- 时间字段统一用 LocalDateTime
- ID 生成策略统一使用雪花算法
```

**实践任务**

1. 检查你当前项目的 CLAUDE.md，是否包含以上信息？补充缺失的部分。
2. 用一个新需求测试：在 CLAUDE.md 补充前后，让 AI 分别生成一段代码。对比风格的统一程度。
3. 实验 Context 的"量"对效果的影响：先只给 Spec，再加项目结构，再加代码规范，再加示例代码。每加一层看输出的变化。

**产出要求**

- 一份优化后的项目 CLAUDE.md
- 一份 Context 层次与代码质量关系的实验笔记

**验收标准**

- AI 在你项目里生成的代码风格基本统一，不需要你大量调整
- 你知道在什么情况下 AI 的代码跑偏了、应该加什么类型的上下文来纠正

#### Day 13：AI 工作流设计实践

**学习内容**

过去两周你体验了各种 AI 使用方式：单次问答、Agentic Coding、SDD、Vibe Coding。今天把这些方式串起来，设计属于你自己的 AI 工作流。

**问题**：在实际工作中，不同任务适合不同的 AI 协作方式。一个错误的策略是把所有任务都扔给 Agent 模式让它自己跑——有些任务你写一个精确 Prompt 让 AI 一次生成，效率反而更高。

**任务类型与 AI 策略映射**：

| 任务类型 | 推荐策略 | 典型例子 |
|----------|----------|----------|

| 技术方案 | Plan Mode 讨论 | 选型、架构设计 |
| 代码 Review | 批量审查 + 人工复核 | Review 一个 PR |

**实践任务**

1. 总结你这两周的 AI 使用体验，参考上表设计一份"我的 AI 工作流"。
2. 用这份工作流去完成一个日常任务，看预设流程是否需要调整。
3. 如果你有团队，跟同事交流你的工作流，看看有没有可以互相借鉴的地方。

**产出要求**

- 一份"我的 AI 工作流"文档（放在 CLAUDE.md 或单独的笔记中）

**验收标准**

- 你不再每个任务临时想"该让 AI 怎么帮忙"——有预设流程
- 工作流文档足够清晰，另一个同事照着做也能执行

#### Day 15：需求分析 + AI

**学习内容**

大多数程序员拿到需求后的第一反应是"想怎么实现"。在 AI 时代，正确的第一反应应该是"先把需求变成 AI 能理解的精确描述"。

**AI 在需求分析中的角色**：

1. **需求澄清**：把用户的一句话需求展开为结构化的功能点列表。

   Prompt 示例：
   ```text
   请帮我把以下一句话需求拆解为可开发的功能点列表：
   "用户可以在线提交报销申请，上传发票照片，财务审核后打款。"
   
   请从以下维度拆解：
   1. 功能点列表（每个功能点一句话描述）
   2. 角色与权限（有哪些角色、各自能做什么）
   3. 数据模型猜测（核心实体、大概字段）
   4. 流程与状态机（有哪些状态、谁触发状态变更）
   5. 潜在的技术风险点
   ```

2. **边界情况挖掘**：AI 很擅长帮你发现需求中的盲区——"如果用户上传的不是发票怎么办？""如果审批人请假了谁来审批？"

3. **需求到 Spec 的转换**：把产品需求文档转换为可执行的 Spec（这个技能 Day 8-9 已经练过了）。

**实践任务**

1. 从你的工作 backlog 中取一个还没开始做的需求。
2. 用 AI 做需求澄清：把需求描述扔给 AI，让它拆解功能点、挖掘边界情况。
3. 对比 AI 的输出和你自己的理解：有没有你没想到的点？有没有 AI 理解错的地方？
4. 基于 AI 的分析结果，形成一份更精确的需求理解文档。

**产出要求**

- 一份"需求分析结果"文档（包含功能点列表、边界情况、风险点）
- 一份标注了 AI 补充内容的原始需求文档对照

**验收标准**

- 需求分析的完整度明显高于你自己纯手工分析
- 至少发现了 3 个你之前没意识到的边界情况

#### Day 16：接口设计 + AI

**学习内容**

接口设计是后端开发的核心环节。一个好的接口设计应该让调用方"不用问就能用"。AI 可以帮你做两件事：根据需求生成接口草稿，以及 Review 已有接口设计的合理性。

**AI 辅助接口设计的流程**：

1. 输入需求描述 + 数据模型 → AI 输出接口设计方案（URL、Method、请求参数、响应结构、错误码）。
2. 人工 Review：检查 URL 命名、参数合理性、响应结构统一性。

**实践任务**

2. 把生成的接口设计贴给 AI 做 Review，检查 RESTful 规范、分页参数统一性、错误码完整性。

4. 如果你使用 Swagger/OpenAPI，让 AI 生成 OpenAPI 3.0 的 YAML 文件。

**产出要求**

- 一份完整的接口设计文档（URL、Method、参数、响应、错误码）
- 一份 AI Review 的结果和改进记录

**验收标准**

- 接口设计遵循 RESTful 规范

#### Day 17：数据库设计 + AI

**学习内容**

数据库设计把需求中的业务实体转化为表结构。对于企业应用，数据库设计往往比代码更"重"——迁移成本高、影响范围大、需要提前想清楚。

**AI 在数据库设计中的角色**：

1. 从需求描述中提取实体，生成 ER 图和建表 DDL。
2. 根据查询场景建议合理的索引。
3. 检查范式、字段类型、命名规范。

**Prompt 示例**：

```text
基于以下需求描述和接口设计，生成 MySQL 数据库设计方案：
[贴上需求和接口设计摘要]

要求：
1. 输出所有表的 DDL（CREATE TABLE 语句）
2. 每张表标注索引建议及理由
3. 标注表之间的关联关系
4. 字段类型选择（优先使用合适而非过大类型，如 TINYINT 存状态而非 INT）
5. 必备字段（id, created_at, updated_at）
6. 软删除策略（如果需要）
```

**实践任务**

1. 基于前两天的需求分析和接口设计，用 AI 生成数据库设计方案。
2. 让 AI Review 生成的 DDL，检查字段类型、索引覆盖、是否有遗漏。
3. 用 AI 生成数据库设计文档（Markdown 格式，包含表说明、字段说明、索引说明）。

**产出要求**

- 一份完整的 DDL 文件
- 一份数据库设计文档

**验收标准**

- DDL 可以在 MySQL 中正常执行

#### Day 19：代码 Review + AI

**学习内容**

传统的 Code Review 靠人工逐行阅读，效率低且容易遗漏。AI 可以在几秒内扫描整个 diff，帮你发现命名问题、潜在 Bug、安全漏洞——但如何有效使用 AI 做 Review，是一门需要练习的技能。

**AI Code Review 的最佳实践**：

1. **分维度 Review，不要一锅炖**：
   - 第一轮：安全性（认证、授权、注入、敏感信息）
   - 第二轮：正确性（逻辑错误、边界处理、状态机）
   - 第三轮：代码质量（命名、重复、耦合、复杂度）
   - 第四轮：规范一致性（是否遵循项目约定）

2. **让 AI 给每个问题标严重级别**：CRITICAL / HIGH / MEDIUM / LOW。你重点看 CRITICAL 和 HIGH。

3. **不要无脑信任 AI 的 Review 结果**：AI 可能误报，也可能漏报（忽略业务层面的安全风险）。

**实践任务**

1. 让 AI Review Day 18 写出来的代码（使用分维度 Review Prompt）。
2. 自己手工对照代码，看 AI 发现的问题中哪些是真的、哪些是误报、哪些它漏了。
3. 对真正的问题进行修复。
4. 让 AI 对修复后的代码再做一次 Review，确认问题已解决。

**产出要求**

- 一份 AI Code Review 报告（含严重级别标注）
- 一份"AI 误报 vs 漏报"对比记录

**验收标准**

- 能判断 AI Review 结论的可靠性，不盲目信任也不全盘否定
- 能设计适合自己团队的 AI Review 流程

#### Day 22：RAG 实战

**学习内容**

RAG（Retrieval-Augmented Generation，检索增强生成）是目前企业级 AI 应用最常见的模式。简单说：**给 LLM 外挂一个知识库，让它回答问题时先查知识库，再结合检索结果生成回答**。

为什么需要 RAG？
- LLM 的知识有截止日期，你公司的内部文档不在它的训练数据里
- LLM 会产生幻觉——编造看似合理但错误的信息
- 你无法把所有项目代码都塞进 Context Window（Token 限制）

**RAG 的核心流程**：

```
1. 文档切片（Chunking）
   文档/代码 → 切成小块（每块 500-1000 tokens）

2. 向量化（Embedding）
   每个文本块 → Embedding 模型 → 向量

3. 存储（Vector Store）
   所有向量存入向量数据库（pgvector / Chroma / Milvus）

4. 检索（Retrieval）
   用户问题 → 向量化 → 在向量库中找最相似的 N 个文本块

5. 生成（Generation）
   检索到的文本块 + 用户问题 → LLM → 最终回答
```

**实践任务**

今天的实践用 Java + Spring Boot 搭建一个最简单的 RAG 演示：

1. 准备 3-5 个文档（项目规范、API 文档、常见问题等）。
2. 搭建最小化 RAG Pipeline：写一个简单的检索方法（基于关键词或向量相似度），把检索结果拼接到 Prompt 中。
3. 对比实验：同一个问题，分别用"不带上下文的 LLM"和"带了项目规范的 RAG"回答，对比结果。

如果时间或环境受限，可以先用 Claude Code 的 `/add-dir` 手动添加上下文来模拟 RAG 效果，理解"有上下文"和"无上下文"的差异即可。

**产出要求**

- 一个可运行的 RAG Pipeline（即使是最简实现）
- 一份"有 RAG vs 无 RAG"的回答质量对比

**验收标准**

- 能用自己的话解释 RAG 的原理和适用场景
- 知道什么时候需要 RAG、什么时候不需要

#### Day 25：AI 使用规范编写

**学习内容**

如果没有规范，5 个人的团队会出现 5 种截然不同的 AI 使用方式：有人把整段核心代码贴给 ChatGPT，有人只用 AI 写注释，有人完全不信 AI。规范的目的不是限制，是**把好用的方式固化为标准，把危险的模式明确禁止**。

**规范应包含的组成部分**：

1. 适用范围：涵盖哪些 AI 工具、哪些开发环节
2. 允许/禁止事项：什么可以做、什么不可以
3. 推荐实践：不同场景下的最佳 AI 使用方式
4. Review 标准：AI 生成代码的审查流程
5. Prompt 模板库：统一团队的质量基准

**示例**：

```markdown
## AI 辅助开发规范 v1.0

### 允许
- 使用 AI 生成模板代码（CRUD、DTO、Mapper 等）
- 使用 AI 分析代码逻辑和查找 Bug
- 使用 AI 生成测试用例和 API 文档
- 使用 AI 做 Code Review 预审

### 禁止
- 直接将包含生产密钥、Token、密码的代码提交给外部 AI
- 将客户的敏感业务数据作为 Prompt 输入
- 将公司核心算法完整提交给外部 AI 分析
- 对 AI 生成的权限检查、安全相关代码不做人工审查直接上线
- 使用 AI 生成的 SQL 不经过 Review 直接在生产环境执行

### 审核要求
- AI 生成的代码必须通过 Code Review
- 涉及安全、支付、权限的代码必须人工走查
- AI 生成的 SQL DDL/DML 必须经过 DBA 或技术负责人 Review
```

**实践任务**

1. 参考上述模板，为你的团队编写一份 AI 使用规范。
2. 考虑你们团队的特殊情况补充条目。
3. 发给至少一个同事征求意见。

**产出要求**

- 一份 AI 使用规范文档

**验收标准**

- 规范内容具体、可执行，不是泛泛的原则
- 至少有一个同事看过并给过反馈

#### Day 26：团队培训方案设计

**学习内容**

**培训方案设计要点**：

#### Day 28：综合项目实战（下）

**学习内容**

基于昨天的 Spec，用 SDD 方法论驱动 AI 生成完整代码，然后进行系统验收。

**操作流程**：

1. **代码生成**（60-90 分钟）：把 Spec + CLAUDE.md 喂给 Claude Code，分层生成：Entity → Mapper → Service → Controller → Test。每层生成后用编译验证。
2. **测试补充**（30 分钟）：用 AI 补充边界测试用例和集成测试数据。跑通全部测试。
3. **文档生成**（20 分钟）：让 AI 基于代码生成 API 文档和 CHANGELOG。
4. **代码 Review**（30 分钟）：用 AI 做第一轮 Review，人工对照代码做第二轮，修复发现的问题。

**实践任务**

1. 完成代码生成、测试、文档、Review 全流程。
2. 确保 `mvn clean test` 全部通过。
3. 记录全流程的时间消耗：需求分析 X 分钟、接口设计 X 分钟……代码生成 X 分钟……验收 X 分钟。
4. 对比：如果不使用 AI，你估计每个环节各需要多少时间？

**产出要求**

- 完整可运行的代码（通过编译和测试）
- API 文档
- Code Review 报告
- 一份时间消耗统计（使用 AI vs 不使用 AI 的对比）

## 常见问题

### Q: 每天 1-2 小时够吗？我有时候加班，一整周都没时间学。

A: 处理方式：

1. 优先保证每周至少 3 天的学习时间，而不是追求每天都做。
2. 工作时间不够就在工作中用——把学习任务嵌入工作。比如你这周要做 CRUD，就用 AI 来做，哪怕多花半小时，也是在学。
3. 跳过的天数不要补，继续往前。这份计划的核心价值不在于"严格打卡"，在于"建立习惯"。
4. 最低要求：每周至少完成本周的重点任务。

### Q: 我公司不让用外部 AI 工具怎么办？

A: 分级处理：

- **最理想**：推动公司采购企业版 API（如 Anthropic API / Azure OpenAI），数据不用于训练，有 SLA 保障。
- **次选**：用本地模型。Ollama + CodeGemma / Llama 3 等可以在本地运行，保密性没问题。
- **兜底**：先用个人项目练手。白天上班正常写，晚上用自己的 Side Project 练 AI 技能。

### Q: 我已经用 AI 写了几个月的代码了，还需要按这个计划走吗？

A: 如果你已经有系统性的 SDD 流程、自己的验收检查清单、知道工具在什么场景强什么场景弱、团队有 AI 使用规范，那挑感兴趣的章节（RAG、MCP、企业落地）快速过一遍即可。如果你只是"用 AI 写代码"但没有方法论和系统性的质量保障——从头走一遍不亏。有经验和有方法是两回事。

### Q: 学完这个 30 天计划，我在团队里算什么水平？

A: 认真完成所有任务（尤其是 Day 27-29 的综合项目）后，你在 AI 辅助开发方面的能力大概率超过团队里 90% 的人。在大多数还没有系统推广 AI 的团队里，你能成为那个"内部 Champion"。

### Q: 为什么要学 Spec-Driven Development？直接让 AI 写不就行了？

A: 没 Spec 用 AI 写代码，像不带图纸让施工队盖房子。Spec 的价值在于：减少返工（AI 有明确目标）、你能验收（逐条对照）、知识沉淀（三个月后回看，Spec 比十行注释都有用）。

### Q: 这个计划对前端开发或全栈开发适用吗？

A: 方法论层面完全适用（SDD、Prompt Engineering、Context Engineering、验收），工具层面需要调整。本书的实践示例用 Java/Spring Boot，前端同学替换为 React/Vue + Node.js 即可。

### Q: 学完这个之后下一步应该学什么？

A: 取决于你的方向：继续深入工程（多 Agent 协作、自主 Debug）、往架构方向（大模型应用架构、向量数据库）、往管理方向（AI 驱动的工程效能度量）、或者保持务实——把手头项目做得更好更快。

## 本章小结

30 天前，你可能觉得"AI 很厉害，但我不知道怎么用在工作里"。30 天后，你应该有了一套完整的武器库：从理解 AI 能力边界（第 1 周），到掌握 Spec-Driven Development 方法论（第 2 周），到在企业场景中实战应用（第 3 周），到掌握 RAG/MCP 进阶技术并推动团队落地（第 4 周）。

但这 30 天只是开始。AI 编程工具每年都在发生质变，一年后回头看，今天的"最佳实践"可能已经过时。关键不是你学了什么具体的技术，而是你建立了一种新的工作习惯：

**拿到需求，先想 Spec 再动手。代码生成后，用检查清单验收而不是凭感觉。用 AI 之前，确保它有了足够的项目上下文。发现 AI 犯错，记录下来而不是抱怨。**

这些习惯，比任何具体的 Prompt 模板都值钱。

如果你正在读这段话，说明你已经走到了本章的结尾——这就去 Day 1 开始吧。不要规划，不要犹豫，打开 Claude Code，随便找一个你今天要做的小功能，先让 AI 写一段试试。最好的学习方法不是看书，不是看视频，不是读教程。是**用起来**。
